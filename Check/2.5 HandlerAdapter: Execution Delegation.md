# HandlerAdapter: Execution Delegation

Let's start with the problem this system is solving, because the design only makes sense once you feel the pressure it's relieving.

## The Problem: DispatcherServlet Doesn't Know What It Got Back

After `HandlerMapping` runs, `DispatcherServlet` holds a handler object. But what *is* that object? It might be a `HandlerMethod` wrapping one of your `@GetMapping` methods. It might be a `ResourceHttpRequestHandler` for serving static files. It might be a legacy `Controller` interface implementation from Spring 1.x. Each of these requires a completely different invocation mechanism — different method signatures, different return value semantics, different ways of writing to the response.

Without `HandlerAdapter`, DispatcherServlet would need a growing chain of `instanceof` checks, and every new handler type would require modifying the dispatcher itself. Instead, Spring applies the **Adapter pattern**: DispatcherServlet asks each registered adapter "can you handle this?" via `supports(handler)`, picks the first that says yes, and delegates the actual invocation to it. DispatcherServlet stays stable regardless of how many handler types are added.

## The Contract: What a HandlerAdapter Promises

The interface has three methods, and understanding what each one *means* is more important than memorizing the signatures.

`supports(handler)` is a simple capability query — it allows DispatcherServlet to find the right adapter without knowing anything about the handler type. `handle(request, response, handler)` does the actual work and returns a `ModelAndView` — *or null*. This null return is critically important: it means "I've already written the response, there's nothing to render." This is how `@ResponseBody` and REST endpoints work — the adapter writes JSON directly and returns null, telling DispatcherServlet to skip the entire ViewResolver pipeline. `getLastModified()` exists for HTTP caching support and is largely deprecated in modern Spring.

## The Four Adapters and What They Cover

Three of the four adapters are fairly simple wrappers. `HttpRequestHandlerAdapter` just casts the handler to `HttpRequestHandler` and calls `handleRequest()` — one line of delegation. `SimpleControllerHandlerAdapter` does the same for the legacy `Controller` interface. `HandlerFunctionAdapter` covers the functional WebMvc programming model.

`RequestMappingHandlerAdapter` is the one that matters for virtually every modern Spring application, and it's arguably the most complex class in all of Spring MVC. The rest of this explanation is primarily about how it works.

## RequestMappingHandlerAdapter: A Pipeline, Not a Method Call

When you call a `@GetMapping` method, the naive picture is: "Spring finds the method and calls it." The reality is that there's an entire pipeline assembling and disassembling around that call, and understanding each stage is what separates surface-level knowledge from genuine understanding.

### Startup vs. Per-Request Work

Like `RequestMappingHandlerMapping`, this adapter front-loads significant work into `afterPropertiesSet()`. At startup, it scans all `@ControllerAdvice` beans and extracts their `@InitBinder` and `@ModelAttribute` methods for later use. It also builds the full chains of `HandlerMethodArgumentResolver`s and `HandlerMethodReturnValueHandler`s — the two pipelines that handle method inputs and outputs. By the time the first request arrives, all of this is ready.

### The ModelAndViewContainer: One Object Carries All State

Before digging into the pipeline stages, you need to understand `ModelAndViewContainer` — a mutable state object that travels through the entire invocation and serves as the communication channel between all the pieces. It holds the model map where attributes accumulate, the view name or object that will eventually be rendered, and most importantly, a `requestHandled` boolean flag. Think of it as the "running tab" for a single request's processing. When any component decides "I've fully handled this response myself," it sets `requestHandled = true`, and downstream components know to stop.

### Stage 1: The WebDataBinderFactory

Before any method arguments are resolved, the adapter creates a `WebDataBinderFactory`. This factory is responsible for creating `WebDataBinder` instances when data binding is needed — turning HTTP request parameters into Java objects. The factory is initialized by collecting `@InitBinder` methods: first from applicable `@ControllerAdvice` beans (global), then from the current controller (local). This ordering means your global customizations always run before controller-specific ones.

A subtle but important point about `@InitBinder`: it runs *per binding operation*, not once per request. If a controller has two `@ModelAttribute` parameters, `@InitBinder` fires twice. This matters if your `@InitBinder` method has side effects like logging or metrics.

### Stage 2: The ModelFactory and @ModelAttribute Execution

Next, the adapter creates a `ModelFactory` and calls `initModel()`, which executes all applicable `@ModelAttribute` methods. Here's the trap that surprises almost everyone: **these methods always run, even for `@ResponseBody` endpoints**. If your `@ModelAttribute` method makes a database call to populate a dropdown list, that database call fires on every request to that controller — including your REST API endpoints where the model is immediately discarded. This is a hidden performance cost that's easy to miss during development.

The execution order is: global `@ControllerAdvice @ModelAttribute` methods first (sorted by `@Order`), then controller-local `@ModelAttribute` methods.

### Stage 3: Argument Resolution — The Full Chain

With the model populated and the binder factory ready, the adapter creates a `ServletInvocableHandlerMethod` and calls `invokeAndHandle()`. This begins by resolving every method parameter through the `HandlerMethodArgumentResolver` chain.

For each parameter, the chain asks every registered resolver "do you support this parameter?" via `supportsParameter()`, and the first one that says yes gets to resolve it. The chain is ordered deliberately, and there's one particularly clever design decision worth understanding: `RequestParamMethodArgumentResolver` is registered **twice** — once early in the chain with `useDefaultResolution=false` (handling only explicitly `@RequestParam`-annotated parameters), and once at the very end with `useDefaultResolution=true` (handling any simple-type parameter even without an annotation). This is why `public String search(String query)` works without an explicit `@RequestParam` — the catch-all registration at the end handles it.

The `@RequestBody` resolver — `RequestResponseBodyMethodProcessor` — is especially important because it delegates to the `HttpMessageConverter` chain to deserialize the request body. It checks the request's `Content-Type`, finds a converter that can read that media type into the target type, and runs the deserialization. Jackson's converter handles `application/json`, for instance.

One consequence of the chain's ordering is critical for customization: resolvers added via `WebMvcConfigurer.addArgumentResolvers()` are **appended after all defaults**. They cannot override built-in behavior because built-in resolvers will claim the parameter first. To truly replace a built-in resolver, you'd need to configure `RequestMappingHandlerAdapter` directly — a much more invasive change.

### Stage 4: The Method Call Itself

Once all arguments are resolved into an array, the actual controller method is invoked via reflection: `method.invoke(bean, resolvedArgs)`. This is the moment your business logic runs. The entire framework above was building up to this single line.

### Stage 5: Return Value Handling

The return value then flows into the `HandlerMethodReturnValueHandler` chain. Each handler checks `supportsReturnType()` and the first match processes the value.

String returns go to `ViewNameMethodReturnValueHandler`, which stores the view name in `mavContainer` for later rendering. `ResponseEntity<T>` returns go to `HttpEntityMethodProcessor`, which extracts the status code, copies the headers to the response, and serializes the body via `HttpMessageConverter`. Methods annotated with `@ResponseBody` are handled by `RequestResponseBodyMethodProcessor`, which serializes the return value based on content negotiation against the client's `Accept` header.

Critically, after writing the response body, **both** `HttpEntityMethodProcessor` and `RequestResponseBodyMethodProcessor` call `mavContainer.setRequestHandled(true)`. This single flag is the mechanism that separates REST APIs from view-rendered responses. When `invokeHandlerMethod()` later calls `getModelAndView()`, it finds `requestHandled = true` and returns `null`. DispatcherServlet receives null and skips ViewResolver and rendering entirely.

If you write a custom `ReturnValueHandler` that writes directly to the response but forgets to set `requestHandled = true`, Spring will attempt to also render a view — producing either `IllegalStateException: response already committed` or garbled output. This is one of the most common mistakes when building custom return value handlers.

## A Mental Exercise

Consider this method and trace what happens step by step:

```java
@PostMapping("/orders")
public ResponseEntity<Order> createOrder(@RequestBody @Valid OrderRequest req) {
    Order saved = orderService.save(req);
    return ResponseEntity.created(URI.create("/orders/" + saved.getId()))
                         .body(saved);
}
```

At startup, the adapter registered `RequestResponseBodyMethodProcessor` for `@RequestBody` resolution and `HttpEntityMethodProcessor` for `ResponseEntity` handling. When the request arrives, `@ModelAttribute` methods run first (even though there's no model being rendered). Then `RequestResponseBodyMethodProcessor.resolveArgument()` reads the JSON body, finds Jackson's converter, and deserializes the JSON into an `OrderRequest`. The `@Valid` constraint triggers Bean Validation before returning to the adapter. The method is invoked with the validated `OrderRequest`. The return value — a `ResponseEntity<Order>` — flows into `HttpEntityMethodProcessor`, which sets the 201 status, copies the `Location` header, serializes `Order` to JSON via Jackson, and sets `requestHandled = true`. DispatcherServlet receives null from `getModelAndView()` and the request is complete.

Every annotation you wrote corresponds to a specific component in this pipeline, and understanding which component handles which annotation is what gives you confident mental models for debugging and customizing Spring MVC behavior.
