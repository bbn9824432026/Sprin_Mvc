# DispatcherServlet Lifecycle: From `init()` to `destroy()`

This topic rewards careful attention because most developers have a vague sense that "DispatcherServlet starts up and handles requests," but the real picture is far more precise — and the details matter both for debugging production issues and for understanding *why* Spring behaves the way it does. Let's build the complete mental model from the ground up.

---

## The Key Insight: The Lifecycle Is Spread Across Four Classes

Before diving into any phase, you need to hold one architectural fact in your head: the lifecycle methods are not all in `DispatcherServlet`. They are deliberately distributed across a four-class inheritance hierarchy, where each class owns a specific responsibility. Getting this wrong on an exam — or in an interview — is a common mistake.

`HttpServletBean` sits at the base and owns the bridge between the raw Servlet API and Spring's world. Its `init(ServletConfig)` method is the very first Spring code that runs after Tomcat instantiates the servlet. `FrameworkServlet` sits above it and owns the WebApplicationContext wiring — creating the child context, setting up ThreadLocals, and funneling all HTTP methods into a single processing path. `DispatcherServlet` itself only owns the MVC-specific work: initializing the special strategy beans and performing the actual request dispatch. Think of it as a division of labor — infrastructure concerns at the bottom, web concerns at the top.

---

## Phase 1 — Instantiation: Two Very Different Constructors

The lifecycle begins when Tomcat instantiates `DispatcherServlet`. What's important here is *which constructor* gets called, because this differs fundamentally between plain Spring MVC and Spring Boot.

In plain Spring MVC, Tomcat calls the **no-arg constructor** via reflection. At this point, no Spring context exists anywhere. The object is a blank servlet waiting to be initialized. `detectAllHandlerMappings`, `detectAllHandlerAdapters`, and `detectAllViewResolvers` are all set to `true` by default — you'll see why these flags matter later.

In Spring Boot, something different happens. Spring Boot's `DispatcherServletAutoConfiguration` creates `DispatcherServlet` as a Spring bean using the **context-accepting constructor**, passing in the already-existing `ApplicationContext`. The servlet receives a pre-built, already-refreshed context at construction time. It doesn't need to create one. This single difference is what drives most of the Boot-vs-plain-MVC behavioral differences throughout the rest of the lifecycle.

---

## Phase 2 — `init()` in `HttpServletBean`: The Init-Param Bridge

After instantiation, Tomcat calls `init(ServletConfig)`. This is the standard Servlet API hook, but `HttpServletBean` uses it to do something clever — it uses Spring's own `BeanWrapper` mechanism to map every `<init-param>` in `web.xml` to a setter call on the `DispatcherServlet` object.

This means when you write this in `web.xml`:

```xml
<init-param>
    <param-name>throwExceptionIfNoHandlerFound</param-name>
    <param-value>true</param-value>
</init-param>
```

Spring internally calls `dispatcherServlet.setThrowExceptionIfNoHandlerFound(true)`. The init-param name maps to a Java bean property name. This is an elegant reuse of Spring's property binding infrastructure — the same mechanism used for configuring any Spring bean is reused here to configure the servlet itself. After all init-params are applied as property values, `HttpServletBean` calls `initServletBean()` and hands control upward to `FrameworkServlet`.

---

## Phase 3 & 4 — Context Creation in `FrameworkServlet`

`FrameworkServlet.initServletBean()` is where the critical work happens. It calls `initWebApplicationContext()`, which is the most complex method in the entire lifecycle. Understanding its logic is crucial because it handles three completely different scenarios with one method.

**Scenario A — Spring Boot path:** The context was passed in via the constructor and is already refreshed. `initWebApplicationContext()` detects that `this.webApplicationContext != null` and that the context `isActive()` — meaning it's already been refreshed. It skips the refresh entirely and goes straight to calling `onRefresh()` manually.

**Scenario B — Plain Spring MVC path:** No context exists yet. The method calls `createWebApplicationContext(rootContext)`, which creates a new `AnnotationConfigWebApplicationContext` or `XmlWebApplicationContext`, sets the root context (from `ContextLoaderListener`) as its parent, and calls `configureAndRefreshWebApplicationContext()`. This triggers `context.refresh()`, which instantiates all your `@Controller`, `ViewResolver`, and other web-layer beans. When refresh completes, it publishes `ContextRefreshedEvent`, which `FrameworkServlet` is listening for — that event triggers `onRefresh()`.

**Scenario C — Shared context:** Multiple `DispatcherServlet` instances can share a pre-built context stored in `ServletContext`. `findWebApplicationContext()` looks for it by a named attribute. This is a less common pattern but worth knowing exists.

The error handling here is also important: if `initWebApplicationContext()` throws any exception, `FrameworkServlet` lets it propagate, Tomcat catches it, and marks the servlet as permanently unavailable. Subsequent requests receive HTTP 503. The servlet does not get a second chance — this is why startup-time wiring errors are so valuable. Spring fails fast and loudly.

---

## Phase 5 — `initStrategies()`: Building the MVC Machinery

Once the child context is ready, `DispatcherServlet.onRefresh()` is called, which immediately calls `initStrategies()`. This is a one-time method that runs nine sub-initializations, each following the same pattern. Let's trace `initHandlerMappings()` as the representative example since it illustrates the most important behavior.

```java
private void initHandlerMappings(ApplicationContext context) {
    this.handlerMappings = null;

    if (this.detectAllHandlerMappings) {
        // This searches BOTH the child context AND the root context (ancestors)
        Map<String, HandlerMapping> matchingBeans =
            BeanFactoryUtils.beansOfTypeIncludingAncestors(
                context, HandlerMapping.class, true, false);

        if (!matchingBeans.isEmpty()) {
            this.handlerMappings = new ArrayList<>(matchingBeans.values());
            // Sorted by @Order / Ordered — priority order matters
            AnnotationAwareOrderComparator.sort(this.handlerMappings);
        }
    } else {
        // detectAllHandlerMappings=false: only the bean named EXACTLY "handlerMapping"
        HandlerMapping hm = context.getBean("handlerMapping", HandlerMapping.class);
        this.handlerMappings = Collections.singletonList(hm);
    }

    // If nothing found anywhere, fall back to DispatcherServlet.properties defaults
    if (this.handlerMappings == null) {
        this.handlerMappings = getDefaultStrategies(context, HandlerMapping.class);
    }
}
```

Notice `BeanFactoryUtils.beansOfTypeIncludingAncestors()` — this is the method that searches the child context *and* walks up to the parent (root) context. This is why `HandlerMapping` beans can technically live in either context and still be found. The results are sorted by `@Order`, so you can control which `HandlerMapping` gets consulted first.

The `detectAllHandlerMappings=false` scenario is an important exam trap. When this flag is false, Spring ignores every `HandlerMapping` bean that isn't named exactly `"handlerMapping"`. This is a very controlled, restricted mode — and if you define a `HandlerMapping` bean with any other name, it is silently ignored. No warning, no error.

After `initStrategies()` completes, all nine strategy types (`HandlerMapping`, `HandlerAdapter`, `ViewResolver`, `MultipartResolver`, `LocaleResolver`, etc.) are stored as instance fields on `DispatcherServlet`. The per-request lookup is then just a field access — no bean lookups at request time.

---

## Phase 6 — Per-Request Processing: The `processRequest()` Funnel

Every HTTP method — GET, POST, PUT, DELETE, PATCH, OPTIONS, TRACE — is overridden in `FrameworkServlet` to call one single method: `processRequest()`. This funneling is intentional. It means there is exactly one place where cross-cutting per-request setup and teardown happens, regardless of the HTTP verb.

`processRequest()` does five conceptually important things around the actual request handling:

**First, it saves and restores ThreadLocals.** Before setting new ThreadLocal values, it saves the existing ones (`previousLocaleContext` and `previousAttributes`). After the request completes, it restores them. This supports *nested dispatch* — when a FORWARD or INCLUDE causes a second `processRequest()` call within the first, the inner call saves the outer call's ThreadLocals, does its work, and restores them when done. The outer call's context is intact when the inner call returns.

**Second, it sets up new ThreadLocals for this request.** `LocaleContextHolder` is populated with the request's locale, and `RequestContextHolder` is populated with a `ServletRequestAttributes` wrapping the `HttpServletRequest` and `HttpServletResponse`. This is what makes it possible to call `RequestContextHolder.getRequestAttributes()` from a service deep in your call stack and still get the current request — the information travels via ThreadLocal rather than as a method parameter.

**Third, it calls `doService()`**, which sets request attributes and then calls `doDispatch()` — the actual MVC dispatch logic.

**Fourth, in the `finally` block** (which always runs, even on exceptions), it restores the previous ThreadLocals and calls `requestAttributes.requestCompleted()`. That last call is more important than it looks — it's what triggers the destruction of `@RequestScope` beans. When the request completes, Spring needs to destroy any request-scoped beans that were created during this request, and `requestCompleted()` is the signal that does that.

**Fifth, also in `finally`**, it calls `publishRequestHandledEvent()`. This fires a `ServletRequestHandledEvent` containing the URL, method, processing time, and failure cause (if any). Importantly, this fires even when the request threw an exception. If you write a listener for this event, you get a complete picture of every request that passed through `DispatcherServlet`.

The ThreadLocal trap that catches many developers is the child thread scenario. ThreadLocals are thread-local by definition — they are not inherited by child threads. If you spawn a new `Thread` (or submit a task to an `ExecutorService`) inside a controller, that thread has *empty* `RequestContextHolder` and `LocaleContextHolder`. Calling `RequestContextHolder.currentRequestAttributes()` on that thread throws `IllegalStateException: No thread-bound request found`. The fix is to capture the values you need on the request thread before spawning the child thread, and pass them explicitly.

---

## Phase 7 — `doService()`: Preparing the Request for Dispatch

`doService()` is a brief but important setup step before `doDispatch()`. Its main job is to set specific attributes on the `HttpServletRequest` that controllers, interceptors, and view renderers can access later. These include the `WebApplicationContext` itself (so code can programmatically access the Spring container), the `LocaleResolver`, the `ThemeResolver`, and both input and output `FlashMap` objects (for passing attributes across redirects).

There's a nuanced piece of logic here for INCLUDE dispatches. When a request is being processed as an `INCLUDE` (one servlet including another), the request attributes already set by the outer dispatch need to be preserved and restored after the inner dispatch completes. `doService()` snapshots the existing attributes before overwriting them, and restores them in its `finally` block. This is what makes nested dispatching work correctly.

---

## Phase 8 — `destroy()`: The Context Ownership Question

`FrameworkServlet.destroy()` has a surprisingly important conditional: it only calls `context.close()` if `webApplicationContextInjected` is `false`.

In plain Spring MVC, `DispatcherServlet` *created* its child context during `init()`. It owns it. When the servlet is destroyed, it should close the context — calling `@PreDestroy` methods, running `DisposableBean.destroy()`, and cleaning up all singleton beans in the child context. `webApplicationContextInjected` is `false` in this case, so the close happens.

In Spring Boot, the context was *passed in* via the constructor. `DispatcherServlet` doesn't own it — Spring Boot's `SpringApplication` does. If `destroy()` tried to close the context, it would destroy the entire application context prematurely. So `webApplicationContextInjected` is `true`, and `destroy()` skips the close. `SpringApplication` registers a JVM shutdown hook that closes the context when the JVM exits.

---

## The Fundamental Timing Difference Between Plain MVC and Spring Boot

This is one of the most important conceptual differences to internalize. The sequence of events is *reversed* between the two approaches.

In **plain Spring MVC**, the sequence is: Tomcat starts → Tomcat initializes `DispatcherServlet` → `context.refresh()` creates all beans. The context is built *after* Tomcat is already running. There's technically a window between Tomcat starting and `DispatcherServlet.init()` completing where the servlet is not yet ready (though requests would get 503 during this window).

In **Spring Boot**, the sequence is: `SpringApplication` creates and refreshes the context (all beans are ready, including `DispatcherServlet` as a bean) → *then* `EmbeddedWebServerFactory` creates and starts Tomcat → Tomcat calls `DispatcherServlet.init()` → `initStrategies()` runs. Tomcat starts *after* all beans are ready. By the time Tomcat is accepting requests, the entire application is fully initialized. There is no window where an incomplete `DispatcherServlet` could receive requests.

This also means that in Spring Boot, if your `ApplicationContext` fails to refresh (a misconfigured bean, a missing dependency), the *entire application fails to start* — Tomcat never even launches. In plain Spring MVC, the root context could succeed, Tomcat could start, and only when `DispatcherServlet.init()` runs does the child context failure become apparent (resulting in a running but broken application serving 503s).

---

## A Mental Model to Tie It Together

Think of the lifecycle as a relay race with four runners. `HttpServletBean` runs the first leg — it receives the baton from Tomcat, reads the init-params, and passes the baton up. `FrameworkServlet` runs the second and third legs — it creates the Spring context (second leg) and then handles every request by setting up and tearing down the ThreadLocal environment (third leg). `DispatcherServlet` runs the final leg — it uses the pre-built MVC machinery to dispatch the request to the right controller and render the response.

The key to remembering which class owns what is to think about *abstraction levels*: `HttpServletBean` is about bridging the Servlet API to Spring's property system. `FrameworkServlet` is about Spring context management and request lifecycle management. `DispatcherServlet` is about Spring MVC request dispatching. Each class does exactly its job and nothing more.
