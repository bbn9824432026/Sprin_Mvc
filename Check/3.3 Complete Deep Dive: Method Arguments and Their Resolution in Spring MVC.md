# Complete Deep Dive: Method Arguments and Their Resolution in Spring MVC

Let me guide you through one of the most intricate systems in Spring MVC - how controller method parameters get their values. This topic builds directly on what we've covered about request routing. Once Spring has determined which handler method should process a request, it faces a new challenge: that method might have five, ten, or even more parameters, each needing a value from somewhere. Understanding how Spring figures out where to get each value, and what happens when things go wrong, is essential for mastering Spring MVC.

## The Fundamental Problem That Argument Resolution Solves

Think about a typical controller method. You might write something like this without giving it much thought:

```java
@PostMapping("/products/{id}")
public Product updateProduct(
    @PathVariable Long id,
    @RequestBody Product product,
    @RequestHeader("Authorization") String token,
    Model model,
    HttpServletRequest request
) {
    // method body
}
```

You're asking Spring to invoke this method, but Java doesn't magically know how to populate these parameters. The method signature is just a declaration - Spring needs to look at each parameter, understand what it represents, fetch the appropriate value from the HTTP request or the Spring container, perform any necessary type conversions, and then invoke your method with all those values in the right order.

This is argument resolution. Every single parameter, regardless of its type or annotation, goes through a resolution process that determines its value. The system is designed to be extensible - Spring provides dozens of built-in resolvers for common patterns, and you can add your own for custom needs.

## How the Resolver Chain Works at Runtime

When Spring is about to invoke your controller method, it doesn't just blindly try to populate parameters. It uses a sophisticated chain-of-responsibility pattern with caching for performance. Let me walk you through what actually happens.

Spring maintains a list of `HandlerMethodArgumentResolver` implementations. Each resolver knows how to handle certain types of parameters. Some resolvers look for specific annotations like `@RequestParam` or `@PathVariable`. Others recognize certain types like `HttpServletRequest` or `Model`. The resolvers are registered in a specific order, and that order matters tremendously.

For each parameter in your method signature, Spring asks each resolver in sequence: "can you handle this parameter?" The question is answered by the resolver's `supportsParameter()` method. The first resolver to return true wins - it becomes responsible for providing the value for that parameter. Once a winner is found, Spring caches that decision. The cache key is the `MethodParameter` object, which uniquely identifies a specific parameter in a specific method. This means the chain walk only happens once per parameter across all requests. Subsequent requests find the cached resolver immediately.

This caching is critical for performance. Imagine an application with hundreds of endpoints, each with multiple parameters. Without caching, every request would walk through the entire resolver chain for every parameter. That would be expensive. With caching, the first request pays the cost of the chain walk, and every subsequent request gets O(1) lookup from the cache.

## The Resolver Chain Order and Why It Matters

The order of resolvers in the chain determines precedence when multiple resolvers could theoretically handle the same parameter. Spring registers its built-in resolvers in a carefully chosen order, and understanding this order helps you predict behavior and debug issues.

Annotation-based resolvers come first. These are resolvers that look for specific annotations on parameters. The `RequestParamMethodArgumentResolver` checks for `@RequestParam`. The `PathVariableMethodArgumentResolver` checks for `@PathVariable`. The `RequestResponseBodyMethodProcessor` checks for `@RequestBody`. These are registered early because annotation presence is a strong, explicit signal of intent.

After annotation-based resolvers come type-based resolvers. These match parameters purely by their type, without requiring any annotation. The `ServletRequestMethodArgumentResolver` handles many types: `HttpServletRequest`, `HttpServletResponse`, `HttpSession`, `Principal`, `InputStream`, `Locale`, and several others. It examines the parameter type and if it matches one of its supported types, it handles it.

The `ModelMethodProcessor` handles the `Model` interface. The `RedirectAttributesMethodArgumentResolver` handles `RedirectAttributes`. These are infrastructure objects that Spring provides to your handler method.

At the very end of the chain are the catch-all resolvers. These are crucial to understand because they provide default behavior for unannotated parameters. There's a second registration of `RequestParamMethodArgumentResolver`, but this time configured with `useDefaultResolution=true`. This version handles any simple type parameter that doesn't have an annotation. If you write a method parameter like `String name` without `@RequestParam`, this catch-all resolver treats it as if you had written `@RequestParam String name`. It looks for a request parameter with the same name as the Java parameter name.

Similarly, there's a second registration of `ServletModelAttributeMethodProcessor` with `supports=true`. This handles any complex type parameter without an annotation. If you have a parameter like `Product product` without `@ModelAttribute`, this catch-all resolver treats it as if you had written `@ModelAttribute Product product` and performs data binding.

This dual-registration pattern is elegant but can be confusing. The same resolver class appears twice in the chain with different configuration. The first appearance handles annotated parameters explicitly. The second appearance provides default behavior for unannotated parameters. The order ensures that explicit annotations always win.

## Understanding RequestParam in Depth

Let me start with one of the most commonly used argument types - request parameters. When you write `@RequestParam`, you're telling Spring to extract a value from the query string or form data.

The annotation has several attributes that control behavior. The `name` attribute (or its alias `value`) specifies which parameter to look for. If you don't specify it, Spring uses the Java parameter name, but this only works if you compile with debug information enabled so the parameter names are preserved in the bytecode.

The `required` attribute defaults to true, which means if the parameter is missing from the request, Spring throws `MissingServletRequestParameterException`, which gets converted to HTTP 400 Bad Request. This is strict by default - if you declare a parameter as required, Spring enforces that requirement before your method even executes.

You can make parameters optional by setting `required=false`. When the parameter is absent, Spring injects null. This works fine for reference types like `String` or `Integer`, but it's a trap for primitive types. If you write `@RequestParam(required=false) int age`, and the age parameter is missing, Spring tries to inject null for the int primitive. This causes a `NullPointerException` during unboxing. The solution is to use the boxed type `Integer` instead, or provide a `defaultValue`.

Speaking of `defaultValue`, when you specify it, the `required` attribute becomes irrelevant. A default value means the parameter is effectively always present - either from the request or from the default. This is the safest way to handle optional parameters because you never deal with null.

There's also special handling for `Optional` type. If your parameter type is `Optional<String>`, Spring recognizes this and treats the parameter as optional regardless of the `required` attribute. When the parameter is absent, you get `Optional.empty()`. When it's present, you get `Optional.of(value)`. This is cleaner than using `required=false` and checking for null.

For multi-valued parameters, you can use `List<String>` or `Set<String>`. When the request has multiple values for the same parameter name like `?tags=java&tags=spring&tags=mvc`, Spring collects all the values into the collection. This works seamlessly without any special configuration.

There's also a variant for collecting all request parameters into a Map. If you annotate a `Map<String, String>` parameter with `@RequestParam`, the `RequestParamMapMethodArgumentResolver` handles it and gives you all request parameters as a map. But here's a critical subtlety: if you have a `Map<String, String>` parameter without any annotation, a completely different resolver handles it - `MapMethodProcessor` gives you the model map, not the request parameters. The annotation completely changes the source of the data.

## PathVariable and URI Template Extraction

Path variables work differently from request parameters because they're part of the URL structure itself, not query parameters. When you define a mapping like `@GetMapping("/products/{id}")`, the curly braces define a URI template variable. Spring's pattern matching extracts these values during the routing process and stores them as a request attribute.

The `PathVariableMethodArgumentResolver` reads this request attribute to get the extracted values. If you write `@PathVariable Long id`, the resolver looks up the value for "id" from the URI template variables, then uses the type conversion system to convert the String value to Long.

Just like with request parameters, you can omit the name if your parameter name matches the template variable name, but this requires parameter names in the bytecode. You can also collect all path variables into a Map with `@PathVariable Map<String, String>`.

Type conversion is particularly important for path variables because they always come from the URL structure. If you have `@PathVariable Long id` and someone requests `/products/abc`, the String "abc" cannot be converted to Long. Spring throws `MethodArgumentTypeMismatchException`, which becomes HTTP 400. This is different from a URL that doesn't match the pattern at all, which would be 404. The URL matched the pattern structurally, but the value couldn't be converted to the expected type.

When you use regex constraints in your URI template like `/products/{id:\\d+}`, you're actually preventing this type conversion error at the routing level. A request to `/products/abc` won't match that pattern at all because "abc" doesn't match the regex `\d+`. You'll get 404 instead of 400, which makes sense because from the client's perspective, that URL doesn't exist in your API.

## RequestBody and Message Converter Integration

The `@RequestBody` annotation is fundamentally different from `@RequestParam` or `@PathVariable` because it's dealing with the request body, not individual parameters or URL segments. The entire process involves HTTP message converters, which are a separate subsystem in Spring MVC.

When Spring sees `@RequestBody Product product`, the `RequestResponseBodyMethodProcessor` takes over. It first examines the Content-Type header to understand what format the request body is in. Then it walks through the list of registered `HttpMessageConverter` implementations, asking each one: "can you read this type from this media type?" The first converter that returns true gets the job.

For JSON with Jackson on the classpath, `MappingJackson2HttpMessageConverter` handles it. The converter reads the request input stream - and this is crucial to understand - it can only be read once. HTTP request bodies are streams, not buffers. After the converter consumes the stream to deserialize your object, that stream is exhausted. If any code tries to read it again (a filter for logging, another resolver, anything), it will get empty bytes.

This one-time-read limitation is why Spring provides `ContentCachingRequestWrapper` for cases where you need to read the body multiple times. This wrapper buffers the entire body in memory so it can be read repeatedly. But it has a cost in memory usage for large payloads.

Validation with `@Valid` or `@Validated` on `@RequestBody` parameters happens after deserialization. Spring deserializes the body into your object, then runs JSR-303 validation on it. If validation fails, it throws `MethodArgumentNotValidException` immediately. Your method never executes. This is a critical difference from `@ModelAttribute`, which we'll discuss shortly.

Many developers try to add a `BindingResult` parameter after `@RequestBody` thinking it will work like `@ModelAttribute`. It doesn't. The `BindingResult` is simply ignored for `@RequestBody`. The exception is always thrown on validation failure. If you want to handle validation errors gracefully, you need to use `@ExceptionHandler` or `@ControllerAdvice` to catch `MethodArgumentNotValidException`.

## ModelAttribute and Data Binding

The `@ModelAttribute` annotation represents a completely different paradigm from `@RequestBody`. Instead of deserializing a structured format like JSON or XML, it performs data binding from individual request parameters using property names.

When Spring resolves a `@ModelAttribute` parameter, it follows a multi-step process. First, it looks for an existing instance in the model. This could have been put there by a `@ModelAttribute` method (a method annotated with `@ModelAttribute` that runs before your handler), or it could be coming from the session if you're using `@SessionAttributes`.

If no instance exists, Spring creates a new one. For simple cases, it calls the default constructor. For more complex scenarios, you can have a `@ModelAttribute` method that acts as a factory, creating and initializing the object before binding.

Once Spring has an instance - either found or created - it performs data binding. This is where `WebDataBinder` comes in. The binder takes all the request parameters and tries to match them to properties on your object by name. If you have a request parameter named "name", it calls `setName()`. If you have "price", it calls `setPrice()`. This happens through reflection and property descriptors.

The binding process supports nested properties with dot notation. A parameter like `address.street` sets the street property on the address property of your object. It supports indexed access for lists: `items[0].quantity` sets the quantity of the first item in an items list. The binder is quite sophisticated in what it can handle.

Type conversion happens during binding. Request parameters are strings, but your object properties might be int, LocalDate, enum, or any other type. The same conversion service used for simple parameters applies here. If conversion fails for a particular field, it doesn't throw an exception immediately - it records a `FieldError` in the `BindingResult`.

After binding, if you have `@Valid` or `@Validated` on the parameter, Spring runs validation. Again, validation errors go into the `BindingResult` rather than throwing an exception - but only if a `BindingResult` parameter is present.

This brings us to one of the most important rules about `BindingResult`: it must immediately follow its corresponding `@ModelAttribute` parameter in the parameter list. The `ErrorsMethodArgumentResolver` that handles `BindingResult` parameters looks at the immediately preceding parameter. If that preceding parameter is not a model attribute, it throws `IllegalStateException`. You cannot have any other parameter between them.

The reason for this strict adjacency requirement is that a method might have multiple `@ModelAttribute` parameters, each needing its own `BindingResult`. The only way to unambiguously associate each result with its corresponding model attribute is by position.

After everything is resolved and bound, Spring adds the model attribute to the model under its name. Then it adds the `BindingResult` to the model under a special key that includes the attribute name. The view can access both to display the bound object and any errors that occurred.

## The BindingResult Difference Between RequestBody and ModelAttribute

This difference trips up many developers and is worth emphasizing. With `@ModelAttribute`, the presence of a `BindingResult` parameter changes the error handling behavior. If you have a `BindingResult`, validation errors are captured there and your method executes. You can check `result.hasErrors()` and decide what to do - maybe return to the form view with error messages.

With `@RequestBody`, the `BindingResult` parameter is basically ignored for validation purposes. If validation fails, `MethodArgumentNotValidException` is thrown regardless of whether you have a `BindingResult` parameter. The exception is thrown before your method executes, so you never get a chance to check the result.

Why this difference? It comes down to the different use cases. `@ModelAttribute` is typically used for form submissions where you want to show the form again with validation errors. The form contains the user's input, and you want to highlight which fields are invalid. Capturing errors in `BindingResult` and re-rendering the form is the natural flow.

`@RequestBody` is typically used for REST APIs where the client is code, not a human user. When validation fails, the appropriate response is to reject the request with a 400 status and error details in a structured format. Throwing an exception that gets handled by exception handlers is the natural flow for this use case.

Understanding this distinction prevents a lot of confusion when validation doesn't work the way you expect based on your experience with the other annotation.

## Type-Based Resolution for Infrastructure Objects

Some parameters don't need any annotation at all because their type uniquely identifies what they represent. The `ServletRequestMethodArgumentResolver` handles a whole family of types related to the servlet API and request processing infrastructure.

If you have an `HttpServletRequest` parameter, Spring injects the current request object. Same for `HttpServletResponse`, `HttpSession`, `Principal`, `InputStream`, `OutputStream`, `Locale`, `TimeZone`, and several other types. The resolver checks the parameter type and if it matches one of these known types, it provides the appropriate object.

This is convenient because these objects are frequently needed, and requiring annotations for them would be verbose. The type itself is sufficient to express intent. There's no ambiguity about what `HttpServletRequest` should contain.

Similarly, `Model` and `ModelMap` are resolved by `ModelMethodProcessor` without any annotation. If you need to add attributes to the model for a view to render, you just declare a `Model` parameter and Spring gives you the current model.

`RedirectAttributes` is handled by its own resolver. This is a special interface for passing data through a redirect. You add flash attributes that survive the redirect and become model attributes in the redirected-to handler.

These type-based resolutions are why you can write very clean handler method signatures without cluttering every parameter with annotations.

## The Catch-All Resolvers and Default Behavior

At the end of the resolver chain are the catch-all resolvers that provide sensible defaults for unannotated parameters. This is where Spring's "convention over configuration" philosophy shows through.

For simple types - String, int, long, boolean, enums, and so on - if you don't annotate the parameter, Spring treats it as a request parameter. The resolver uses the Java parameter name to look up a request parameter with that name. So `void method(String query)` behaves as if you wrote `@RequestParam String query`. This works beautifully when your parameter names match your request parameter names, which is common in practice.

For complex types - any type that's not considered "simple" - Spring treats unannotated parameters as model attributes. So `void method(Product product)` behaves as if you wrote `@ModelAttribute Product product`, and data binding happens automatically.

This automatic behavior is convenient but can be surprising if you don't expect it. It means adding a parameter to your method signature might trigger data binding or parameter extraction even without any annotation. Understanding what the catch-all resolvers do helps you predict this behavior.

The tricky case is `Map`. An unannotated `Map<String, String>` parameter is resolved by `MapMethodProcessor` and gives you the model map, not request parameters. If you want request parameters, you must use `@RequestParam Map<String, String>`. The annotation changes the source completely. This is a common source of bugs when developers assume an unannotated Map will contain request parameters.

## Type Conversion and Its Failure Modes

Request parameters and path variables are always strings in the HTTP protocol, but your Java parameters might be integers, dates, enums, or custom types. Spring's type conversion system bridges this gap, but it's important to understand how it works and what happens when conversion fails.

The `ConversionService` is responsible for converting between types. Spring registers many built-in converters: string to number, string to boolean, string to enum, string to date/time types, and many others. When a resolver needs to convert a value, it asks the conversion service.

For numbers, the conversion is straightforward. The string "42" becomes the integer 42 or the long 42L depending on the target type. But what if the string is "abc" and you're trying to convert to Long? The converter throws `ConversionFailedException`, which gets wrapped in `MethodArgumentTypeMismatchException`, which gets handled by Spring's exception resolver to produce HTTP 400 Bad Request.

For enums, the conversion uses `Enum.valueOf()`, which is case-sensitive. If you have an enum `Status` with values `ACTIVE` and `INACTIVE`, the string "ACTIVE" converts successfully, but "active" throws a conversion exception. You get a 400 error. If you need case-insensitive enum binding, you need a custom converter.

For dates and times, things get more complex. There's no universal string format for dates. The string "2024-03-15" could be parsed as a LocalDate with ISO format, but the string "03/15/2024" won't parse without additional configuration. You typically use `@DateTimeFormat` annotations on your fields or parameters to specify the expected format. Without this, default converters might not handle your date strings correctly.

When conversion fails, the behavior depends on where the parameter came from. For `@RequestParam` and `@PathVariable`, conversion failure throws an exception that becomes HTTP 400. The request was malformed from the perspective of your API's contract.

For `@ModelAttribute` data binding, conversion failures for individual fields are recorded as field errors in the `BindingResult` rather than throwing exceptions. The binding process continues for other fields. This allows you to show users which specific fields had problems rather than failing the entire binding.

## Session Attributes and the Confusion Around Them

There are two different annotation patterns involving sessions, and they're often confused: `@SessionAttributes` at the class level and `@SessionAttribute` on method parameters.

`@SessionAttributes` is a class-level annotation that declares which model attribute names should be stored in the HTTP session. When you put an attribute with one of those names into the model, Spring automatically stores it in the session. On subsequent requests, Spring retrieves it from the session back into the model. This is used for multi-step forms where you're building up an object across several requests.

When you use `@SessionAttributes`, you're working through the model. You add attributes to the model in your handler methods. Spring sees that those attribute names are declared in `@SessionAttributes` and synchronizes them to the session. You don't directly manipulate the session.

To clear session attributes managed by `@SessionAttributes`, you use the `SessionStatus` parameter. Calling `sessionStatus.setComplete()` clears all attributes declared in `@SessionAttributes` from the session. This is typically done after successfully completing a multi-step process.

`@SessionAttribute` on a method parameter is completely different. It's for reading a specific attribute directly from the HTTP session, not from the model. This is useful when you have session data that was set by a completely different part of your application, not managed by `@SessionAttributes`. The resolver reads directly from `request.getSession().getAttribute()`.

The naming similarity between these two features causes confusion, but they serve different purposes and work differently. One is about synchronizing model attributes to session storage. The other is about reading arbitrary session attributes directly.

## Creating Custom Argument Resolvers

While Spring provides resolvers for most common needs, you sometimes want custom argument resolution for your own patterns. Maybe you have a pagination scheme that you want to inject as a parameter, or you want to extract and decode a JWT token into a custom object.

Creating a custom resolver involves implementing the `HandlerMethodArgumentResolver` interface. You implement `supportsParameter()` to indicate which parameters you handle, and `resolveArgument()` to actually produce the value.

The `supportsParameter()` method receives a `MethodParameter` object that describes the parameter. You can check its type, its annotations, its name, its position - whatever you need to determine if this is a parameter you should handle. Return true if you claim it, false if you don't.

The `resolveArgument()` method is where you produce the actual value. You receive the `NativeWebRequest` which gives you access to the underlying request and response. You receive the `ModelAndViewContainer` which holds the model and metadata about view handling. You might receive a `WebDataBinderFactory` for cases where you need to create a binder for complex binding scenarios.

Your resolver can do whatever makes sense: read headers, read request attributes, access the request body, consult the session, look up beans from the Spring container, anything. You construct and return the object that should be injected as that parameter's value.

To register your custom resolver, you implement `WebMvcConfigurer` and override `addArgumentResolvers()`. You add your resolver to the list. It's important to understand that resolvers you add this way are appended to the end of the chain, after all built-in resolvers. This means if a built-in resolver claims the parameter first, your resolver never gets a chance.

If you need your resolver to run before built-ins, you can't use `addArgumentResolvers()`. You'd need to access the `RequestMappingHandlerAdapter` bean directly and replace its entire resolver list. This is rarely necessary but occasionally required for very specific scenarios.

## The Resolver Cache and Performance Implications

Let me explain the caching mechanism in more detail because it has important performance implications. The first time Spring invokes a handler method, it must walk the resolver chain for each parameter to find the resolver that handles it. For a method with five parameters, that's potentially five walks through a chain of thirty resolvers.

Spring caches the winning resolver for each `MethodParameter`. The `MethodParameter` object uniquely identifies a parameter by its declaring class, method, and position. After the first request, when Spring needs to resolve that same parameter again, it checks the cache first. If there's a cached resolver, it uses it directly without walking the chain. This reduces the cost from O(n) to O(1) where n is the number of resolvers.

The cache is a `ConcurrentHashMap` that grows as your application handles different endpoints. Each distinct parameter in each handler method gets its own cache entry. For an application with hundreds of endpoints and thousands of total parameters, the cache could have thousands of entries. The memory cost is negligible - each entry is just a reference to the resolver object, and resolvers are singletons. The performance benefit is substantial.

One implication is that the resolver chain is effectively frozen after the first request to any given parameter. If you had logic that tried to dynamically change which resolver handles a parameter based on runtime conditions, it wouldn't work after the first request because of caching. This is by design - parameter resolution is meant to be deterministic based on the parameter's declaration, not on runtime state.

## Understanding Error Scenarios and Their HTTP Status Codes

Different argument resolution failures produce different HTTP status codes, and understanding which failure produces which status helps you debug issues and design proper error responses.

When a required `@RequestParam` is missing, you get `MissingServletRequestParameterException`, which becomes HTTP 400 Bad Request. This is a client error - they didn't provide a required parameter.

When type conversion fails for `@RequestParam` or `@PathVariable`, you get `MethodArgumentTypeMismatchException`, which also becomes HTTP 400. Again, client error - they provided a value that couldn't be converted to the expected type.

When `@RequestBody` deserialization fails because the JSON is malformed, you get `HttpMessageNotReadableException`, which becomes HTTP 400. The client sent a body that couldn't be parsed.

When `@RequestBody` deserialization succeeds but validation fails, you get `MethodArgumentNotValidException`, which becomes HTTP 400. The body was parseable but didn't meet validation constraints.

When a parameter type is completely unsupported and no resolver claims it, you get `IllegalStateException`, typically with HTTP 500 Internal Server Error. This is actually a programming error in your controller, not a client error, though it manifests as a runtime exception.

Understanding these error codes helps you interpret what went wrong when a request fails, and it helps you design appropriate error responses for your API clients.

## Special Handling for Certain Parameter Patterns

Some parameter patterns have special semantics worth understanding. When you inject `HttpServletResponse` or `OutputStream` or `Writer`, Spring sets `requestHandled=true` in the model and view container. This signals that you're writing the response directly, so Spring shouldn't attempt view resolution. This is the same flag that `@ResponseBody` sets.

This means if your method returns a view name but also has an `OutputStream` parameter, the view name is ignored. Spring assumes you wrote the response directly via the stream.

The `HttpSession` parameter is interesting because requesting it will create a session if one doesn't exist. Just declaring `HttpSession session` as a parameter causes session creation. If you want to only read from an existing session without creating one, you should use `HttpServletRequest` and call `getSession(false)` which returns null if no session exists.

Parameters of type `Principal` get special handling because they might come from Spring Security or servlet container authentication. The resolver checks `request.getUserPrincipal()`. If you're using Spring Security, the `Principal` will be the authenticated user. If you need the full `Authentication` object, you should use Spring Security's authentication injection, not the generic `Principal` parameter.

## The Relationship Between Arguments and Return Values

It's worth noting that many argument resolvers also serve as return value handlers. The `RequestResponseBodyMethodProcessor` we discussed handles both `@RequestBody` parameters and `@ResponseBody` return values. The `HttpEntityMethodProcessor` handles both `HttpEntity<?>` parameters and `ResponseEntity<?>` return values.

This dual role makes sense because these concepts are symmetric. If you can read something from a request into a parameter, you can write something from a return value into a response. The same message converters that deserialize request bodies can serialize response bodies. The same content negotiation logic applies in both directions.

Understanding this symmetry helps you see the bigger picture of how Spring MVC processes requests end-to-end. Arguments come in through resolvers, your method executes, return values go out through handlers, all coordinated by the dispatcher servlet.

# Final Summary for Deep Understanding and Recall

Let me synthesize the key principles that will stick with you beyond memorizing specific resolver names.

**Argument resolution is a chain-of-responsibility with caching**. Each parameter is offered to resolvers in sequence until one claims it. That decision is cached so subsequent requests for the same parameter skip the chain walk entirely. This architecture balances flexibility with performance - you can have dozens of resolvers without making every request expensive.

**Annotation presence changes which resolver handles a parameter, not just how it's configured**. A `Map` parameter without annotation goes to `MapMethodProcessor` and gets the model. The same `Map` with `@RequestParam` goes to `RequestParamMapMethodArgumentResolver` and gets request parameters. The annotation completely changes the source of data, not just options on the same source.

**BindingResult adjacency is a hard rule, not a suggestion**. The `BindingResult` must immediately follow its corresponding `@ModelAttribute` with no parameters in between. Even infrastructure parameters like `HttpServletRequest` can't separate them when the servlet request comes after the model attribute. Any gap causes `IllegalStateException` at runtime. The resolver literally looks at the preceding parameter index to find the bound object.

**RequestBody validation throws exceptions, ModelAttribute validation populates BindingResult**. This isn't a bug or inconsistency - it's by design for different use cases. REST APIs with `@RequestBody` should fail fast with exceptions that error handlers convert to structured error responses. Form submissions with `@ModelAttribute` should capture errors to re-display the form with user input and error messages. The presence of `BindingResult` changes behavior for model attributes but is ignored for request body.

**Type conversion failures for parameters become 400 errors, for fields become FieldErrors**. When Spring can't convert "abc" to Long for a `@PathVariable`, that's a malformed request - 400 status. When Spring can't convert "abc" to an integer field during `@ModelAttribute` binding, that's a validation issue - field error in the result. The same conversion service produces different error handling based on context.

**The catch-all resolvers at the end of the chain provide implicit behavior**. Unannotated simple types become request parameters. Unannotated complex types become model attributes with data binding. This is convenient when it matches your intent, surprising when it doesn't. Explicitly annotating parameters makes behavior predictable and self-documenting.

**Custom resolvers added via configuration go after all built-ins**. You cannot override Spring's handling of `@RequestParam` or `@PathVariable` by adding a custom resolver through `WebMvcConfigurer`. Your resolver only sees parameters that none of the built-ins claimed. If you need to intercept before built-ins, you must manipulate the adapter's resolver list directly, which is rarely necessary.

**Session attributes come in two flavors with completely different mechanics**. Class-level `@SessionAttributes` synchronizes model attributes to session storage automatically. Method parameter `@SessionAttribute` reads directly from session without model involvement. Despite similar names, they solve different problems. One is about transparent session backing for model attributes across requests. The other is about accessing arbitrary session data.

**Request body streams are single-use**. After `@RequestBody` consumes the input stream to deserialize your object, that stream is exhausted. Filters or other code that tries to read it afterward gets nothing. If you need to read the body multiple times, you must use `ContentCachingRequestWrapper` to buffer it, accepting the memory cost for large payloads.

**Resolver order encodes priority and fallback strategy**. Annotation-specific resolvers first means explicit annotations always win. Type-specific resolvers next means infrastructure objects don't need annotations. Catch-all resolvers last provide sensible defaults for common cases. This ordering is why you can mix annotated and unannotated parameters in the same method signature and get predictable behavior.

Understanding argument resolution deeply means understanding that each parameter goes through a deterministic process: find the responsible resolver based on annotations and type, invoke that resolver to produce a value with type conversion and validation as needed, cache the resolver choice for performance, and handle failures with appropriate HTTP status codes. This process runs for every parameter on every request, but caching ensures it's fast after the first invocation. Master this flow and you master controller method design in Spring MVC.
