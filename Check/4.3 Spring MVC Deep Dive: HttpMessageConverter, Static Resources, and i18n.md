## Topic 4.3 — HttpMessageConverter

### The Core Idea: What Problem Does This Solve?

When a browser or API client sends a JSON body to your Spring controller, *something* has to turn those raw bytes into a Java object. And when your controller returns a `Product` object, *something* has to turn that Java object back into JSON bytes. That "something" is the `HttpMessageConverter` system.

Think of it as a two-directional bridge — reading converts HTTP body bytes into Java objects (deserialization), and writing converts Java objects into HTTP body bytes (serialization). Every `@RequestBody`, `@ResponseBody`, `ResponseEntity`, and `HttpEntity` flows through this mechanism.

### The Interface Contract

The `HttpMessageConverter<T>` interface defines four fundamental questions a converter must answer:

**"Can you read this?"** — `canRead(Class<?> clazz, MediaType mediaType)` asks whether the converter can deserialize a given Java type from a given media type (like `application/json`).

**"Can you write this?"** — `canWrite(Class<?> clazz, MediaType mediaType)` asks whether it can serialize a given type into a given media type.

**"What media types do you support?"** — `getSupportedMediaTypes()` declares the converter's capabilities.

**"Read/write it then."** — `read()` and `write()` do the actual work.

There's also `GenericHttpMessageConverter<T>`, which extends the base interface to handle *parameterized generic types* like `List<Product>`. This is critical — a plain `HttpMessageConverter` only sees the raw `Class`, so it can't distinguish `List<Product>` from `List<String>`. The generic variant receives the full `Type` object, which carries that generic parameter information.

### How Converter Selection Works: The Chain

Spring registers a chain of converters, and when a response needs to be written, Spring walks this chain asking each converter `canWrite(returnType, mediaType)`. **The first converter that returns `true` wins.** This ordering is critical and is one of the biggest source of subtle bugs.

The default registration order (which you must know cold for exams) is:

`ByteArrayHttpMessageConverter` → `StringHttpMessageConverter` → `ResourceHttpMessageConverter` → `SourceHttpMessageConverter` → `AllEncompassingFormHttpMessageConverter` → `MappingJackson2HttpMessageConverter` → `MappingJackson2XmlHttpMessageConverter` → `Jaxb2RootElementHttpMessageConverter`

The `canWrite()` check uses `isCompatibleWith()`, not an exact match. This means a converter declaring `*/*` (all media types) can technically claim to write *any* media type — which is exactly how `ByteArrayHttpMessageConverter` and `StringHttpMessageConverter` behave. The more specific match (like `application/json`) beats `*/*` during content negotiation because Spring sorts compatible pairs by specificity.

### The AbstractHttpMessageConverter Template

Most converters extend `AbstractHttpMessageConverter<T>`, which implements the template method pattern. You override three methods: `supports(Class<?>)` to declare which Java types you handle, `readInternal()` to do deserialization, and `writeInternal()` to do serialization. The base class handles the boring parts — setting `Content-Type` headers, setting `Content-Length`, flushing the output stream.

### Jackson: The Default JSON Converter

`MappingJackson2HttpMessageConverter` is the workhorse of most Spring REST APIs. It supports `application/json` and `application/*+json` (wildcard JSON subtypes like `application/vnd.api+json`).

The real work happens in `AbstractJackson2HttpMessageConverter`. During **deserialization**, it reads the charset from the `Content-Type` header, creates a `JsonParser`, and calls `objectMapper.readValue(inputStream, javaType)`. If the JSON is malformed, it throws `HttpMessageNotReadableException` — which Spring automatically translates to **HTTP 400 Bad Request** via `DefaultHandlerExceptionResolver`.

During **serialization**, it creates a `JsonGenerator` and calls `objectMapper.writer().writeValue(generator, value)`. If serialization fails (say, a circular reference), it throws `HttpMessageNotWritableException` — which becomes **HTTP 500** because Spring does *not* handle this automatically. This asymmetry (400 vs 500) is a common exam trap.

#### The ObjectMapper Singleton

The `ObjectMapper` is a **singleton shared across all requests**. It is thread-safe for reading and writing, but you must configure it only during application startup. Calling `objectMapper.configure()` at request time in a multi-threaded environment creates a race condition. In Spring Boot, you configure it through `Jackson2ObjectMapperBuilderCustomizer` or by declaring an `ObjectMapper` bean.

Key configuration decisions to understand:

`WRITE_DATES_AS_TIMESTAMPS=false` plus `JavaTimeModule` is the right way to handle Java 8+ date/time types as ISO strings rather than epoch numbers. `FAIL_ON_UNKNOWN_PROPERTIES=false` prevents 400 errors when clients send extra fields. `NON_NULL` inclusion means null fields are simply omitted from responses, reducing payload size.

#### Jackson Annotations You Must Know

`@JsonProperty("product_name")` renames the field in JSON. `@JsonIgnore` excludes a field from **both** serialization and deserialization — a common trap when developers intend to hide only the output but accidentally prevent the field from being set from input too. Use `@JsonProperty(access = WRITE_ONLY)` or `READ_ONLY` for one-directional control.

`@JsonView` is fascinating: it lets you annotate fields with marker interfaces, and then annotate controller methods with `@JsonView(SomeView.class)`. The converter detects the view annotation on the method, calls `objectMapper.writerWithView(viewClass)`, and only serializes fields whose `@JsonView` annotation is assignable from the requested view. The hierarchy works through inheritance — a field annotated with `Views.Public.class` is included when any view that extends `Public` is active.

`@JsonTypeInfo` + `@JsonSubTypes` handles polymorphism: when a `Payment` abstract class needs to deserialize into `CardPayment` or `BankTransfer` based on a `"type"` JSON field, these annotations make Jackson do the right thing automatically.

### The StringHttpMessageConverter Charset Trap

This one bites developers constantly. The default charset for `StringHttpMessageConverter` is **ISO-8859-1**, not UTF-8. If you return a `String` containing `"café"` and the client expects UTF-8, the response will be garbled. Spring Boot auto-configures UTF-8, but in plain Spring MVC you must explicitly replace it:

```java
// In extendMessageConverters:
converters.removeIf(c -> c instanceof StringHttpMessageConverter);
converters.add(0, new StringHttpMessageConverter(StandardCharsets.UTF_8));
// Add at position 0 to preserve priority — StringHttpMessageConverter
// supports */* and would otherwise interfere with other converters
```

Also worth noting: `StringHttpMessageConverter` supports `*/*`, which means it can match *any* accept type. When a `@RestController` method returns `String` and the client sends `Accept: application/json`, the specificity rules prevent Jackson from winning — but converter ordering matters here too.

### configureMessageConverters vs extendMessageConverters

This is one of the most important distinctions. `configureMessageConverters(List)` **replaces the entire default chain** — if you add converters here but don't explicitly add Jackson, your REST endpoints will return 406 for JSON requests. `extendMessageConverters(List)` **adds to the existing chain**, preserving Jackson and all defaults. You should almost always use `extendMessageConverters`.

### Custom Converters

Writing a custom converter is straightforward: extend `AbstractHttpMessageConverter<T>`, declare your supported media types in the constructor, implement `supports()` to declare which Java types you handle, then implement `readInternal()` and `writeInternal()`. The YAML converter in the document is a perfect example — it wraps Jackson's YAML support into a Spring converter, supporting `application/yaml` and `text/yaml` media types.

### ResponseBodyAdvice

`ResponseBodyAdvice` lets you intercept the response body just before the converter writes it. The `beforeBodyWrite()` method receives the return value, the selected content type, and the converter that will be used. You can modify the body (wrapping it in an envelope), add response headers, or return `null` to suppress the body entirely. It's the right place for cross-cutting concerns like wrapping every response in a `{"data": ..., "requestId": "..."}` envelope.

### Jackson XML vs JAXB: The Ordering Problem

Both `MappingJackson2XmlHttpMessageConverter` and `Jaxb2RootElementHttpMessageConverter` serve XML, but the default registration puts Jackson XML *first*. Since Jackson XML's `supports(Class<?>)` returns `true` for any class (Jackson can serialize anything), it always wins over JAXB even when your class has `@XmlRootElement`. If you want JAXB to handle annotated classes, you must manually reorder in `extendMessageConverters()` to place JAXB before Jackson XML.

## Topic 4.4 — Static Resources, Caching, and Resource Handling

### The Architecture: More Than "Just Files"

Serving static files in Spring MVC is not a passthrough. It's a full pipeline involving resolver chains, transformer chains, HTTP caching headers, conditional GET support, and version-based cache busting. Understanding this pipeline explains why Spring's static resource handling is production-grade.

The pipeline works like this: a request for `/static/css/app.css` hits `DispatcherServlet`, finds no `@RequestMapping` match, then falls to `SimpleUrlHandlerMapping` (registered at near-lowest priority) which routes `/**` static patterns to `ResourceHttpRequestHandler`. That handler then orchestrates the full pipeline.

### ResourceHttpRequestHandler: Step by Step

The handler runs through a precise sequence. First, it validates the HTTP method — only GET and HEAD are allowed for static resources. Then it runs the **ResourceResolver chain** to find the actual `Resource` object. If nothing is found, it calls `response.sendError(404)` directly — bypassing Spring's exception handler chain entirely (more on this important trap shortly). Then it determines the media type from the file extension using `MediaTypeFactory`. Then it checks conditional GET headers to potentially return 304. Then it applies the **ResourceTransformer chain**. Finally, it streams the content to the response.

### The ResourceResolver Chain

Resolvers are tried in order; the first one to return a non-null `Resource` wins. The typical production chain looks like:

**CachingResourceResolver** wraps everything and caches resolved `Resource` objects in memory (a `ConcurrentHashMap`). Once a resource has been resolved, subsequent requests skip the entire chain and return the cached result directly.

**EncodedResourceResolver** checks `Accept-Encoding` header. If the client accepts gzip and `app.css.gz` exists, it returns that file with `Content-Encoding: gzip`. The `Content-Type` remains `text/css` — the encoding header tells the browser to decompress, but the type is still the original media type. This lets you pre-compress assets at build time rather than compressing on every request.

**VersionResourceResolver** handles versioned URLs. With `addContentVersionStrategy("/**")`, it computes an MD5/SHA hash of the file content and embeds it in the filename: `app-a1b2c3d4.css`. When a request arrives for `/css/app-a1b2c3d4.css`, it extracts the hash, derives the real filename (`app.css`), loads it, recomputes the hash, verifies they match, and returns the resource. Hashes are computed on first access (not startup) and then cached.

**PathResourceResolver** is the fallback — it does the actual file lookup by trying each configured location in order and returning the first readable file it finds.

### HTTP Caching: Getting This Right in Production

HTTP caching has two distinct mechanisms that work together. **Cache-Control max-age** tells the browser to serve from its local cache for N seconds without even making a network request. **Conditional GET with Last-Modified** gives the browser a way to check if a resource changed after the max-age expires, potentially getting a lightweight 304 response instead of re-downloading.

The production pattern for versioned assets is:

```java
.setCacheControl(
    CacheControl.maxAge(Duration.ofDays(365))
        .cachePublic()
        .immutable()
    // immutable tells browsers: never revalidate during max-age.
    // Safe ONLY because the URL changes when content changes.
)
```

The `immutable` directive is the key insight. It tells browsers to cache for 365 days and *never* bother sending a conditional GET. This is safe precisely because `VersionResourceResolver` changes the URL when the content changes — the browser sees a new URL and fetches fresh content. **Using `immutable` with non-versioned URLs is a serious bug** — users would be stuck with cached content for up to a year after updates.

For resources that aren't versioned (like user-uploaded images), you use `Last-Modified` with a shorter max-age instead:

```java
.setCacheControl(CacheControl.maxAge(Duration.ofDays(30)).cachePublic())
.setUseLastModified(true)
// Enables 304 responses when files haven't changed
```

One important subtlety: `Last-Modified` comparison uses **second-level precision** (the HTTP date format `Mon, 04 Mar 2024 10:00:00 GMT` has no sub-second component). If a file is modified within the same second as the cached timestamp, Spring may incorrectly return 304. This is an HTTP specification limitation.

### `resourceChain(true)` vs `resourceChain(false)`

The boolean parameter is misleading — it doesn't enable or disable the chain itself. `resourceChain(true)` wraps the chain with `CachingResourceResolver`, enabling in-memory caching of resolved resources. `resourceChain(false)` skips that wrapper, so every request walks the full chain. Use `false` during development so you see file changes immediately; use `true` in production for performance.

### ResourceTransformer Chain

Transformers process resource content *after* resolution but *before* it's sent to the client. They operate purely in memory — the original file on disk is never touched.

`CssLinkResourceTransformer` rewrites CSS `@import` and `url()` references to their versioned equivalents. So if your `app.css` says `@import "base.css"`, the transformer rewrites it to `@import "base-a1b2c3d4.css"`. This is critical for **deep cache busting** — without it, a browser might cache the versioned `app.css` forever but still fetch the un-versioned `base.css` from the server. Important dependency: `CssLinkResourceTransformer` requires `VersionResourceResolver` to be in the chain *before* it, because it needs to know what versioned URL to generate.

### ResourceUrlProvider and Template Integration

`ResourceUrlProvider` is the bridge between the resolver chain and your templates. In Thymeleaf, when you write `th:src="@{/css/app.css}"`, Thymeleaf calls `ResourceUrlProvider.getForLookupPath("/css/app.css")` internally. That method computes the hash and returns `/css/app-a1b2c3d4.css`. The rendered HTML gets the versioned URL automatically. Spring Boot auto-registers `ResourceUrlEncodingFilter` to make this work transparently.

### The Static Resource 404 Trap

When `ResourceHttpRequestHandler` can't find a file, it calls `response.sendError(HttpServletResponse.SC_NOT_FOUND)` directly. This **completely bypasses** Spring's `@ExceptionHandler` and exception resolver chain. You cannot catch a missing static resource in `@ExceptionHandler(NoHandlerFoundException.class)` — that exception is for unmatched URL mappings, not missing files. Custom error pages for missing static resources must be configured at the Servlet container level or via Spring Boot's `ErrorController`.

### WebJars and WebJarsResourceResolver

WebJars package JavaScript/CSS libraries as JAR files, allowing you to manage front-end dependencies through Maven or Gradle. The `WebJarsResourceResolver` is clever: it reads the version from `META-INF/maven/.../pom.properties` inside the JAR and constructs the full versioned path. This means your template can reference `/webjars/bootstrap/css/bootstrap.min.css` (no version), and the resolver automatically finds `/webjars/bootstrap/5.3.0/css/bootstrap.min.css`. When you upgrade Bootstrap, just update the Maven dependency — no template changes needed.

## Topic 4.5 — Internationalization (i18n)

### The Four-Component Architecture

i18n in Spring MVC is not just loading property files. It's a coordinated system of four components that must work together precisely.

**LocaleResolver** determines which `Locale` is active for a given request. **LocaleContextHolder** is a `ThreadLocal` that stores the resolved locale so it's accessible anywhere during the request processing thread. **MessageSource** resolves message codes to locale-specific text. **LocaleChangeInterceptor** allows users to change their locale by adding a query parameter.

### LocaleResolver Variants: Choosing the Right One

**AcceptHeaderLocaleResolver** reads the browser's `Accept-Language` header. It's stateless and HTTP-standard — no server-side storage needed. The matching is intelligent: it tries exact matches (language + country) first, then language-only matches, then falls back to `defaultLocale`. The critical limitation: `setLocale()` throws `UnsupportedOperationException` because you cannot change what the browser sends in its headers. **Never pair this with `LocaleChangeInterceptor`** — you'll get a runtime exception.

**SessionLocaleResolver** stores the locale in `HttpSession`. It supports `setLocale()` by writing to the session. The locale persists for the session lifetime and falls back to `defaultLocale` (if configured) or the `Accept-Language` header on first access.

**CookieLocaleResolver** stores the locale in a browser cookie. The `cookieMaxAge` property controls persistence: `-1` creates a session cookie (dies when browser closes), and a positive value in seconds creates a persistent cookie that survives browser restarts. This is ideal for stateless architectures or mobile apps.

**FixedLocaleResolver** always returns the same locale regardless of anything — useful for single-language internal tools where you never want locale switching.

### The Critical Locale Change Timing Trap

Here's a subtle but important behavior: when `LocaleChangeInterceptor` fires in `preHandle()` and calls `localeResolver.setLocale()`, it stores the new locale in the session or cookie. But `FrameworkServlet.processRequest()` already ran *before* interceptors and already set `LocaleContextHolder` from the *previous* session value. So for the **current request**, `LocaleContextHolder.getLocale()` still returns the old locale. Messages in the current response are still rendered in the old language. The new locale becomes active starting with the **next request**. This surprises developers who expect an immediate language switch on the same page load.

### LocaleContextHolder and Thread Inheritance

`LocaleContextHolder` uses `InheritableThreadLocal` by default. This means child threads spawned from the request processing thread automatically inherit the current locale context. This is important for async operations — if your request spawns a thread to send a localized email, that thread will have the correct locale without any extra work. This contrasts with `RequestContextHolder`, which uses plain `ThreadLocal` and is *not* inherited by child threads.

### MessageSource: Configuration Details That Matter

The MessageSource bean **must be named exactly `"messageSource"`**. Spring's `ApplicationContext` looks for this specific name. Naming it anything else means Spring won't find it for automatic locale resolution — a silent failure where all messages revert to raw codes.

`setDefaultEncoding("UTF-8")` is **mandatory**. Java `.properties` files default to ISO-8859-1. Without this setting, French accents, German umlauts, Arabic characters, Chinese characters — all will be corrupted. This is a silent bug: the application starts without error, but internationalized content looks like garbage.

`setFallbackToSystemLocale(false)` is recommended for predictability. With `true` (the default), the fallback chain is: `messages_fr_FR.properties` → `messages_fr.properties` → JVM system locale file → `messages.properties`. If your production server has a Japanese JVM locale and `messages_ja.properties` exists, French users might get Japanese error messages. With `false`, the chain is simply: `messages_fr_FR.properties` → `messages_fr.properties` → `messages.properties`.

**`ResourceBundleMessageSource`** uses Java's `ResourceBundle` mechanism — fast, but requires restart for changes. **`ReloadableResourceBundleMessageSource`** polls files for changes (configurable with `setCacheSeconds(n)`) and supports filesystem locations, allowing runtime message updates without restart.

### Message Code Fallback and MessageFormat

When you call `messageSource.getMessage("welcome", new Object[]{"Alice"}, Locale.FRENCH)`, Spring first tries `messages_fr.properties`, then `messages.properties`, then throws `NoSuchMessageException` (or returns the code itself if `useCodeAsDefaultMessage=true`).

The argument replacement uses `java.text.MessageFormat`. `{0}` gets the first argument, `{1}` the second, and so on. `MessageFormat` also supports format specifiers: `{0,number,currency}` formats as locale-sensitive currency, `{0,date,short}` as a short date. Watch out for single quotes in messages — MessageFormat uses single quotes for escaping, so a literal apostrophe must be written as `''` (two single quotes). This causes confusing bugs like `"You're welcome"` rendering as `"Youre welcome"` — the `'` is eaten as an escape character.

### Validation Message Code Hierarchy

When bean validation fails (say, `@NotNull` on the `name` field of a `Product` class), Spring's `DefaultMessageCodesResolver` generates a list of codes from most specific to least specific:

`NotNull.product.name` → `NotNull.name` → `NotNull.java.lang.String` → `NotNull`

`MessageSource` tries them in this order, and the first match wins. This allows you to define general messages at the `NotNull` level and override them with field-specific messages when needed. To enable this with localized messages, you must wire the MessageSource into the validator explicitly:

```java
LocalValidatorFactoryBean validator = new LocalValidatorFactoryBean();
validator.setValidationMessageSource(messageSource());
// Without this, Hibernate Validator uses its own built-in messages
// which are NOT locale-resolved through Spring's MessageSource
```

### Thymeleaf and JSP Integration

In Thymeleaf, `#{key}` is the message expression — it calls `MessageSource.getMessage(key, locale)` automatically. `#{key(${arg})}` passes parameters. `${#locale}` gives you the current `Locale` object, and `${#locale.language}` gives the two-letter language code.

Locale switching links in Thymeleaf look like `th:href="@{${#request.requestURI}(lang='fr')}"` — this appends `?lang=fr` to the current URL, which `LocaleChangeInterceptor` picks up on the next request.

In JSP, `<spring:message code="key"/>` uses Spring's `MessageSource`. The `fmt:message` JSTL tag also works *if* you're using `JstlView` — Spring wires its `MessageSource` into JSTL's message resolution. Without `JstlView`, JSTL uses its own separate resource bundle mechanism and completely ignores Spring's `MessageSource`.

### The Complete Wiring

Putting it all together, the correct production configuration is:

```java
// 1. Resolver that supports setLocale()
@Bean
public LocaleResolver localeResolver() {
    SessionLocaleResolver r = new SessionLocaleResolver();
    r.setDefaultLocale(Locale.ENGLISH); // Fallback for new sessions
    return r;
}

// 2. MessageSource with correct encoding
@Bean
public MessageSource messageSource() {
    ReloadableResourceBundleMessageSource s =
        new ReloadableResourceBundleMessageSource();
    s.setBasenames("classpath:messages/messages",
                   "classpath:messages/validation");
    s.setDefaultEncoding("UTF-8");          // Critical!
    s.setFallbackToSystemLocale(false);     // Predictable fallback
    s.setCacheSeconds(3600);
    return s;
}

// 3. Interceptor wired to a compatible resolver
@Override
public void addInterceptors(InterceptorRegistry registry) {
    LocaleChangeInterceptor lci = new LocaleChangeInterceptor();
    lci.setParamName("lang");  // ?lang=fr triggers locale change
    registry.addInterceptor(lci);
}

// 4. Validator wired to MessageSource for localized validation errors
@Bean
public LocalValidatorFactoryBean validator() {
    LocalValidatorFactoryBean v = new LocalValidatorFactoryBean();
    v.setValidationMessageSource(messageSource());
    return v;
}
```

## Key Cross-Cutting Principles to Remember

Across all three topics, a few patterns recur that are worth internalizing. **First-match-wins ordering** governs both the converter chain and the resource resolver chain — position matters enormously. **The separation between current-request and next-request effects** appears in both locale changes (LocaleChangeInterceptor stores for next request) and resource chain caching. **Silent failures** are the most dangerous bugs: wrong charset encoding, misconfigured message bean names, and missing `setValidationMessageSource()` all produce wrong behavior without throwing exceptions. **`immutable` cache-control combined with content-hashed URLs** is the gold standard for static asset performance — safe indefinite caching with automatic invalidation when content changes.
