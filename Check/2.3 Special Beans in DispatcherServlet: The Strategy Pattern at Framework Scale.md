# Special Beans in DispatcherServlet: The Strategy Pattern at Framework Scale

Let's start with the right mental model, because everything else flows from it.

## The Core Idea: DispatcherServlet Knows Nothing

This sounds paradoxical — DispatcherServlet is the front controller of Spring MVC, so surely it *does* everything? In reality, it does almost nothing itself. It's more like an **orchestra conductor**: it knows the order in which things should happen, but the actual music is played by the special beans it discovers in the WebApplicationContext.

This is the **Strategy pattern** applied at framework scale. Every decision DispatcherServlet needs to make — "how do I find the right handler for this URL?", "how do I invoke that handler?", "what do I do if an exception is thrown?", "how do I turn a view name into rendered HTML?" — is delegated to a pluggable strategy object. You swap strategies by registering different beans. You don't touch DispatcherServlet itself at all.

There are nine special bean types. Let's build understanding from the most fundamental ones outward.

---

## HandlerMapping: Finding the Right Handler

When a request arrives, DispatcherServlet's first question is: *which piece of code should handle this?* That's the `HandlerMapping`'s job. It implements one critical method — `getHandler(request)` — which returns a `HandlerExecutionChain` (the handler plus any applicable interceptors), or `null` if it can't handle this request.

Multiple `HandlerMapping`s can be registered, and they're tried in order. The most important by far is `RequestMappingHandlerMapping`, which handles your `@RequestMapping`-annotated methods.

Here's something worth understanding deeply: `RequestMappingHandlerMapping` does its heavy lifting **at startup**, not per request. During initialization, it scans all `@Controller` beans, reflects over their methods, and builds a registry mapping `RequestMappingInfo` objects (which encode URL patterns, HTTP methods, content type conditions, etc.) to `HandlerMethod` objects. At request time, it just does a fast lookup in that registry. This is why adding `@RequestMapping` to a class that isn't in the component-scan path silently fails — the registry was already built.

---

## HandlerAdapter: Invoking the Handler

Here's a subtle design question: once you have the handler object, how do you call it? The answer isn't obvious, because `HandlerMapping` can return *different kinds* of handler objects. It might return a `HandlerMethod` (from `@RequestMapping`), an `HttpRequestHandler` (for static resources), or a legacy `Controller` interface implementation. Each requires a completely different invocation mechanism.

Rather than polluting DispatcherServlet with a giant `if-instanceof` chain, Spring uses the Adapter pattern. Each `HandlerAdapter` answers `supports(handler)` to say whether it knows how to invoke that kind of handler. DispatcherServlet asks each adapter in turn until one claims the handler, then delegates.

`RequestMappingHandlerAdapter` is by far the most sophisticated. Its `handle()` method orchestrates the entire argument resolution → method invocation → return value handling pipeline. Think about everything that has to happen for a single controller method call: path variables extracted from the URL, request body deserialized from JSON, query parameters bound and type-converted, the method invoked via reflection, and the return value either turned into a view name or serialized back to JSON. Each of those steps is handled by chains of `HandlerMethodArgumentResolver`s and `HandlerMethodReturnValueHandler`s inside the adapter.

If you ever create a custom handler type (returned by a custom `HandlerMapping`) and forget to register a corresponding `HandlerAdapter`, you'll get `ServletException: No adapter for handler`. This is because *no* adapter's `supports()` returns true — a clean fail-fast behavior.

---

## HandlerExceptionResolver: When Things Go Wrong

Every exception thrown during handler execution flows into the `HandlerExceptionResolver` chain. Each resolver's `resolveException()` method can return one of three things, and the semantics matter precisely:

Returning `null` means "I can't handle this — pass it to the next resolver." Returning an empty `ModelAndView` (created with `new ModelAndView()`) means "I've handled this — I already wrote the response, there's nothing more to render." Returning a `ModelAndView` with a view name means "I've handled this — render this view." If *no* resolver handles the exception, DispatcherServlet re-throws it and the Servlet container generates a generic HTTP 500.

The default chain has three members in order. `ExceptionHandlerExceptionResolver` handles your `@ExceptionHandler` methods — it looks in the current controller class first, then searches `@ControllerAdvice` beans. `ResponseStatusExceptionResolver` handles exceptions annotated with `@ResponseStatus`. `DefaultHandlerExceptionResolver` handles Spring's own internal exceptions (like `HttpRequestMethodNotSupportedException` → 405), but only sets the status code — it writes no response body, which surprises people who expect a JSON error payload.

A common trap: returning `new ModelAndView()` when you forgot to actually write anything to the response produces a silent HTTP 200 with an empty body. The "exception handled" signal and the "response written" action are entirely your responsibility.

---

## ViewResolver: From Name to Rendered Output

When a handler returns a view name like `"product/list"`, DispatcherServlet needs to turn that string into an actual rendered response. That's the `ViewResolver`'s job — it converts a view name (and a `Locale`) into a `View` object, which then renders itself.

Multiple resolvers are tried in order, and the first non-null result wins. This ordering has one critical implication: `InternalResourceViewResolver` — which resolves JSP paths — must **always be last**. Why? Because unlike every other resolver, it *never returns null*. It always creates an `InternalResourceView` for any name you give it, without checking whether the JSP file actually exists. If you put it first, it intercepts every view name before Thymeleaf, FreeMarker, or any other resolver gets a chance. The 404 for a missing JSP only surfaces later, when the view actually tries to forward to a non-existent file.

`ContentNegotiatingViewResolver` takes a different approach — instead of resolving view names itself, it delegates to *all* other resolvers and picks the result that best matches the client's `Accept` header. It belongs first in the chain, acting as a smart coordinator.

## The One Bean Found by Name: MultipartResolver

Here's the most important anomaly in the entire special bean system, and it's a classic exam trap. Every special bean is discovered by **type** — DispatcherServlet scans the context for all beans implementing `HandlerMapping`, all beans implementing `ViewResolver`, and so on. There is exactly one exception: `MultipartResolver`.

DispatcherServlet looks for it by the specific name `"multipartResolver"`. The relevant code inside `initMultipartResolver()` calls `context.getBean("multipartResolver", MultipartResolver.class)`. If your bean is named `"fileUploadResolver"` or `"myMultipartResolver"` or anything else, DispatcherServlet doesn't find it, sets its internal field to `null`, and file uploads silently fail with `MultipartException`.

Also worth noting: `MultipartResolver` has no default implementation. Unlike every other special bean type (which have fallback defaults registered in `DispatcherServlet.properties`), if you don't configure a `MultipartResolver`, file uploads simply don't work.

---

## LocaleResolver: More Subtle Than It Looks

The default `AcceptHeaderLocaleResolver` reads the `Accept-Language` HTTP header — simple enough. But there's a trap: its `setLocale()` method throws `UnsupportedOperationException`. You cannot change the Accept-Language header; it's sent by the browser.

This matters because `LocaleChangeInterceptor` — the standard way to let users switch languages with `?lang=fr` — calls `setLocale()` internally. If you configure `LocaleChangeInterceptor` with the default `AcceptHeaderLocaleResolver`, you get a runtime exception the moment someone tries to change their language. You need `SessionLocaleResolver` (stores locale in the session) or `CookieLocaleResolver` (stores locale in a cookie) for locale switching to work.

---

## FlashMapManager: Surviving the Redirect

The Post-Redirect-Get pattern solves a real problem (duplicate form submissions on browser refresh), but it creates a new one: how do you pass data — like a success message — across the redirect boundary without putting it in the URL?

`FlashMapManager` solves this. Before the redirect, it stores your flash attributes in the `HttpSession` tagged with the redirect target URL. When the redirect target is requested, `FlashMapManager.retrieveAndUpdate()` finds the matching flash map, copies its contents onto the request, and *immediately removes it from the session*. One-time use, keyed to the correct URL.

The key behavioral detail is "exactly one redirect." If you redirect A → B → C, flash attributes set on A are available on B, but they're consumed and removed there. C never sees them. If you need data at C, B's handler must explicitly re-save it.

---

## RequestToViewNameTranslator: The Silent Default

This one is easy to miss because it activates silently. When a controller method returns `void` or `null` — no explicit view name — DispatcherServlet needs to figure out what to render anyway. `DefaultRequestToViewNameTranslator` derives a view name from the request URI by stripping the leading slash and file extension.

So a `@GetMapping("/account/settings")` method returning `void` implicitly targets view name `"account/settings"`. With `InternalResourceViewResolver` using prefix `/WEB-INF/views/` and suffix `.jsp`, that becomes `/WEB-INF/views/account/settings.jsp`. Clean and convention-driven, but invisible — which is exactly where bugs hide if you don't know it exists.

---

## The Big Picture: What Ties It All Together

It's worth stepping back to see the flow of a single request through all these strategies in sequence. A multipart POST arrives → `MultipartResolver` wraps it. `HandlerMapping` finds the right controller method. `HandlerInterceptor`s run their pre-handle logic. `HandlerAdapter` resolves all arguments, invokes the method, handles the return value. If an exception occurs, the `HandlerExceptionResolver` chain takes over. If a view name is produced, `ViewResolver` turns it into a `View` and renders it. `LocaleResolver` informs the entire process of the user's locale. `FlashMapManager` handles anything crossing a redirect boundary.

DispatcherServlet itself is just the coordinator moving the request through this pipeline. Every meaningful decision is made by a swappable strategy bean. That's the design philosophy of the entire system — and understanding it is what lets you confidently customize Spring MVC at exactly the right extension point.
