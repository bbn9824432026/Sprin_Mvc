# Spring MVC Initialization: XML, Annotation, and Java Config

This topic is really about understanding *evolution* — how Spring MVC configuration changed over nearly two decades, why it changed, and what each style is actually doing under the hood. Once you understand the "why" behind each era, the mechanics become much easier to remember. Let's walk through all three styles and build a complete mental model.

---

## Why Three Styles Exist at All

Spring MVC was born in 2003, when XML was the dominant way to configure Java applications. As the Java ecosystem matured, annotation processing improved, and the Servlet specification itself evolved to support programmatic registration. Spring responded to each shift. The result is three configuration styles — XML-only, annotation-driven hybrid, and pure Java config — that represent three distinct eras. All three are still valid, all three still appear in production codebases, and all three are fair game on certifications. You need to be fluent in all of them, not just the newest one.

---

## Era 1 — Pure XML Configuration

In the earliest Spring MVC applications, you had three files doing all the work: `web.xml` declared your servlets and listeners, `root-context.xml` held your service layer beans, and `dispatcher-servlet.xml` held your web layer beans. Everything was explicit and verbose — no magic, but also no compile-time safety and no IDE refactoring support.

The single most important element in this style is `<mvc:annotation-driven/>`. This one line in `dispatcher-servlet.xml` is what makes `@RequestMapping`, `@ResponseBody`, `@Valid`, and `@ExceptionHandler` actually work. Without it, Spring doesn't know how to map annotated methods to URLs or how to serialize your objects to JSON. Specifically, it registers two critical infrastructure beans: `RequestMappingHandlerMapping`, which scans your `@Controller` classes and builds a registry of URL-to-method mappings, and `RequestMappingHandlerAdapter`, which is responsible for actually invoking those methods, resolving arguments like `@PathVariable` and `@RequestBody`, and handling return values. It also registers the default `HttpMessageConverter` instances — including the Jackson converter for JSON — and exception resolvers for `@ExceptionHandler` and `@ResponseStatus`.

One thing `<mvc:annotation-driven/>` deliberately does *not* register is `InternalResourceViewResolver`. View resolution is considered application-specific, so you always define that yourself as a separate `<bean>` declaration. This distinction trips people up — they expect `<mvc:annotation-driven/>` to set up everything, but the view resolver is your responsibility.

A complete pure XML setup looks like this:

```xml
<!-- dispatcher-servlet.xml — the web layer's configuration -->
<beans xmlns="http://www.springframework.org/schema/beans"
       xmlns:mvc="http://www.springframework.org/schema/mvc"
       xmlns:context="http://www.springframework.org/schema/context">

    <!-- Registers HandlerMapping, HandlerAdapter, MessageConverters, etc. -->
    <mvc:annotation-driven/>

    <!-- Scan ONLY @Controller classes into the child context -->
    <context:component-scan base-package="com.example"
                            use-default-filters="false">
        <context:include-filter type="annotation"
            expression="org.springframework.stereotype.Controller"/>
    </context:component-scan>

    <!-- Must be defined manually — not registered by annotation-driven -->
    <bean class="org.springframework.web.servlet.view.InternalResourceViewResolver">
        <property name="prefix" value="/WEB-INF/views/"/>
        <property name="suffix" value=".jsp"/>
    </bean>

    <!-- Static resources, interceptors, CORS all declared here -->
    <mvc:resources mapping="/static/**" location="/static/"/>

</beans>
```

## Era 2 — Annotation-Driven Hybrid

The second era kept `web.xml` and XML namespace configuration for infrastructure, but let Java annotations do the mapping work. You still wrote `<mvc:annotation-driven/>` and `<context:component-scan/>` in XML, but your controllers were now Java classes annotated with `@Controller` and `@RequestMapping` rather than XML bean definitions that mapped URLs manually. This hybrid style dominated from roughly 2008 to 2015 and is extremely common in legacy codebases.

The key insight here is that annotations and XML work *together*. The XML tells Spring where to look and what infrastructure to set up. The annotations on your Java classes tell Spring the specifics of how to handle each request. Neither alone is sufficient — the XML without the annotations gives you no mappings; the annotations without `<mvc:annotation-driven/>` in XML give you no infrastructure to process them.

## Era 3 — Pure Java Configuration

The third era arrived with Servlet 3.0 (2009) and Spring 3.1 (2011), which together made `web.xml` completely optional. Understanding *why* this became possible requires understanding one Servlet 3.0 feature: `ServletContainerInitializer`.

### The Zero-XML Bootstrap Mechanism

Before Servlet 3.0, `web.xml` was the only way to tell Tomcat about your servlets, listeners, and filters. Servlet 3.0 introduced `ServletContainerInitializer` — an interface that a library can implement and declare in a file at `META-INF/services/jakarta.servlet.ServletContainerInitializer` inside its JAR. When Tomcat starts, it uses Java's `ServiceLoader` mechanism to scan all JARs on the classpath, find these service files, instantiate the named classes, and call their `onStartup()` method. This all happens before any `web.xml` processing.

Spring's `spring-web.jar` contains exactly this file, pointing to `SpringServletContainerInitializer`. When Tomcat finds it and calls `onStartup()`, Spring's initializer scans the classpath for any class implementing `WebApplicationInitializer` and calls `onStartup()` on each one. Your configuration class — which extends `AbstractAnnotationConfigDispatcherServletInitializer` — is one of these. So the chain is: Tomcat finds Spring's service file → Spring finds your initializer → your initializer registers `ContextLoaderListener` and `DispatcherServlet` programmatically. The net result is exactly equivalent to `web.xml`, just expressed in Java.

### The Initializer Class Hierarchy

Spring provides a hierarchy of abstract classes, each one adding more convenience:

`WebApplicationInitializer` is the raw interface — you handle everything manually. `AbstractContextLoaderInitializer` handles root context setup for you. `AbstractDispatcherServletInitializer` adds `DispatcherServlet` registration. `AbstractAnnotationConfigDispatcherServletInitializer` is the top of the hierarchy and assumes you're using `@Configuration` classes — this is what you'll use in almost all cases.

```java
// This single class replaces web.xml entirely
public class WebAppInitializer
    extends AbstractAnnotationConfigDispatcherServletInitializer {

    // Which @Configuration classes build the ROOT context?
    @Override
    protected Class<?>[] getRootConfigClasses() {
        return new Class[]{RootConfig.class};
    }

    // Which @Configuration classes build the CHILD (web) context?
    @Override
    protected Class<?>[] getServletConfigClasses() {
        return new Class[]{WebConfig.class};
    }

    @Override
    protected String[] getServletMappings() {
        return new String[]{"/"};
    }

    // Add filters — these are mapped to DispatcherServlet automatically
    @Override
    protected Filter[] getServletFilters() {
        CharacterEncodingFilter cef = new CharacterEncodingFilter();
        cef.setEncoding("UTF-8");
        cef.setForceEncoding(true);
        return new Filter[]{cef};
    }

    // Customize the DispatcherServlet registration itself
    @Override
    protected void customizeRegistration(ServletRegistration.Dynamic reg) {
        // Makes 404s handleable by @ExceptionHandler instead of container
        reg.setInitParameter("throwExceptionIfNoHandlerFound", "true");
    }
}
```

One important behavior to internalize: `AbstractAnnotationConfigDispatcherServletInitializer` always calls `registration.setLoadOnStartup(1)` — there is no way to change this without overriding the method. The child context always initializes at startup. Also, returning `null` from `getRootConfigClasses()` is completely valid and simply means no `ContextLoaderListener` is registered — you get the single-context pattern discussed in the previous topics.

### `@EnableWebMvc` — The Java Equivalent of `<mvc:annotation-driven/>`

Just as `<mvc:annotation-driven/>` was the critical line in XML configuration, `@EnableWebMvc` is the critical annotation in Java configuration. They are directly equivalent in what they register. Mechanically, `@EnableWebMvc` is implemented as `@Import(DelegatingWebMvcConfiguration.class)`, which in turn extends `WebMvcConfigurationSupport`. That support class defines `@Bean` methods for `RequestMappingHandlerMapping`, `RequestMappingHandlerAdapter`, all the exception resolvers, all the default message converters, and the conversion service. In other words, `@EnableWebMvc` is just a shorthand for importing a pre-written `@Configuration` class full of infrastructure bean definitions.

The reason this design matters is the customization story: `DelegatingWebMvcConfiguration` auto-wires all `WebMvcConfigurer` beans from the context and delegates to their methods. This is why implementing `WebMvcConfigurer` in your `WebConfig` class allows you to customize MVC behavior — `DelegatingWebMvcConfiguration` collects your implementation and calls its methods during bean setup.

```java
@Configuration
@EnableWebMvc   // = @Import(DelegatingWebMvcConfiguration.class)
@ComponentScan(
    basePackages = "com.example",
    useDefaultFilters = false,
    includeFilters = @Filter(type = FilterType.ANNOTATION, classes = Controller.class)
)
public class WebConfig implements WebMvcConfigurer {

    // View resolution — equivalent to InternalResourceViewResolver @Bean
    @Override
    public void configureViewResolvers(ViewResolverRegistry registry) {
        registry.jsp("/WEB-INF/views/", ".jsp");
    }

    // Interceptors — equivalent to <mvc:interceptors>
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(new LoggingInterceptor())
                .addPathPatterns("/**")
                .excludePathPatterns("/static/**");
    }

    // Static resources — equivalent to <mvc:resources>
    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        registry.addResourceHandler("/static/**")
                .addResourceLocations("/static/");
    }

    // CORS — equivalent to <mvc:cors>
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
                .allowedOrigins("https://myapp.com")
                .allowedMethods("GET", "POST", "PUT", "DELETE");
    }
}
```

## The Most Critical Trap: `configure` vs `extend` for Message Converters

This is probably the single most important `WebMvcConfigurer` distinction to internalize, because getting it wrong causes silent failures that are hard to debug.

`WebMvcConfigurer` gives you two methods for customizing message converters. `configureMessageConverters()` is called *before* the default converters are added, and once you add anything to the list, Spring skips adding the defaults. This means if you call this method and add only a custom XML converter, you've removed Jackson entirely — every REST endpoint that returns JSON will start getting 406 Not Acceptable responses, with no exception or error log to tell you why.

`extendMessageConverters()` is called *after* the default converters are added. This is what you should almost always use — you're adding to or modifying an existing, working set of converters rather than replacing it.

```java
// DANGEROUS — replaces ALL defaults including Jackson
@Override
public void configureMessageConverters(List<HttpMessageConverter<?>> converters) {
    converters.add(new MappingJackson2XmlHttpMessageConverter());
    // After this: ONLY XML converter exists. JSON is dead.
}

// SAFE — defaults remain, you're just adding or tweaking
@Override
public void extendMessageConverters(List<HttpMessageConverter<?>> converters) {
    // Add XML support on top of existing converters (Jackson still here)
    converters.add(new MappingJackson2XmlHttpMessageConverter());
    
    // OR: find and customize the existing Jackson converter
    converters.stream()
        .filter(c -> c instanceof MappingJackson2HttpMessageConverter)
        .map(c -> (MappingJackson2HttpMessageConverter) c)
        .findFirst()
        .ifPresent(c -> {
            c.getObjectMapper().setSerializationInclusion(
                JsonInclude.Include.NON_NULL);
        });
}
```

The same `configure` vs `extend` logic applies to exception resolvers: `configureHandlerExceptionResolvers()` replaces defaults, `extendHandlerExceptionResolvers()` adds to them.

## A Subtle Trap: `@ControllerAdvice` in the Wrong Context

When you write component scan filters for the child context, the natural instinct is to include only classes annotated with `@Controller`. But `@ControllerAdvice` — the annotation for global exception handlers — is *not* annotated with `@Controller`. It's annotated with `@Component`. This means if you write `includeFilters = @Filter(classes = Controller.class)`, your `@ControllerAdvice` classes end up in the root context instead of the child context.

The problem is that `ExceptionHandlerExceptionResolver` (registered by `@EnableWebMvc`) only looks for `@ExceptionHandler` methods in the *child* context. If your `@ControllerAdvice` is in the root context, the exception resolver never finds it, and global exception handling silently stops working. The fix is to include `ControllerAdvice.class` in your filter alongside `Controller.class` when configuring the child context's component scan.

## Spring Boot: How It Replaces All of This

Spring Boot doesn't pick one of the three styles — it replaces the entire startup mechanism. `SpringApplication.run()` creates a single `AnnotationConfigServletWebServerApplicationContext`, runs auto-configuration (which includes `WebMvcAutoConfiguration` — the Boot equivalent of `@EnableWebMvc` — and `DispatcherServletAutoConfiguration` which registers `DispatcherServlet` as a bean), starts an embedded Tomcat, and registers the `DispatcherServlet` with it programmatically. None of the three initialization styles above applies. `ContextLoaderListener` is never used. `WebApplicationInitializer` is never called. The `DispatcherServlet` is a Spring bean, not a Tomcat-managed servlet.

The critical Boot-specific trap is that adding `@EnableWebMvc` to any `@Configuration` class in a Boot application disables `WebMvcAutoConfiguration` entirely. Jackson stops working, static resources stop serving, and view resolvers disappear. In Boot, you customize MVC by implementing `WebMvcConfigurer` *without* `@EnableWebMvc`.

## Bringing It Together: The Mapping Between Styles

Every XML element has a direct Java config equivalent, and understanding that mapping makes both styles easier to reason about:

`<mvc:annotation-driven/>` maps to `@EnableWebMvc`. `<context:component-scan/>` maps to `@ComponentScan`. `<mvc:interceptors>` maps to `addInterceptors()`. `<mvc:resources/>` maps to `addResourceHandlers()`. `<mvc:cors/>` maps to `addCorsMappings()`. `<mvc:default-servlet-handler/>` maps to `configureDefaultServletHandling()`. `<tx:annotation-driven/>` maps to `@EnableTransactionManagement`. And `<mvc:view-controller/>` maps to `addViewControllers()` — which is worth noting as a special case: view controllers let you map a URL directly to a view name without writing a controller class, but they cannot add model attributes or call services because there is no controller logic involved.

Once you've built this mental map, you can read a configuration in any style and mentally translate it to the others — which is exactly the skill certification questions are testing.
