# HandlerMapping: The Internal Resolution Chain

Let's approach this topic by asking the right question first, because it reframes everything that follows. When an HTTP request arrives at your application, *something* has to bridge the gap between a raw URL like `/api/products/42` and a specific Java method like `ProductController.getById()`. That bridge is `HandlerMapping`, and understanding how it works internally — not just that it exists — is what this topic is really about.

## The Fundamental Design: A Template Method Wrapping a Strategy

Before diving into any specifics, notice the architectural choice Spring made. `AbstractHandlerMapping` implements `getHandler()` as a **template method** — a fixed sequence of steps that every `HandlerMapping` follows — while delegating the actual "find the handler for this URL" work to a `getHandlerInternal()` method that subclasses override. This means the logic for assembling interceptors, handling CORS, resolving bean names, and wrapping everything in a `HandlerExecutionChain` is written *once* in the abstract class and inherited by everyone.

The practical implication is that when you write a custom `HandlerMapping`, you only need to implement the interesting part: "given this request, which handler applies?" Everything else is handled for you.

## Building the Registry: Work Done at Startup, Not Per Request

The most important thing to understand about `RequestMappingHandlerMapping` — the implementation that powers all your `@GetMapping` and `@PostMapping` methods — is that its heaviest work happens **at startup, not per request**.

When Spring instantiates this bean during Phase 11 of `context.refresh()`, it calls `afterPropertiesSet()`, which triggers a full scan of every `@Controller` bean in the application context. For each controller, it reflects over every method, looking for `@RequestMapping` annotations (including the shortcuts like `@GetMapping`). For each annotated method, it creates a `RequestMappingInfo` object — a rich data structure encoding not just the URL pattern, but also the HTTP method constraint, content type conditions, parameter conditions, and anything else specified in the annotation.

There's a subtlety worth pausing on: when a controller has a class-level `@RequestMapping` and a method-level one, Spring **combines** them. The URL paths are concatenated, and the conditions are intersected. So a class annotated with `@RequestMapping(value="/api", consumes="application/json")` and a method annotated with `@GetMapping("/products")` produces a combined mapping for `GET /api/products` requiring `Content-Type: application/json`. Method-level conditions override class-level conditions when they conflict.

All of these `RequestMappingInfo → HandlerMethod` pairs are stored in a `MappingRegistry` protected by a `ReentrantReadWriteLock`. Multiple threads can read the registry simultaneously during request handling, while the write lock is only needed during registration — which essentially only happens at startup. This is what makes the per-request lookup fast.

## The Per-Request Algorithm: How a URL Becomes a Handler

When a request arrives, `getHandlerInternal()` runs this sequence, and understanding each step reveals why Spring behaves the way it does in edge cases.

First, it extracts the "lookup path" — the URL with the context path and servlet path stripped away, leaving just the part your mappings care about. Then it tries a fast direct lookup for exact URL matches (no wildcards). Only if that fails does it fall back to iterating all registered patterns and testing each one — which is slower but still typically very fast in practice because of internal indexing.

Once it has a list of candidate matches, it **sorts them by specificity** and picks the winner. This sorting is where the rules you've probably encountered in practice come from: a literal path like `/products/featured` beats a URI template like `/products/{id}`, which beats a single wildcard `/products/*`, which beats a double wildcard `/products/**`. The intuition is that more concrete mappings should win over more general ones.

After selecting the best match, Spring checks whether the second-best match is *equally specific*. If it is, that's ambiguous — two different handlers with the same priority for the same request — and Spring throws `IllegalStateException`. Importantly, some of these ambiguity checks happen at **startup** (for exact duplicates) and some at **request time** (for complex condition ties). Knowing this distinction matters: a startup ambiguity crashes your application immediately, while a request-time ambiguity only crashes when a specific request triggers it.

## URI Templates and Regex: More Power Than You Might Think

You're probably comfortable with `{id}` capturing a path segment, but there are a few subtleties worth understanding at depth.

First, `{id}` and `{productId}` are **identical patterns** as far as Spring is concerned. The variable name is just a label for extraction; the pattern shape is the same. Registering both will cause an ambiguous mapping error. Only a regex constraint makes them genuinely different: `{id:\\d+}` (digits only) and `{slug:[a-z-]+}` (lowercase letters and hyphens) are distinct patterns because they match different input.

Second, when a regex constraint doesn't match — say you have `{id:\\d+}` and someone requests `/products/abc` — Spring doesn't throw a type conversion error. It simply doesn't consider this mapping a match at all and falls through to the next option. If nothing else matches, the result is 404, not 400. The regex is a pattern-matching filter, not a validation rule.

Third, without the `:.+` syntax, dots truncate path variables by default. A request for `/files/report.pdf` would give you `filename="report"`, not `filename="report.pdf"`. This surprises many developers who assume path variables capture the whole segment including dots.

## What Happens When No Match Is Found

This is where many developers have an incomplete mental model. The assumption "no mapping = 404" is only partially true.

When `lookupHandlerMethod()` finds no candidates, it calls `handleNoMatch()`, which does something more sophisticated: it checks whether the URL *partially* matched any registered mappings. If your URL matches a registered pattern but the HTTP method is wrong, you get **405** (Method Not Allowed) along with an `Allow` header listing what's accepted. Wrong `Content-Type`? **415**. Wrong `Accept` header? **406**. Wrong query parameters? **400**. Only if the URL itself doesn't match anything at all do you get **404**.

This matters for debugging: if you're seeing a 405 instead of a 404, it means Spring *found* your URL — you just used the wrong HTTP method. That's often a much faster path to diagnosing the problem.

## Assembling the HandlerExecutionChain: Interceptors Aren't Magic

Once `getHandlerInternal()` returns a handler, `getHandlerExecutionChain()` wraps it with applicable interceptors. This is where interceptors registered via `WebMvcConfigurer.addInterceptors()` actually get attached to a request.

Each interceptor is either a plain `HandlerInterceptor` (which applies to every request) or a `MappedInterceptor` (which carries URL include/exclude patterns and is only added to the chain if the request URL matches). The chain is assembled in registration order, which determines the `preHandle()` execution order. The `postHandle()` and `afterCompletion()` methods execute in **reverse** order — the last interceptor to run `preHandle()` is the first to run `postHandle()`.

There's a crucial nuance in `afterCompletion()`: it only runs for interceptors whose `preHandle()` returned `true`. The `interceptorIndex` field tracks the furthest point the pre-handle phase reached. If interceptor 2 returns `false`, interceptors 3, 4, 5 never had `preHandle()` called, so `afterCompletion()` is never called on them either. This is actually good design — it means cleanup logic in `afterCompletion()` can assume that the corresponding `preHandle()` ran.

One more thing about `afterCompletion()`: exceptions thrown inside it are **swallowed**. They're logged, but they never propagate. This means you cannot use `afterCompletion()` for logic that needs to signal errors — if it throws, the error disappears silently from the caller's perspective.

## The Multiple HandlerMapping Chain: First Match Wins

There's one more layer to the picture. DispatcherServlet doesn't just use one `HandlerMapping` — it holds a list of them ordered by their `@Order` value. It iterates the list and returns the result from the **first one that gives a non-null answer**. This is why `RequestMappingHandlerMapping` at order 0 handles your `@RequestMapping` methods before `BeanNameUrlHandlerMapping` at order 1 ever gets a chance to look.

The practical implication: if you register a custom `HandlerMapping` at order -1 (before `RequestMappingHandlerMapping`), and it returns a handler for `/api/**`, then `RequestMappingHandlerMapping` is never consulted for those URLs, even if you have matching `@GetMapping` methods. This can cause subtle bugs where a registered `@Controller` method appears to be ignored.

## A Mental Model to Tie It Together

Think of the whole system as a two-phase operation. Phase one happens at startup: Spring reads your code, builds a sorted registry of pattern-to-method mappings, and locks it in. Phase two happens per request: Spring does a fast lookup in that registry, applies specificity rules to pick the winner, wraps it with applicable interceptors, and hands the result back to DispatcherServlet. The speed of phase two depends on the quality of phase one — and that's exactly why Spring invests so much work at startup.

Understanding this two-phase design also explains several behaviors that seem mysterious at first: why adding a new `@GetMapping` to a running application has no effect (the registry is already built), why duplicate patterns fail at startup rather than silently overriding each other, and why complex `@Conditional`-driven configurations need to resolve completely before the registry forms.
