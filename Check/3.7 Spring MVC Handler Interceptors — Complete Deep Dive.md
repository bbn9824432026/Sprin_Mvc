# Spring MVC Handler Interceptors — Complete Deep Dive

Let's build a thorough mental model of this topic from the ground up. Think of interceptors as **toll booths on a highway** — every request travelling to a destination (your controller) passes through them, and you decide at each booth whether to let it through, inspect it, or turn it back.

---

## Where Do Interceptors Live in the Stack?

Before diving into the mechanics, you need to understand *where* interceptors fit — because this determines everything about what they can and cannot do.

The HTTP request lifecycle has three distinct interception points, each at a different architectural layer:

**Servlet Filters** sit at the outermost layer — the servlet container level, before Spring even sees the request. They have access to the raw `HttpServletRequest` and `HttpServletResponse`, but they don't have direct access to the Spring application context (they can get it indirectly via `DelegatingFilterProxy`). They fire for every URL pattern they're registered against, regardless of whether Spring MVC handles it.

**Handler Interceptors** sit inside the `DispatcherServlet`, after Spring has already received the request and identified which handler (controller method) should process it. Because they're Spring beans, they have full access to the application context via `@Autowired`. They only fire for requests that matched a handler mapping — not arbitrary URLs.

**AOP Advice** operates deepest — at the bean method level. It works through Spring's proxy mechanism and can intercept individual method calls on your controllers, with access to method arguments and return values.

The crucial insight is: **each level can see more about the request than the previous one, but each also runs later in the process.** Filters know nothing about controllers; interceptors know which handler method will run but not its arguments yet; AOP can see everything but operates purely at the Java method level, not the HTTP level.

## The Three-Method Contract — The Heart of Everything

`HandlerInterceptor` defines three methods, and understanding precisely *when* each runs and *what guarantees* each has is what the entire topic is built on.

### `preHandle()` — The Gatekeeper

```java
boolean preHandle(HttpServletRequest request,
                  HttpServletResponse response,
                  Object handler) throws Exception
```

This runs **before the handler method is invoked**. The return value is the key: return `true` to allow the request to continue down the chain; return `false` to stop it dead.

What can you do here? You can read request headers, set request attributes (to pass data to the handler and other interceptors), write directly to the response, and short-circuit the entire request. What you *cannot* do is access the handler's method arguments (they haven't been resolved yet) or the return value (the handler hasn't run yet).

The `Object handler` parameter is also important. Most of the time it will be a `HandlerMethod` instance — which gives you the controller class, the actual `java.lang.reflect.Method`, and any annotations on the method or controller. But it can also be a `ResourceHttpRequestHandler` for static files. **Always check `handler instanceof HandlerMethod` before casting** — forgetting this causes `ClassCastException` when serving CSS or JavaScript.

### `postHandle()` — The View Decorator (with a Major Caveat)

```java
void postHandle(HttpServletRequest request,
                HttpServletResponse response,
                Object handler,
                @Nullable ModelAndView modelAndView) throws Exception
```

This runs **after the handler executes, but before view rendering**. For classic MVC with Thymeleaf or JSP templates, this is useful because you can add attributes to the `ModelAndView` and they'll appear in the rendered page.

Here is the single most important trap in this entire topic: **for `@ResponseBody` and `@RestController` endpoints, the response body has already been written to the output stream by the time `postHandle()` is called.** The `modelAndView` parameter will be `null`. You cannot modify the JSON or XML body. You may not even be able to add response headers because the response might already be committed. This trips up many developers who write `postHandle()` logic assuming they can modify REST responses.

`postHandle()` is also skipped entirely if the handler threw an exception, if a prior interceptor's `preHandle()` returned false, or if async processing was started.

### `afterCompletion()` — The Guaranteed Cleanup Hook

```java
void afterCompletion(HttpServletRequest request,
                     HttpServletResponse response,
                     Object handler,
                     @Nullable Exception ex) throws Exception
```

Think of this as the `finally` block of the interceptor world. It runs **after everything** — after view rendering, after exception handling. If *this interceptor's own* `preHandle()` returned `true`, then `afterCompletion()` is guaranteed to run, no matter what happens afterward.

The `ex` parameter is particularly useful: it's `null` on success and carries the exception if something went wrong. This makes it ideal for performance logging (you know the total time including rendering), resource cleanup (clearing MDC, releasing thread-locals), and audit logging.

One critical subtlety: **exceptions thrown inside `afterCompletion()` are swallowed** — they get logged but don't propagate, because the response is already committed at this point. Always wrap cleanup code in try-catch inside `afterCompletion()`.

## The Execution Order Rules — The Most Exam-Tested Part

Imagine three interceptors registered in order A, B, C. The execution pattern is:

For a normal request, `preHandle()` runs in **forward order** (A → B → C), then the handler executes, then `postHandle()` runs in **reverse order** (C → B → A), then view rendering, then `afterCompletion()` runs in **reverse order** (C → B → A).

The reverse order for `postHandle` and `afterCompletion` is intentional and elegant — it mirrors how you'd nest try-finally blocks. The outermost interceptor (A) is the last to clean up, giving it the widest view of what happened.

### What Happens When `preHandle()` Returns False?

This is where Spring uses a subtle internal tracking variable called `interceptorIndex`. Every time a `preHandle()` returns `true`, the index is updated to that interceptor's position. When a `preHandle()` returns `false`, Spring stops the chain immediately and calls `triggerAfterCompletion()` starting from `interceptorIndex` down to 0.

So if B's `preHandle()` returns false: A already succeeded (index=0), B just failed (not counted), C never ran. Spring calls `afterCompletion()` only for A — because that's the range [0..interceptorIndex]. B itself does *not* get `afterCompletion()` called because it returned false. C never got a chance to run `preHandle()`, so it gets nothing.

A common mistake is thinking B would get `afterCompletion()`. It does not — it returned false, so Spring considers it as having opted out.

### What Happens When the Handler Throws an Exception?

All three `preHandle()` methods already ran and returned `true`. The handler throws. Spring stores this as `dispatchException` and skips `postHandle()` for all interceptors. Then it calls `afterCompletion()` in reverse order (C, B, A) with the exception object as the `ex` parameter. **`afterCompletion()` is not skipped just because there was an exception** — it's designed to always run for cleanup.

### What if an Exception Happens in `postHandle()`?

Say C and B's `postHandle()` ran fine, but B's throws an exception. Spring stores it as `dispatchException` and skips A's `postHandle()`. Then `afterCompletion()` runs for all three (C, B, A) with that exception. The key insight: a `postHandle()` exception does not stop `afterCompletion()` from running.

## Async Requests — `preHandle()` Called Twice

When a handler returns a `Callable<T>` or `DeferredResult<T>`, the request lifecycle splits across two separate dispatches. This is where `AsyncHandlerInterceptor` comes in, which adds a fourth method: `afterConcurrentHandlingStarted()`.

On the **first dispatch** (the initial Servlet thread): `preHandle()` runs, the handler starts the async operation, then `afterConcurrentHandlingStarted()` is called instead of `postHandle/afterCompletion`, and the Servlet thread is released.

On the **second dispatch** (when the async result is ready, on a new Servlet thread): `preHandle()` is called **again**, the result is processed, then `postHandle()` and `afterCompletion()` run normally.

The important trap here is that `preHandle()` firing twice can double-count metrics, double-check auth tokens, or create duplicate side effects. The fix is to check `request.getDispatcherType() == DispatcherType.ASYNC` inside `preHandle()` and skip logic that shouldn't run twice.

## Registering Interceptors and Path Patterns

You register interceptors via `WebMvcConfigurer.addInterceptors()`:

```java
@Override
public void addInterceptors(InterceptorRegistry registry) {
    registry.addInterceptor(new AuthInterceptor())
            .addPathPatterns("/api/**", "/admin/**")
            .excludePathPatterns("/api/public/**", "/api/auth/**")
            .order(3); // Lower number = runs earlier in preHandle
}
```

A critical detail: **if you don't specify any path patterns, the interceptor applies to everything** — including static resource requests handled by `ResourceHttpRequestHandler`. This means your authentication interceptor would run for every `.css`, `.js`, and `.png` file. Always specify patterns or exclusions.

Pattern semantics: `/**` matches all paths including sub-paths; `/api/**` matches `/api/` and anything beneath it; `/api/*` matches only one level deep (not `/api/products/42`). **Exclude patterns always win over include patterns** for the same URL.

## Passing Data Through the Request

A clean pattern that interceptors enable is storing data in request attributes for downstream use. In `preHandle()`, you compute something (trace ID, authenticated user, start timestamp) and store it via `request.setAttribute("key", value)`. The handler method can then access it either via `request.getAttribute("key")` directly, or more elegantly via `@RequestAttribute("key")` on a method parameter — Spring's `RequestAttributeMethodArgumentResolver` handles the lookup automatically. Then `afterCompletion()` can read the start timestamp to compute total elapsed time.

The MDC pattern for structured logging is a particularly important use of this: set `MDC.put("traceId", ...)` in `preHandle()` and **always** clear it in `afterCompletion()` with `MDC.remove(...)`. Because HTTP servers use thread pools and threads are reused, failing to clean MDC means the next request on that thread will carry the previous request's trace ID in its logs — a subtle and hard-to-debug corruption.

## Built-In Interceptors Worth Knowing

Spring ships several ready-made interceptors. `LocaleChangeInterceptor` watches for a query parameter (e.g., `?lang=fr`) and updates the request's locale via whatever `LocaleResolver` is configured — note it throws `UnsupportedOperationException` with `AcceptHeaderLocaleResolver` since that resolver is read-only. `WebContentInterceptor` sets `Cache-Control` headers on responses, letting you define caching policies per URL pattern. `UserRoleAuthorizationInterceptor` checks `request.isUserInRole()` for simple role-based access control.

---

## The Deprecated `HandlerInterceptorAdapter`

Before Java 8, interfaces couldn't have default methods, so Spring provided `HandlerInterceptorAdapter` as an abstract class with no-op implementations of all three methods — you'd extend it and only override what you needed. Since Java 8, the `HandlerInterceptor` interface itself has default no-op implementations for all three methods, making the adapter completely redundant. Spring 5.3 deprecated it. Modern code implements `HandlerInterceptor` directly. You'll still see it in legacy codebases and some exam questions.

## A Mental Model to Tie It All Together

Think of the interceptor chain like a stack of transparent sleeves that a document (the request) passes through on the way in, and back out through on the way back. The outermost sleeve (A) is the first to touch the request going in, and the last to touch it coming out. If B refuses to let the document through, A still gets to do its cleanup on the way back out, but B and C do not — B made the decision, and C never even saw the document.

`postHandle()` is like a note you can stick on the document before it's photocopied (rendered to the view) — but if the document is a digital file that gets emailed directly (`@ResponseBody`), the email was already sent before you had a chance to stick your note on it. `afterCompletion()` is the shredder receipt — it tells you the document was processed (or what went wrong), and it triggers regardless of whether the document reached its destination or got blocked along the way.

## Quick Mental Checklist for Exam Questions

When you see a question about interceptors, ask yourself: Which interceptors successfully completed `preHandle()`? That determines the `interceptorIndex`, which determines who gets `afterCompletion()`. Did the handler throw, or did a `postHandle()` throw? Both skip remaining `postHandle()` calls but both still trigger `afterCompletion()` for all successful `preHandle()` interceptors. Is this a `@ResponseBody` endpoint? Then `modelAndView` is null in `postHandle()` and the response body can't be modified there. Is it async? Then `preHandle()` fires twice — be careful about side effects.

This topic rewards careful, methodical thinking about the sequence of events more than memorizing facts. Once you can trace through any given scenario step-by-step, you'll get every question right.
