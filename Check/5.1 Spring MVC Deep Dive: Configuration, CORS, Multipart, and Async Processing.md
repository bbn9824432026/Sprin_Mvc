# Spring MVC Deep Dive: Configuration, CORS, Multipart, and Async Processing

## Topic 5.1 — Spring MVC Configuration

### Why Configuration Matters at This Level

Understanding Spring MVC configuration deeply means understanding *what beans get created, in what order, by what mechanism, and what each one does*. Most developers treat configuration as a black box — they know `@EnableWebMvc` "enables" Spring MVC, but they don't know what that actually means at the bean level. That gap is exactly where interview questions and production bugs live.

### Three Eras, One Runtime

The most important insight to internalize first is that XML configuration, Java configuration, and Spring Boot auto-configuration all produce **essentially the same set of runtime beans**. They're three different *declaration styles*, not three different frameworks. The `<mvc:annotation-driven/>` XML tag, `@EnableWebMvc`, and Spring Boot's `WebMvcAutoConfiguration` all ultimately result in the same `RequestMappingHandlerMapping`, `RequestMappingHandlerAdapter`, and exception resolvers being present in the application context. This equivalence is routinely tested in exams.

### @EnableWebMvc: What It Actually Does

When you add `@EnableWebMvc` to a `@Configuration` class, Java itself processes that annotation at startup. The annotation has `@Import(DelegatingWebMvcConfiguration.class)` on it, which means Spring imports `DelegatingWebMvcConfiguration` as a configuration class. That class extends `WebMvcConfigurationSupport`, which is the *actual bean factory* — the class containing `@Bean` methods for every piece of Spring MVC infrastructure.

Think of the relationship like this: `@EnableWebMvc` is just a doorbell. Pressing the doorbell (adding the annotation) imports `DelegatingWebMvcConfiguration`, which is a smart delegator that collects all `WebMvcConfigurer` beans from your application context and calls their methods. The actual construction work happens in `WebMvcConfigurationSupport`, which `DelegatingWebMvcConfiguration` inherits from.

`WebMvcConfigurationSupport` registers the following beans explicitly through `@Bean` methods: `RequestMappingHandlerMapping`, `RequestMappingHandlerAdapter`, `HandlerExceptionResolverComposite` (which wraps the three standard resolvers), `FormattingConversionService`, `Validator`, `ContentNegotiationManager`, and the resource handler mapping. What it does *not* register, and this is a critical exam trap, is `InternalResourceViewResolver`. If you need JSP support, you must add that bean explicitly.

### DelegatingWebMvcConfiguration: The Composite Pattern

`DelegatingWebMvcConfiguration` implements the composite pattern. It has a field of type `WebMvcConfigurerComposite`, and during startup it uses `@Autowired` to inject *all* beans in the application context that implement `WebMvcConfigurer`. This is important: if you have three `@Configuration` classes all implementing `WebMvcConfigurer`, all three get collected into this composite, and when Spring calls `addInterceptors()` on the composite, it calls `addInterceptors()` on *all three* in sequence. This is how modular configuration works in practice — you can split your CORS configuration, your security interceptors, and your resource handlers into separate configuration classes, and they all get applied.

The ordering between multiple `WebMvcConfigurer` implementations is controlled by `@Order` annotations. Without `@Order`, the order is undefined between configurers, though it's deterministic within a single application run.

### The configureMessageConverters vs extendMessageConverters Distinction

This is worth understanding precisely because it's a common source of subtle bugs. In `WebMvcConfigurationSupport.getMessageConverters()`, the flow is: first call `configureMessageConverters(list)` which gives implementations a chance to replace the defaults; if the list is still empty after that call, add all the default converters (ByteArray, String, Resource, Jackson if present, etc.); then always call `extendMessageConverters(list)` regardless.

The consequence: if you add *anything* to `configureMessageConverters()`, defaults are skipped. If you add a YAML converter there, you've lost Jackson. `extendMessageConverters()` is the safe choice for almost every scenario because it always adds to whatever is already there. In Spring Boot specifically, `configureMessageConverters()` is even more dangerous because the defaults include auto-configured Jackson with all the Boot-specific settings.

### Spring Boot Auto-Configuration: The Conditional Chain

Spring Boot's approach is elegant: `WebMvcAutoConfiguration` is annotated with `@ConditionalOnMissingBean(WebMvcConfigurationSupport.class)`. This single condition is the entire mechanism by which Boot gracefully coexists with or yields to manual configuration.

When you don't add `@EnableWebMvc`, no `WebMvcConfigurationSupport` subclass exists in the context, so the condition passes and `WebMvcAutoConfiguration` activates. `WebMvcAutoConfiguration` itself extends `DelegatingWebMvcConfiguration` (which extends `WebMvcConfigurationSupport`), so it's doing the same infrastructure work as `@EnableWebMvc` but with opinionated defaults — auto-configured Jackson, static resource locations from `classpath:/static/`, content negotiation from properties, and so on.

Boot also adds things that `@EnableWebMvc` doesn't: a `WelcomePageHandlerMapping` that serves `index.html`, auto-registration of `ResourceUrlEncodingFilter` for versioned static URLs, and reading of `spring.mvc.*` properties through `WebMvcProperties`.

`DispatcherServletAutoConfiguration` runs separately and is responsible for creating the `DispatcherServlet` bean itself and registering it with the embedded Tomcat via `DispatcherServletRegistrationBean`. This is something you'd have to do manually in `web.xml` in a traditional deployment.

### The @EnableWebMvc in Spring Boot Problem

This deserves its own mental model. When you add `@EnableWebMvc` to a Spring Boot application: it imports `DelegatingWebMvcConfiguration`, which is a `WebMvcConfigurationSupport` subclass, which now exists in the context, which causes `WebMvcAutoConfiguration`'s `@ConditionalOnMissingBean` check to fail, which causes `WebMvcAutoConfiguration` to back off entirely.

The result: Jackson auto-configuration is gone, static resource handlers are gone, view resolver from properties is gone, welcome page is gone, locale resolver is gone. Your application doesn't crash — it just stops working correctly in subtle ways. REST endpoints return 406 because Jackson isn't registered. CSS files return 404 because the static handler isn't registered. JSP rendering breaks because the view resolver from properties isn't active.

The fix is simple: never add `@EnableWebMvc` in a Spring Boot application. Just implement `WebMvcConfigurer` directly on your `@Configuration` class, which extends Boot's auto-configuration rather than replacing it.

### Customisation Layers in Spring Boot

Think of it as a stack. At the bottom, `WebMvcAutoConfiguration` provides sensible defaults. On top of that, your `WebMvcConfigurer` implementations customize those defaults without replacing them. On top of that, you can declare specific `@Bean` overrides to replace individual infrastructure beans. And at the top, if you truly need complete control, `@EnableWebMvc` replaces the whole foundation — but now you're responsible for everything.

For 99% of applications, the right answer is staying at Layer 2: implement `WebMvcConfigurer`, don't add `@EnableWebMvc`.

### @WebMvcTest: Slice Testing

`@WebMvcTest` is Spring Boot's way of testing only the web layer in isolation. It loads `@Controller`, `@ControllerAdvice`, `@JsonComponent`, `WebMvcConfigurer` implementations, and converters — but explicitly excludes `@Service`, `@Repository`, and generic `@Component` beans. This means every dependency your controller has must be provided via `@MockBean`. Forgetting a `@MockBean` causes `UnsatisfiedDependencyException`, which is a common test failure that confuses developers who are used to full-context integration tests.

## Topic 5.2 — CORS Configuration

### The Mental Model You Must Have

The single most important thing to understand about CORS is that it is a *browser* mechanism, not a server security mechanism. The server receives *every* request regardless of its origin. CORS headers in the server response are instructions to the *browser* about whether the JavaScript that made the request is allowed to read the response. A curl command, Postman, or server-to-server HTTP call ignores CORS entirely.

This has a critical implication: CORS is not a substitute for authentication and authorization. A browser that respects CORS headers will block unauthorized cross-origin reads — but a malicious actor bypasses the browser entirely. Real security lives in your authorization layer.

### Same-Origin and Why the Browser Cares

The browser's Same-Origin Policy exists to protect users. If the SOP didn't exist, a malicious website at `evil.com` could make JavaScript calls to `yourbank.com/api/transfer`, and since your browser holds your banking session cookie, those requests would be authenticated. Your bank's response would go back to `evil.com`'s JavaScript, which could read your balance, initiate transfers, and so on.

CORS is the mechanism by which `yourbank.com` explicitly says: "I trust `myapp.example.com` to read my responses." The server's `Access-Control-Allow-Origin` header is that declaration of trust.

Origin is defined as scheme + host + port. All three must match for same-origin. `http` vs `https` is different. Port 3000 vs port 443 is different. `api.example.com` vs `app.example.com` is different.

### Simple vs Pre-flight Requests

Not every cross-origin request requires a pre-flight. The browser distinguishes between "simple" requests and requests that require negotiation.

A request is simple if it uses GET, HEAD, or POST; has no custom headers beyond a safe list; and if POST, uses `application/x-www-form-urlencoded`, `multipart/form-data`, or `text/plain` as the Content-Type. Simple requests go directly to the server with an `Origin` header, and the server's response either includes or omits `Access-Control-Allow-Origin`. The browser then either allows or blocks the JavaScript from reading the response.

Everything else triggers a pre-flight. Sending `Content-Type: application/json` triggers pre-flight. Using `DELETE` or `PUT` triggers pre-flight. Adding an `Authorization` header triggers pre-flight. In modern REST APIs, nearly every meaningful request triggers pre-flight because they use `application/json` and `Authorization`.

The pre-flight is an `OPTIONS` request sent automatically by the browser before the actual request. It carries `Access-Control-Request-Method` (telling the server what method the actual request will use) and `Access-Control-Request-Headers` (telling the server what custom headers will be sent). The server responds with `Access-Control-Allow-*` headers declaring what it permits. If the pre-flight succeeds (2xx response with matching headers), the browser sends the actual request. The pre-flight result is then cached for the duration specified in `Access-Control-Max-Age`, so the browser doesn't repeat it on subsequent requests.

### How Spring Processes CORS Internally

Spring integrates CORS handling into `AbstractHandlerMapping.getHandler()`. After finding the appropriate handler for a request, if the request has an `Origin` header, Spring fetches the `CorsConfiguration` for that handler (combining any `@CrossOrigin` annotation with any global configuration), then wraps the handler execution chain in CORS-aware components.

For pre-flight requests (OPTIONS with `Access-Control-Request-Method`), Spring *replaces* the actual handler with a `PreFlightHandler` that just processes the CORS negotiation and returns 200 or 204 with no body. For actual CORS requests, Spring adds a `CorsInterceptor` to the front of the interceptor chain; this interceptor validates the `Origin` against allowed origins and sets the `Access-Control-Allow-Origin` response header. If the origin isn't allowed, `DefaultCorsProcessor` calls `response.sendError(403)`.

One automatic behavior worth knowing: Spring's `DefaultCorsProcessor` always adds `Vary: Origin` to CORS responses. This is necessary to prevent proxy caches from serving one user's CORS response (which says `Access-Control-Allow-Origin: app.example.com`) to a request from a different origin. The `Vary` header tells caches that the origin is part of the cache key.

### The allowedOrigins("*") + Credentials Trap

This is the most common CORS configuration mistake, and Spring enforces the CORS specification by throwing `IllegalArgumentException` at startup if you try it. The CORS specification prohibits browsers from sending credentials (cookies, `Authorization` headers) with a wildcard-origin response. The reason: if `Allow-Origin: *` were allowed with credentials, it would completely undermine the SOP — any website could make credentialed requests to any API.

The solution is `allowedOriginPatterns("*")` or a specific origin pattern. When you use patterns, Spring dynamically responds with `Access-Control-Allow-Origin: <actual-request-origin>` — the specific origin from the request, not the wildcard. Since a specific origin is echoed back, credentials mode works correctly.

The pattern syntax is flexible: `"https://*.example.com"` matches any subdomain, `"http://localhost:[*]"` matches any localhost port (useful in development), and `"https://{app,admin}.example.com"` matches specific named subdomains.

### Global vs Per-Handler CORS

You can configure CORS globally via `WebMvcConfigurer.addCorsMappings()` or per-handler via `@CrossOrigin`. When both apply to the same endpoint, they're merged using `CorsConfiguration.combine()`. The rule is: properties explicitly set in the handler annotation override global values; properties not set in the annotation fall back to global values.

Understanding the defaults of `@CrossOrigin` matters here. An empty `origins` array means "not set" (not "allow none" and not "allow all"). An empty `methods` array means "use the HTTP methods declared on the @RequestMapping". An empty `allowCredentials` string means "not set" (not false). So `@CrossOrigin` with no arguments is not as permissive as it looks — it inherits most behavior from the global configuration.

### Spring Security Integration: The Ordering Problem

Spring Security's filter chain runs *before* `DispatcherServlet`. If your Spring Security configuration requires authentication for all requests and doesn't explicitly handle CORS, the pre-flight OPTIONS request arrives at Spring Security's authorization filter, has no authentication token (because pre-flight requests don't carry credentials), and gets rejected with 401 before Spring MVC ever sees it. The browser receives a 401 for the pre-flight, treats it as failure, and never sends the actual request.

The fix is `http.cors(Customizer.withDefaults())`. This adds Spring Security's own `CorsFilter` to the security filter chain *before* authentication filters. The `CorsFilter` reads from a `CorsConfigurationSource` bean, handles the pre-flight entirely, and allows it through. For actual CORS requests, it validates the origin and sets the headers before authentication happens.

Explicitly permitting OPTIONS with `.requestMatchers(HttpMethod.OPTIONS, "/**").permitAll()` is an alternative but less robust approach — it bypasses authentication for OPTIONS but doesn't add CORS response headers, so it still depends on Spring MVC's CORS processing further down the pipeline.

## Topic 5.3 — Multipart File Upload

### What Multipart Actually Is

HTTP normally sends a request with a single body and a single `Content-Type`. Multipart changes this: the body is divided into multiple "parts" separated by a boundary string (a long random-looking delimiter). Each part has its own mini-headers (`Content-Disposition`, `Content-Type`) and its own body. One part might be the text value of a form field; another might be binary JPEG data. The server must parse this structure before it can access any of the data.

### The Parsing Pipeline

`DispatcherServlet.checkMultipart()` runs very early, at the start of `doDispatch()`. It checks whether the request's `Content-Type` starts with `multipart/`. If it does, and a `MultipartResolver` bean is present, it calls `resolveMultipart()` which parses the entire multipart body and returns a `MultipartHttpServletRequest` — a wrapper around the original request that provides access to all the parsed parts through `getFile()`, `getFileMap()`, and so on.

Cleanup always happens in the `finally` block of `doDispatch()`. Even if the handler throws an exception, even if an interceptor rejects the request, `cleanupMultipart()` is called and temporary files on disk are deleted. This is important to understand from a resource management perspective.

### The Bean Name Trap

Unlike most Spring beans which are found by type, `DispatcherServlet` looks up the `MultipartResolver` by *name* — specifically the string `"multipartResolver"`. It calls `context.getBean("multipartResolver", MultipartResolver.class)`. If your bean is named anything else — `"fileUploadResolver"`, `"myMultipartResolver"`, anything — it won't be found. The exception is caught silently, `multipartResolver` field is set to null, multipart is never parsed, and your handler receives an unparsed request where `@RequestParam("file") MultipartFile` is null. This fails silently in a confusing way.

The fix is always `@Bean(name = "multipartResolver")`. In Spring Boot, `MultipartAutoConfiguration` handles this correctly for you — it creates the `StandardServletMultipartResolver` with the right name automatically.

### StandardServletMultipartResolver vs CommonsMultipartResolver

`StandardServletMultipartResolver` is the modern choice. It uses the Servlet 3.0 `Part` API that's built into any modern servlet container, requires no extra dependencies, and is what Spring Boot auto-configures. The limitation is that size limits must be configured at the servlet level via `MultipartConfigElement`, not on the resolver bean itself.

`CommonsMultipartResolver` uses Apache Commons FileUpload, requires an extra Maven dependency, and configures size limits directly on the bean. It supports Servlet 2.x environments and offers per-file size limits (`setMaxUploadSizePerFile()`), but it's heavier and slower for large files. In any modern application, prefer `StandardServletMultipartResolver`.

### @RequestParam vs @RequestPart: The Key Distinction

Both annotations can receive a `MultipartFile`, but they differ in one important way: `@RequestPart` can deserialize a part's body through Spring's `HttpMessageConverter` chain. When a part has `Content-Type: application/json`, `@RequestPart("metadata") ProductMetadata data` will have Jackson deserialize the JSON into a `ProductMetadata` object — exactly like `@RequestBody` but for one part of a multipart request. `@RequestParam` doesn't do this; it treats part values as strings and uses form-field binding.

For simple file uploads, `@RequestParam("file") MultipartFile file` is perfectly sufficient. Use `@RequestPart` when you need to send rich structured data (JSON) alongside a file in the same request — a common pattern for "upload a product image plus metadata."

### File Security: The Rules That Matter

File upload is one of the most common attack surfaces in web applications. Several rules apply absolutely.

Never use `getOriginalFilename()` directly as a path component for storage. The filename is entirely client-controlled. A malicious client can send `../../../../etc/passwd` as the filename, and if you naively resolve that against your storage directory, you'll overwrite system files. Always extract just the filename with `FilenameUtils.getName()`, and better yet, generate a UUID-based filename entirely.

Never trust `getContentType()`. This is the `Content-Type` header from the multipart part, which the client controls. A client can upload an executable and claim it's `image/jpeg`. Always verify the actual content by reading the "magic bytes" — the first few bytes of the file that identify its format. JPEG files start with `0xFF 0xD8 0xFF`. PNG files start with `0x89 PNG`. PDF files start with `%PDF`.

Verify that the resolved storage path stays within your intended storage directory. After resolving the filename against your base path, check `targetPath.startsWith(storageLocation)`. This catches path traversal attempts that might slip through sanitization.

Store uploaded files outside the web root so they're not directly accessible via URL. If you store files in `webapp/uploads/`, they're potentially accessible at `https://yoursite.com/uploads/malicious.js`. Serving uploaded files through a controller that applies access control is the secure pattern.

### Exception Handling for Uploads

`MaxUploadSizeExceededException` is thrown when a file exceeds configured size limits. Critically, Spring's `DefaultHandlerExceptionResolver` does *not* handle this exception, so without an explicit `@ExceptionHandler`, it propagates as HTTP 500. The correct response is 413 Payload Too Large. Always add an explicit handler. The same applies to `MissingServletRequestPartException` (required file part not sent) and `MultipartException` (malformed multipart request) — these should return 400 Bad Request.

## Topic 5.4 — Async Processing

### The Thread Problem That Async Solves

Servlet containers like Tomcat have a thread pool with a bounded size (typically 200 threads). The default Spring MVC model holds a thread for the entire duration of request processing. If your handler calls an external API that takes 3 seconds to respond, that Servlet thread sits blocked for those 3 seconds. With 200 threads, you can handle at most 200 such slow requests simultaneously — the 201st request goes into a queue or gets rejected.

Async processing releases the Servlet thread as soon as the handler returns a `Callable`, `DeferredResult`, or similar async type. The thread goes back to the pool immediately. The actual work happens on a different thread or through an event callback. When the result is ready, Spring performs a second dispatch on a new Servlet thread to write the response. The net effect: your thread pool can handle far more concurrent slow requests because threads aren't blocked waiting.

### Callable<T>: The Simple Async

`Callable<T>` is the simplest async pattern. Your controller method returns a `Callable` (a lambda that returns the actual result). Spring's `CallableMethodReturnValueHandler` submits this callable to a configured `TaskExecutor` and returns immediately. The Servlet thread is released. When the callable finishes, Spring performs an async dispatch back to `DispatcherServlet`, which processes the result as if the handler had returned `T` directly — including `@ResponseBody` serialization, interceptors, and everything else.

The critical configuration point: without an explicit `TaskExecutor`, Spring uses `SimpleAsyncTaskExecutor`, which creates a *new thread per request*. This is not a thread pool — it doesn't bound the number of threads, and under load you can exhaust system resources. Always configure `ThreadPoolTaskExecutor` via `configureAsyncSupport()`. This is one of those details that works fine in testing (low concurrency) but breaks catastrophically in production.

### DeferredResult<T>: Event-Driven Async

`DeferredResult` takes a fundamentally different approach. Instead of running work on a thread pool, it simply suspends the request until *something else* sets the result. No thread is held at all — not the Servlet thread, not a worker thread. The request just waits.

This is ideal for long-polling patterns. Your handler creates a `DeferredResult`, registers it somewhere (a map keyed by user ID, an event bus subscription), and returns it immediately. The Servlet thread is released. When a notification arrives — from a message broker, an event listener, another request — the handler that receives that notification calls `deferredResult.setResult(value)`. This triggers an async dispatch to `DispatcherServlet`, which writes the response.

`setResult()` can be called from any thread, at any time, in any context. If the `DeferredResult` has already timed out or completed, `setResult()` is silently ignored and returns `false`. Always handle the timeout case by providing a default value in the constructor: `new DeferredResult<>(30_000L, Collections.emptyList())` — if 30 seconds pass without `setResult()` being called, the empty list is returned as the response.

### WebAsyncTask<T>: Callable With Lifecycle Hooks

`WebAsyncTask` wraps a `Callable` but adds timeout and error callbacks that `Callable` alone doesn't support. The `onTimeout()` callback lets you return a fallback value or throw an exception when the timeout expires. `onError()` handles exceptions thrown by the callable. `onCompletion()` runs cleanup code. This is the right choice when you need a thread-pool-based async operation but also need control over what happens at the boundaries of the async lifecycle.

### The Two-Dispatch Model and Interceptors

Understanding that async requests involve two `doDispatch()` calls is essential for understanding interceptor behavior, which is a rich source of exam questions.

On the *first dispatch*, `preHandle()` is called on all interceptors. The handler runs and returns a `Callable` or `DeferredResult`. Spring detects that async processing has started (`asyncManager.isConcurrentHandlingStarted()` is true). `postHandle()` is **not called** — because there's no response to post-process yet. Instead, `afterConcurrentHandlingStarted()` is called (an `AsyncHandlerInterceptor`-only method). The Servlet thread is released.

On the *second dispatch* (when the result is ready), `preHandle()` is called **again** on all interceptors. The result is processed as a normal return value. `postHandle()` is called. `afterCompletion()` is called.

The implication: any code in `preHandle()` that has side effects (incrementing a rate limit counter, logging a request, updating a metric) will execute twice for async requests. The fix is to check `request.getDispatcherType() == DispatcherType.ASYNC` inside `preHandle()` and skip the side effect on the second pass.

If you have logic that was in `postHandle()` (for interceptors in sync code), you need to move it to `afterConcurrentHandlingStarted()` for the async case, since `postHandle()` doesn't fire on the first dispatch.

### CompletableFuture and Reactive Types

`CompletableFuture<T>` is handled by `DeferredResultMethodReturnValueHandler`, which adapts it to a `DeferredResult` internally. The important nuance is that `CompletableFuture.supplyAsync()` without an explicit executor uses `ForkJoinPool.commonPool()`, which is a shared, bounded pool not designed for blocking I/O. If your completable futures are doing network calls, you should provide an explicit executor: `CompletableFuture.supplyAsync(task, myIoExecutor)`.

Spring MVC also supports `Mono<T>` and `Flux<T>` from Project Reactor, adapted through `ReactiveTypeReturnValueHandler`. `Mono<T>` is adapted to `DeferredResult` — it resolves when the Mono completes. `Flux<T>` with `text/event-stream` produces is routed through the `SseEmitter` mechanism, streaming each element as a Server-Sent Event. This is how you build reactive-style streaming endpoints in Spring MVC without switching to Spring WebFlux.

### Timeout Handling

The default timeout for async requests is configured via `configureAsyncSupport().setDefaultTimeout(milliseconds)`. When timeout expires for a `Callable`, Spring throws `AsyncRequestTimeoutException`. `DefaultHandlerExceptionResolver` maps this to HTTP 503 Service Unavailable. For `DeferredResult`, the timeout result value (if provided in the constructor) is used as the response. For `WebAsyncTask`, the `onTimeout()` callback's return value becomes the response.

One thing worth verifying in your implementations: `AsyncRequestTimeoutException` must be handled explicitly if you want a response other than 503. Add `@ExceptionHandler(AsyncRequestTimeoutException.class)` in a `@ControllerAdvice` if you need custom timeout responses.

## Cross-Cutting Themes Across All Four Topics

Looking across these four topics, several architectural principles repeat in different forms.

**First-match-wins ordering** appears in converter chains (Topic 4.3), resource resolver chains (Topic 4.4), and the CORS pre-flight interceptor insertion (Topic 5.2). Position in a chain matters everywhere.

**Silent failures from wrong bean names** appear in both the `MultipartResolver` (must be `"multipartResolver"`) and `MessageSource` (must be `"messageSource"`). Spring looks up certain beans by name, not by type, and when the name is wrong, the failure is silent rather than a clear exception.

**The "configure replaces, extend adds" pattern** shows up in message converters (`configureMessageConverters` vs `extendMessageConverters`) and exception resolvers (`configureHandlerExceptionResolvers` vs `extendHandlerExceptionResolvers`). Any Spring MVC hook named `configure*` replaces defaults; one named `extend*` adds to them.

**Double execution in two-pass processing** affects both CORS pre-flight (two HTTP requests) and async interceptors (two `doDispatch()` calls). In both cases, code that should run once must guard against running twice by checking the context.

**Security through verification, not trust** runs through both file uploads (never trust `Content-Type` or `getOriginalFilename()`) and CORS (it's a browser hint, not a server security boundary). The actual security must live in your authorization layer, not in surface-level checks.
