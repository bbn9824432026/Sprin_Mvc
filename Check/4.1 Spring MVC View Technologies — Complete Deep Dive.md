# Spring MVC View Technologies — Complete Deep Dive

Let's build a complete mental model of this topic by starting from first principles. The central question is: once your controller has done its work and assembled a model, *how does that data become an HTML page?* The answer is surprisingly nuanced, and the differences between the three main technologies run deeper than just syntax.

---

## The Core Abstraction — What a "View" Actually Is

Spring MVC deliberately separates the *what* (the controller produces a model and a view name) from the *how* (the view technology renders the output). This separation is embodied in a single interface:

```java
public interface View {
    void render(
        @Nullable Map<String, ?> model,
        HttpServletRequest request,
        HttpServletResponse response
    ) throws Exception;
}
```

That's it. Every view technology — JSP, Thymeleaf, FreeMarker, even a JSON serializer — implements this one method. The `model` is a map of named Java objects. The `response` is where the output goes. What happens in between is entirely up to the technology.

The rendering process inside `AbstractView.render()` has three phases that every technology shares. First, it merges static and dynamic model attributes. Second, it sets the `Content-Type` header on the response. Third, it calls `renderMergedOutputModel()`, which is where each technology diverges completely. This third step is the heart of everything that follows.

---

## JSP — The Servlet Container Does the Work

### What JSP Really Is

Here is the most important conceptual point about JSP: **it is not a Spring technology**. JSP is part of the Servlet specification, and it is processed by the Servlet container — in the case of Tomcat, a sub-component called the Jasper compiler. When a JSP file is first requested, Jasper reads the `.jsp` file, generates a Java class that extends `HttpServlet`, compiles it to bytecode, loads it, and executes it. On subsequent requests, the compiled class is reused unless the source file has changed.

Spring's role is simply to hand the request off to the container. This handoff happens via `InternalResourceView`, whose `renderMergedOutputModel()` does two things in sequence. First, it calls `exposeModelAsRequestAttributes()`, which loops through every entry in the model map and calls `request.setAttribute(key, value)` for each one. Second, it gets a `RequestDispatcher` from the container and calls `forward(request, response)`. At that point, Spring is done. The container takes over, the JSP servlet executes, and HTML is written to the response.

Understanding the `forward()` mechanism is crucial. This is an *internal server-side dispatch* — the same `HttpServletRequest` object is passed to the JSP. Nothing is lost. This is why `${product.name}` works in JSP — Expression Language resolves variables by looking through the scopes in order: page scope, then request scope, then session scope, then application scope. Spring's model data lands in request scope via `setAttribute()`, so EL finds it there.

### The `/WEB-INF/` Security Pattern

JSP files are almost always placed under `/WEB-INF/views/` and this is intentional. The Servlet specification guarantees that the container will refuse direct browser access to anything under `/WEB-INF/`. A user typing `http://yourapp.com/WEB-INF/views/product/list.jsp` directly will get a 404. However, a server-side `RequestDispatcher.forward()` *can* access `/WEB-INF/`. This means JSPs are only accessible through Spring's controller pipeline — which ensures the model is always populated before the template runs. Without this, a JSP accessed directly would throw NullPointerExceptions trying to render empty model data.

This is why JSP files **must** live in `src/main/webapp/WEB-INF/views/`, not in `src/main/resources/`. `RequestDispatcher` operates on web application paths, not classpath paths. Putting JSPs in the classpath is a common Spring Boot migration mistake.

### JstlView — The i18n Upgrade

`InternalResourceViewResolver` has a small but important configuration option: `resolver.setViewClass(JstlView.class)`. Without this, JSTL's `<fmt:message>` tag uses JSTL's own resource bundle mechanism, completely bypassing Spring's `MessageSource`. With `JstlView`, Spring exposes its `MessageSource` as the JSTL `LocalizationContext`, so `<fmt:message>` uses Spring's internationalization infrastructure — which supports property files, database-backed messages, and more. Always use `JstlView` when combining JSP with Spring's i18n.

## Thymeleaf — Pure Java, No Container Involvement

### The Architectural Difference

Thymeleaf takes a fundamentally different approach. It is a pure Java library that reads a template file, processes it internally, and writes the resulting HTML directly to the response writer. There is no `RequestDispatcher.forward()`. There is no Servlet container compilation step. Spring's `AbstractThymeleafView.render()` builds a Thymeleaf `IContext` object (which wraps the model, the request, the session, and the locale), then calls `templateEngine.process(templateName, context, response.getWriter())`. That single method call does everything.

The consequence of this architecture is profound: Thymeleaf templates are valid HTML files. A browser can open them directly without any server. The `th:text="${product.name}"` attribute doesn't break the HTML structure — the browser just ignores the `th:` namespace and renders the fallback content. A designer can look at the template in a browser and see a realistic page layout, even with placeholder content. JSP templates cannot do this because `<c:forEach>` and `<%@ ... %>` directives are not valid HTML and look like garbage in a browser.

### Expression Language — The Four Types

Thymeleaf has four distinct expression syntaxes that serve different purposes, and mixing them up is a common source of confusion.

The `${...}` syntax is a variable expression — it accesses the model. `${product.name}` reads the `name` property from the `product` object in the model. The `@{...}` syntax is a URL expression — it builds URLs and automatically prepends the context path. `@{/products/{id}(id=${product.id})}` generates `/myapp/products/42`, which is essential for applications deployed under a context path. The `#{...}` syntax is a message expression — it reads from Spring's `MessageSource` for internationalisation. `#{product.list.title}` looks up the key in `messages.properties`. Finally, the `*{...}` syntax is a selection expression — it accesses properties of whatever object is bound with `th:object`. Inside a form with `th:object="${productForm}"`, you write `*{name}` instead of `${productForm.name}`.

### Form Binding and Validation

Thymeleaf's form integration is one of its strongest features. When you write `<form th:object="${productForm}">` and then `<input th:field="*{name}"/>`, Thymeleaf generates the `id`, `name`, and `value` attributes automatically, reading the current value from the bound object. More importantly, `#fields` is a built-in utility object that exposes validation state. `${#fields.hasErrors('name')}` returns true if the `BindingResult` (which Spring MVC automatically adds to the model when validation fails) contains errors for the `name` field. This is a critical dependency: **if there is no `BindingResult` in the model, `#fields` methods throw a runtime exception**. This happens when you try to render a form without going through a controller method that has `BindingResult` in its signature.

### Caching

Thymeleaf's `SpringResourceTemplateResolver` has a `setCacheable()` property. In production, caching is enabled: a template is read and parsed once, and the parsed representation is stored in memory. Subsequent requests reuse the cached version, making template processing very fast. In development, you set `resolver.setCacheable(false)` (or `spring.thymeleaf.cache=false` in Spring Boot), which causes the template file to be re-read on every request. Forgetting to disable this during development means you edit a template, refresh the browser, and see no change — one of the most frustrating debugging experiences in Spring development.

## FreeMarker — A Different Flavour of the Same Idea

FreeMarker is architecturally similar to Thymeleaf in the ways that matter most: it's a pure Java template engine, it processes templates directly without Servlet container involvement, and `FreeMarkerViewResolver` returns `null` for missing templates. The syntax is different — it uses `<#if>`, `<#list>`, and `${variable}` directives — but the rendering pipeline is identical in structure.

FreeMarker has one particularly useful null-handling idiom worth knowing: the `!` operator provides a default value (`${product.description!("No description")}`) and the `??` operator tests for existence (`${product.description??}`). This explicit null handling is less graceful than Thymeleaf but very explicit. FreeMarker has traditionally been popular for email template generation and document generation because it doesn't presuppose HTML as the output format.

## The ViewResolver Chain — The Most Exam-Critical Mechanism

### How DispatcherServlet Finds the Right View

When a controller returns a string like `"product/list"`, `DispatcherServlet` calls `resolveViewName()` which iterates through the registered `ViewResolver` beans in order of priority (lowest `order` number wins). It calls `resolveViewName()` on each resolver in turn and stops at the first one that returns a non-null result.

Here is the single most important rule about view resolver ordering: **`InternalResourceViewResolver` must always be last**. The reason is that its `canHandle()` method always returns `true` — it will create a `View` for any string whatsoever, regardless of whether the corresponding JSP file actually exists. That file existence check doesn't happen until `render()` time, when the container tries to forward to a path that doesn't exist. If you place `InternalResourceViewResolver` first, it intercepts every view name, and Thymeleaf and FreeMarker never get a chance to run.

`ThymeleafViewResolver` and `FreeMarkerViewResolver`, by contrast, check for template existence at resolution time. If `dashboard.html` doesn't exist in the templates directory, `ThymeleafViewResolver.resolveViewName()` returns `null`, and the chain moves on. This makes them safe to place first in the chain.

### Caching in AbstractCachingViewResolver

Both `ThymeleafViewResolver` and `InternalResourceViewResolver` extend `AbstractCachingViewResolver`, which stores resolved `View` instances in a `ConcurrentHashMap` keyed by `viewName + Locale`. The locale is part of the key because the same view name might need to render differently in different locales (though usually the same `View` object handles locale internally via the `MessageSource`).

There is a subtle but important detail about failed resolutions. When a `ThymeleafViewResolver` fails to find a template, it doesn't just return `null` and forget. It stores a special `UNRESOLVED_VIEW` sentinel value in the cache. On subsequent requests for the same view name in the same locale, it finds the sentinel immediately and returns `null` without re-checking the filesystem. This is an important performance optimisation, but it has a consequence: if you add a new template file while the application is running with caching enabled, the resolver may have already cached a failure for that name. The new template won't be discovered until the cache is cleared or the application restarted.

Another critical thread-safety point: because `View` instances are cached and shared across all concurrent requests, every `View` implementation must be thread-safe. All built-in implementations from Thymeleaf, FreeMarker, and Spring are designed this way. Custom `View` implementations that store per-request state in instance fields will have race conditions.

## Model Attribute Access — The Same Surface, Different Depths

All three technologies use the same syntax on the surface: `${product.name}` works in JSP, Thymeleaf, and FreeMarker. But the mechanism underneath is different for each.

For JSP, Spring's `exposeModelAsRequestAttributes()` method explicitly calls `request.setAttribute("product", productObject)` for every model entry before forwarding. JSP EL *requires* this because it resolves variables from request scope. Without `exposeModelAsRequestAttributes()`, JSP EL would find nothing.

For Thymeleaf, the model is passed directly into the `IContext` object. Spring *also* calls `exposeModelAsRequestAttributes()` for consistency (so `${#request.getAttribute("product")}` also works in Thymeleaf), but Thymeleaf doesn't need it — it reads from the `IContext` directly.

For FreeMarker, the model is converted into FreeMarker's `TemplateModel` object hierarchy, which FreeMarker uses for its own type-safe template rendering.

---

## Internationalisation — Getting i18n Right in Each Technology

All three technologies integrate with Spring's `MessageSource`, but through slightly different mechanisms. In JSP, you need `JstlView` and the Spring tag library (`<spring:message code="key"/>`) or JSTL's `<fmt:message>` (only works with Spring MessageSource if `JstlView` is configured). In Thymeleaf, `#{key}` is natively wired to Spring's `MessageSource` — no extra configuration needed. In FreeMarker, you expose Spring's `RequestContext` as a template variable (via `resolver.setRequestContextAttribute("spring")`) and call `${spring.getMessage("key")}`.

---

## The Big Picture — Choosing and Combining Technologies

In practice, you rarely mix view technologies in the same application, but the Spring architecture supports it. The ViewResolver chain allows a Thymeleaf resolver first, a FreeMarker resolver second, and JSP last. The controller just returns a string; the resolvers figure out which technology handles it based on template availability.

For new projects, Thymeleaf is the default choice in Spring Boot and the generally recommended path because of its natural HTML templates, strong Spring integration, built-in Spring Security dialect, and active development. FreeMarker is an excellent choice for non-HTML templates (emails, text documents) where Thymeleaf's HTML-centric design would feel unnatural. JSP is a legacy technology — it works, it's stable, and it will be around indefinitely, but new projects shouldn't start with it unless there's a specific reason (like an existing JSP-heavy codebase being incrementally migrated).

The most important mental model to carry away from this topic is the **rendering mechanism contrast**: JSP hands the work to the Servlet container via `forward()`; Thymeleaf and FreeMarker do the work themselves in Java. This single architectural difference explains why JSP files must be in `WEB-INF` on the web path, why Thymeleaf templates can be anywhere on the classpath, why Thymeleaf templates render in a plain browser, and why `InternalResourceViewResolver` must always be last. Once you internalise that one difference, most of the other facts in this topic fall naturally into place.
