# WebApplicationContext Bootstrapping: How Spring Comes Alive

Let's build a mental model from the ground up before diving into the mechanics. The core question this topic answers is: **what actually happens between "you deploy your WAR file" and "your first HTTP request is handled"?**

## The Big Picture: What is Bootstrapping?

Think of bootstrapping like opening a restaurant for the day. You don't just unlock the door and start serving customers. First you stock the kitchen, hire and brief the staff, set the tables, test the equipment, and only *then* open for business. Spring does something analogous — and every step has to happen in the right order.

The entire process is orchestrated by a single method: **`AbstractApplicationContext.refresh()`**. This is one of the most important methods in all of Spring. Every `WebApplicationContext` — whether XML-based or annotation-based — funnels through here. It's `synchronized`, meaning only one thread can bootstrap a context at a time.

It has 12 phases. Let's walk through them conceptually, and I'll flag which ones matter most.

---

## Phase 1: Preparation (`prepareRefresh`)

Before anything else, Spring marks itself as "active" and timestamps the startup. The crucial web-specific thing here is `initPropertySources()`.

In a web context, this adds the `ServletContext` and `ServletConfig` init-params (from your `web.xml`) into the `Environment`'s property sources. This is why `@Value("${someContextParam}")` can resolve values you've defined as `<context-param>` entries — they're injected into the property system *right here*, before any beans are even defined.

Think of this phase as Spring putting on its glasses so it can read its configuration correctly.

---

## Phase 2: BeanFactory Creation (`obtainFreshBeanFactory`)

This phase creates the `DefaultListableBeanFactory` — the actual container that holds everything — and reads your configuration. For XML, it parses the XML files. For annotation-based config, it registers your `@Configuration` class.

**Critical insight:** at the end of Phase 2, you have *BeanDefinitions* — not beans. A `BeanDefinition` is like a recipe card. It describes a bean (its class, scope, whether it's lazy, its init method, etc.) but no actual objects have been created yet. This is a distinction that trips up many developers.

---

## Phase 3: BeanFactory Preparation (`prepareBeanFactory`)

Spring now configures the factory itself with essential infrastructure. It registers `ApplicationContextAwareProcessor` — the thing that allows beans to implement `ApplicationContextAware` and receive a reference to the context. It also registers resolvable dependencies, which is why `@Autowired ApplicationContext ctx` works anywhere in your application.

---

## Phase 4: Web-Specific Post-Processing (`postProcessBeanFactory`)

This is where `WebApplicationContext` differentiates itself from a plain `ApplicationContext`. Three important things happen:

Spring registers `ServletContextAwareProcessor` (so beans can implement `ServletContextAware`), registers the web-specific **scopes** — `request`, `session`, and `application` — and makes `ServletContext` itself injectable. After this phase, `@RequestScope` and `@SessionScope` are valid options for your beans.

---

## Phase 5: BeanFactoryPostProcessors — The Most Important Phase

This is where the real magic of annotation-driven configuration happens. The star player is `ConfigurationClassPostProcessor`, which processes your `@Configuration` classes.

Here's what it does in sequence: it scans packages specified in `@ComponentScan` and registers all `@Controller`, `@Service`, `@Repository`, and `@Component` classes as `BeanDefinition`s. It processes `@Bean` methods and registers those as definitions too. It loads property files from `@PropertySource`. It evaluates `@Conditional` annotations.

After Phase 5, the BeanDefinition registry is **complete and frozen** — no new definitions will be added. Also in this phase, `PropertySourcesPlaceholderConfigurer` resolves all `${...}` placeholders, so `@Value("${server.port}")` now has a concrete value.

A key exam/interview point: **`@ComponentScan` scanning happens in Phase 5, not Phase 11**. The *definitions* are registered in 5, the *instances* are created in 11.

---

## Phase 6: BeanPostProcessor Registration

Now Spring finds all `BeanPostProcessor` beans and registers them in priority order. The crucial ones are:

`AutowiredAnnotationBeanPostProcessor` — handles `@Autowired` and `@Value`. `CommonAnnotationBeanPostProcessor` — handles `@PostConstruct` and `@PreDestroy`. `AnnotationAwareAspectJAutoProxyCreator` — creates AOP proxies for `@Transactional`, `@Cacheable`, etc.

Here's a subtle trap: **BeanPostProcessor beans are themselves instantiated in this phase**, before regular beans. If a `BeanPostProcessor` `@Autowired`s a regular service, that service gets created *early*, before all post-processors are registered, potentially missing AOP proxying. Spring logs a warning about this. The fix is to use lazy `BeanFactory.getBean()` lookups inside the processor instead of `@Autowired`.

---

## Phases 7–10: Infrastructure Setup

These phases set up supporting infrastructure: the `MessageSource` for i18n, the `ApplicationEventMulticaster` for event publishing (synchronous by default — an important gotcha), and listener registration. In Spring Boot, Phase 9 (`onRefresh`) is also where the embedded Tomcat instance is *created* (though not yet started).

---

## Phase 11: Bean Instantiation — The Heart of the Matter

`finishBeanFactoryInitialization()` is where all your singleton beans actually come to life. For each non-lazy singleton, Spring runs this sequence:

First, the constructor is invoked to create a raw instance. Then, crucially, an early reference to this raw instance is registered in the three-level singleton cache (more on this shortly). Then `populateBean()` runs — this is where `@Autowired` fields are set via reflection. Then the Aware interfaces are called (`BeanNameAware`, `BeanFactoryAware`, etc.). Then `@PostConstruct` methods fire. Then `afterPropertiesSet()` (from `InitializingBean`). And *finally*, `postProcessAfterInitialization()` runs — this is where AOP proxies are created.

That last point is crucial: **`@PostConstruct` runs before the AOP proxy exists**. So if you call `this.someMethod()` inside `@PostConstruct` on a `@Transactional` bean, you're calling the raw instance — the transaction won't apply. This is one of Spring's most common "gotcha" moments.

---

## Phase 12: Finishing Up (`finishRefresh`)

Lifecycle beans start (in Spring Boot, this is when Tomcat *actually starts accepting connections*), and then — as the very last act — `ContextRefreshedEvent` is published. This is why `@EventListener(ContextRefreshedEvent.class)` is reliable for "do this after everything is ready" logic.

---

## The Three-Level Cache and Circular Dependencies

This deserves special attention because it's architecturally fascinating. Consider two beans, A and B, where A needs B and B needs A.

Spring's solution uses three maps: `singletonObjects` (fully initialized beans), `earlySingletonObjects` (early-exposed beans), and `singletonFactories` (factories that can produce early references).

When creating A, Spring first instantiates the raw A object, then immediately registers a factory for it in level 3. When it then tries to populate A's fields and needs B, it starts creating B. When B needs A, Spring finds A's factory in level 3, calls it to get an early reference (which may already be an AOP proxy shell), and gives that to B. B completes, A completes, and the cycle resolves.

This is why **field injection can resolve circular dependencies but constructor injection cannot**. With constructor injection, you need the dependency *to exist before the object can even be created*, so there's no opportunity to register an early reference. Spring correctly fails fast with `BeanCurrentlyInCreationException`. The field injection workaround *works*, but a circular dependency is almost always a sign of a design problem worth addressing.

---

## A Mental Exercise

To test your understanding: if you have a `@Service` class annotated with `@Transactional`, and inside its `@PostConstruct` method you call `this.getClass().getSimpleName()`, what do you get? Think through the phase sequence before reading on.

You get the plain class name — something like `"UserService"` — not a proxy name. Because `@PostConstruct` fires in the `postProcessBeforeInitialization` step, and the AOP proxy isn't created until `postProcessAfterInitialization`, which runs after. The `this` reference inside `@PostConstruct` is always the raw, unproxied object.

---

## The Key Takeaways

The entire bootstrap process can be summarized in four principles: configuration is *read* (Phase 2–5) long before beans are *created* (Phase 11); `BeanPostProcessor`s are special and get instantiated early with potential side effects; `@PostConstruct` happens before proxying; and `ContextRefreshedEvent` is the reliable "everything is ready" signal. Keep these four ideas straight, and most Spring lifecycle questions become tractable.
