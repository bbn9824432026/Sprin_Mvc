# ContextLoaderListener vs DispatcherServlet Context — Deep Dive

The previous topic gave you the *what* — two contexts in a hierarchy. This topic answers the *how* and *what can go wrong* — the internal mechanics, precise timing, failure modes, and edge cases that separate someone who understands the concept from someone who truly understands the system. Let's build this up layer by layer.

---

## What Each One Actually Does Internally

### ContextLoaderListener — The Root Context Builder

`ContextLoaderListener` implements the Servlet API's `ServletContextListener` interface. This is significant because `ServletContextListener` is a *container-level* hook — it fires when the `ServletContext` itself is created, before any servlet or filter is initialized. Spring is essentially saying "give me the very first callback the container offers, so I can build the root context before anything else happens."

When `contextInitialized()` fires, the real work is delegated to `ContextLoader.initWebApplicationContext()`. Walk through what this method does and you'll understand the entire root context bootstrap:

The very first thing it does is check whether a root context already exists in the `ServletContext`. If it finds one, it immediately throws an `IllegalStateException` — this is the guard against accidental double-registration of `ContextLoaderListener`. You'll see this exception in production if someone mistakenly adds `ContextLoaderListener` twice, either in `web.xml` or through a framework that registers it automatically alongside your own registration.

After the guard, it determines *which type* of `ApplicationContext` to create. The default is `XmlWebApplicationContext`, which reads XML config files. If you specify `contextClass=AnnotationConfigWebApplicationContext` as a context param, it uses that instead. This is the "type selection" step that the hybrid XML-plus-Java-config approach from the previous topic relied on.

Then it instantiates the context, sets the `ServletContext` reference on it, reads the `contextConfigLocation` parameter to know where your config files or classes are, and finally calls `context.refresh()`. That last call is where all the real work happens — beans are discovered, instantiated, proxied, and wired. After `refresh()` completes successfully, the root context is stored in `ServletContext` under the key `ROOT_WEB_APPLICATION_CONTEXT_ATTRIBUTE`. This storage step is the handshake — it's how `DispatcherServlet` will later find the root context and set it as its parent.

### The `context.refresh()` Journey

Both the root and child contexts go through the same `AbstractApplicationContext.refresh()` sequence — this is worth understanding because it explains *why* certain failures happen at startup rather than at request time.

The refresh sequence begins by creating the `BeanFactory` and reading all your bean definitions (scanning annotations, parsing XML, processing `@Configuration` classes). At this point beans are *defined* but not yet *created*. Then it registers post-processors — including `AutowiredAnnotationBeanPostProcessor` (which handles `@Autowired`) and `AnnotationAwareAspectJAutoProxyCreator` (which handles AOP and therefore `@Transactional`). Then — and this is the critical moment — it calls `finishBeanFactoryInitialization()`, which instantiates every singleton bean. This is where `@Autowired` is resolved, dependencies are injected, `@PostConstruct` methods run, and AOP proxies are created.

The implication is important: **if any singleton bean fails to wire up — a missing dependency, a configuration error, a `@PostConstruct` method that throws — the entire context refresh fails, and the application does not start.** Spring fails fast and loudly, which is exactly what you want. The alternative (lazily creating beans on first request) would mean a misconfigured bean might go undetected until a user triggers that code path in production.

### DispatcherServlet — The Child Context Builder

`DispatcherServlet` creates its child context inside `FrameworkServlet.initWebApplicationContext()`, which is called from `DispatcherServlet.init()`. The process mirrors what `ContextLoaderListener` does, with two additions.

First, it looks up the root context from `ServletContext` using `WebApplicationContextUtils.getWebApplicationContext()`. If `ContextLoaderListener` ran successfully, this returns the root context. If `ContextLoaderListener` was omitted, it returns `null`. Critically, **Spring handles the null case gracefully** — if there's no root context, `DispatcherServlet` simply creates its child context with no parent. This is not an error. This is, in fact, exactly what happens in Spring Boot.

Second, after the child context is refreshed, `DispatcherServlet` calls `initStrategies()`. This is a web-MVC-specific initialization step that examines the child context for `HandlerMapping` beans, `HandlerAdapter` beans, `ViewResolver` beans, and so on. It builds the internal machinery that `doDispatch()` will use later to process requests. This step has no equivalent in the root context — it's the thing that transforms a generic Spring container into a fully wired MVC dispatcher.

## The Initialization Timeline — Precise Ordering

This is the part of the topic that certification questions love to probe, because the ordering has real consequences. Let's trace the exact sequence:

When Tomcat starts your application, the first thing created is the `ServletContext`. This triggers `ServletContextListener` callbacks — so `ContextLoaderListener.contextInitialized()` fires first. The root context is built and all your `@Service`, `@Repository`, and infrastructure beans are instantiated. `ContextRefreshedEvent` is published from the root context.

Only after all `ServletContextListener` callbacks complete does the container initialize filters. This is crucial: **filters are initialized after `ContextLoaderListener` but before `DispatcherServlet`.** If you have a Spring Security filter (which is itself a filter) that needs to look up beans from a Spring context, it can safely access the root context — it already exists. But it cannot access beans from the child (web) context — that doesn't exist yet.

After filters are initialized, Tomcat initializes servlets. If `load-on-startup=1` is set on `DispatcherServlet`, it initializes now — the child context is built, controllers are instantiated, `initStrategies()` runs. If `load-on-startup` is not set (or is negative), the servlet is *not* initialized at startup. The first HTTP request to arrive will trigger `DispatcherServlet.init()` on that request's thread, paying the full initialization cost while that user waits. In production, this typically manifests as a timeout or an extremely slow first response after deployment — a subtle but serious bug. Always set `load-on-startup=1`.

## What Happens When Things Go Wrong — Failure Analysis

Understanding failure modes is often more valuable than understanding the happy path, because failures are what you'll actually debug in production.

**Two `ContextLoaderListener` registrations.** The second one fires, checks `ServletContext`, finds the root context attribute already set, and throws `IllegalStateException` immediately. The application fails to start. This happens more often than you'd expect — some frameworks or Spring integrations automatically register `ContextLoaderListener`, and you add your own on top of it.

**Config file not found.** If `contextConfigLocation` points to a nonexistent XML file, `XmlWebApplicationContext` tries to load it during `refresh()`, fails to resolve the resource, and throws a `BeanDefinitionStoreException` wrapping a `FileNotFoundException`. Application fails to start. This also applies to the child context — a missing `dispatcher-servlet.xml` (or whatever the default convention resolves to) causes the same failure during `DispatcherServlet.init()`.

**A bean fails to wire up during `refresh()`.** Say a `@Service` has an `@Autowired` dependency that doesn't exist. During `finishBeanFactoryInitialization()`, Spring tries to create that bean, fails to find the dependency, and throws `UnsatisfiedDependencyException`. Context refresh fails. Whichever context failed (root or child) does not become operational. If the root context fails, the entire application typically fails to start. If the root context succeeds but the child context fails, the root context is alive but `DispatcherServlet` is not initialized — all requests to it return HTTP 500.

**The silent transaction failure.** This one deserves special attention because it doesn't cause a startup failure at all. If your `WebConfig`'s component scan accidentally picks up `ProductService` (because the scan covers too broad a package), the child context creates its own `ProductService` instance. This instance is a plain Java object — no transaction proxy, because the `TransactionManager` and `@EnableTransactionManagement` live in the root context and only proxy beans within that context. When the controller autowires `ProductService`, Spring finds the child context's local instance first (local registry is always checked before delegating to the parent). The controller gets the non-transactional copy. `@Transactional` does nothing. No warning, no exception, no indication anything is wrong until data corruption occurs.

## The Single-Context Pattern — Understanding When the Hierarchy Disappears

One of the most important things to internalize is that **`ContextLoaderListener` is optional**. This surprises many developers who have come to think of the two-context hierarchy as fundamental to Spring MVC. It is a best practice, not a requirement.

If you return `null` from `getRootConfigClasses()` in `AbstractAnnotationConfigDispatcherServletInitializer`, Spring sees the null, skips registering `ContextLoaderListener`, and creates only the child context with no parent. All beans — services, repositories, controllers, infrastructure — live in that single context. This works perfectly. The trade-off is that multiple `DispatcherServlet` instances can no longer share beans (each would have their own isolated context with no shared root), and the architectural separation between web and service layers is not enforced at the container level.

Spring Boot is essentially formalizing this single-context pattern and making it the default. `SpringApplication.run()` creates one `AnnotationConfigServletWebServerApplicationContext` that contains everything. `DispatcherServlet` is registered as a bean within this context rather than as a separate entity that creates its own context. The embedded Tomcat is also managed within this same context. The entire two-context hierarchy simply doesn't exist in Boot — which is why the duplicate-scan bug and the `ContextRefreshedEvent`-fires-twice problem are both Boot-immune.

## Two Easy Things to Confuse — `containsBean()` vs `containsLocalBean()`

This is a subtle API distinction that the topic document flags as an exam trap, and it reveals something important about how the hierarchy search works.

`applicationContext.containsBean("productService")` searches the local context *and* walks up the parent chain. So if `productService` lives in the root context and you call this on the child context, it returns `true`.

`applicationContext.containsLocalBean("productService")` searches *only* the local context's own registry. The same call on the child context returns `false` if `productService` is only in the root.

This distinction matters when you're doing programmatic context inspection or writing infrastructure code that needs to know *where* a bean actually lives, not just whether it's accessible.

## The `ContextRefreshedEvent` Propagation — One More Time, Precisely

This was mentioned in the previous topic but the mechanics deserve a precise statement here. When the child context finishes refreshing and publishes `ContextRefreshedEvent`, Spring's event system propagates that event *upward* through the parent chain. So any `@EventListener` for `ContextRefreshedEvent` that is registered in the root context will receive both the root's own event and the child context's event. A listener in the child context only receives the child's event — events do not propagate downward.

The practical consequence is that "run once on startup" initialization code placed in a root-context bean will run twice. The standard fix is checking `event.getApplicationContext().getParent() == null` — only the root context has no parent, so this condition being true means you're handling the root's own event, not the child's.

## A Mental Model to Tie It Together

Think of the whole system as a two-stage rocket launch. `ContextLoaderListener` is the first stage — it fires before anything else, builds the foundation (services, data access, security), and then hands off. It stores its result (the root context) in a shared location (`ServletContext`) so the second stage can find it. `DispatcherServlet` is the second stage — it fires after the first stage is complete, finds the first stage's output, builds the web layer on top of it, and then the full system is in orbit (ready to serve requests).

If the first stage fails, the rocket never launches. If the first stage succeeds but the second stage fails, you have a root context running but no web handling. And if you remove the first stage entirely (no `ContextLoaderListener`), the second stage just has to carry everything itself — which works, but means you can only have one rocket at a time.
