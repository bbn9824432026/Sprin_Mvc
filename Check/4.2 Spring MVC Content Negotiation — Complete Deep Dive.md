# Spring MVC Content Negotiation — Complete Deep Dive

Let's build this topic from first principles, because content negotiation is one of those subjects where memorising the rules without understanding the *why* leads to constant confusion. The core question the entire mechanism is trying to answer is deceptively simple: when a client asks your server for data, how does the server decide what *format* to send it in?

---

## The Fundamental Problem

Imagine you have a `/api/products` endpoint. A browser might want HTML. A JavaScript frontend might want JSON. An enterprise integration might want XML. A data analyst might want CSV. You don't want four separate endpoints — you want one endpoint that serves the right format to the right client. Content negotiation is the machinery that makes this happen automatically.

The process has two phases that always happen in sequence. First, Spring determines what the *client wants*. Second, Spring determines what the *server can produce*. The negotiation is finding the best match between those two lists, and if no match exists, returning HTTP 406 Not Acceptable.

---

## Phase 1 — What Does the Client Want?

The `ContentNegotiationManager` is responsible for Phase 1. It holds an ordered list of strategies, and it asks each strategy in turn: "given this request, what media types does the client want?" The first strategy that returns a specific answer (anything other than `*/*`) wins, and the others are not consulted.

### The Accept Header Strategy

The most fundamental strategy reads the HTTP `Accept` header, which the client sends to declare its preferences. The header can express multiple acceptable types with quality factors (the `q` parameter) that indicate relative preference:

```
Accept: text/html, application/json;q=0.9, */*;q=0.8
```

This says: "I prefer HTML (q=1.0 implied), I'll accept JSON at 90% preference, and I'll accept literally anything at 80% preference." Spring parses this header and sorts the types by their quality values. `*/*` is a wildcard meaning "anything" — browsers include it as a last resort so they don't get a 406 from servers that can't produce HTML.

### The Parameter Strategy

An alternative strategy watches for a query parameter like `?format=json`. When enabled with `favorParameter(true)`, this strategy runs *before* the Accept header strategy. If the parameter is present and maps to a known media type, it wins immediately without even looking at the Accept header.

There's a critical trap here: if you include `?format=pdf` but `pdf` is not registered in the format-to-media-type mappings, the strategy throws `HttpMediaTypeNotAcceptableException` immediately, giving a 406. This is different from *omitting* the parameter entirely, which returns `*/*` and falls through to the next strategy. An unknown value is treated as an error, not as silence.

### The Path Extension Strategy (Deprecated)

For completeness: there used to be a third strategy that inferred format from URL extensions like `/products.json`. This was deprecated in Spring 5.3 and removed in Spring 6 due to security vulnerabilities around content-type spoofing and suffix pattern exploits. Don't use it in new code.

### How the Strategies Interact

The interaction is a strict cascade. With `favorParameter=true` and `ignoreAcceptHeader=false`, the order is: check the parameter first, and if it returns a specific type, stop there; otherwise fall through to the Accept header. Setting `ignoreAcceptHeader=true` simply removes the Accept header strategy from the chain entirely — which is useful for APIs where you always want clients to use the parameter. However, and this is a major trap we'll return to, ignoring the Accept header in the `ContentNegotiationManager` does *not* globally disable Accept header processing everywhere in Spring MVC.

---

## Phase 2 — What Can the Server Produce?

The server's producible types come from two sources: the `@RequestMapping(produces=...)` annotation on the handler method (if present), and the set of all registered `HttpMessageConverter` instances and what they support.

`HttpMessageConverter` is the workhorse interface. Each implementation knows how to read a certain type from the request body and write a certain type to the response body, and it declares which Java classes and which media types it handles. The default converter chain that Spring registers automatically includes `MappingJackson2HttpMessageConverter` for JSON (if Jackson is on the classpath), `MappingJackson2XmlHttpMessageConverter` for XML, `StringHttpMessageConverter` for plain text, and several others.

---

## The Matching Algorithm — writeWithMessageConverters()

Once Spring has both lists (what the client wants and what the server can produce), it runs a matching algorithm inside `writeWithMessageConverters()`. The steps are worth understanding in detail because they explain several otherwise mysterious behaviours.

First, it builds a list of *compatible pairs* by checking every acceptable type against every producible type using `MediaType.isCompatibleWith()`. The compatibility rules follow a wildcard logic: `*/*` is compatible with everything; `text/*` is compatible with any `text/something` type; two concrete types like `application/json` and `application/xml` are only compatible if they're the same.

Second, it calls `MediaType.sortBySpecificityAndQuality()` on the compatible list. This sorting has a subtle but important rule: **specificity beats quality**. If you have `Accept: application/json;q=0.5, */*;q=1.0`, you might naively think `*/*` at quality 1.0 should win. It doesn't. `application/json` is a concrete, specific type and sorts above `*/*` regardless of the quality values. This matters enormously for understanding browser request handling, where browsers send a wildcard `*/*` at low quality but specific types like `text/html` at high quality.

Third, it picks the first concrete type from the sorted list, finds the first `HttpMessageConverter` that can write that type, and uses it to write the response body and set the `Content-Type` header.

---

## The produces Annotation — Two Distinct Roles

This is the single most important conceptual distinction in this topic, and it's almost always misunderstood. The `produces` attribute on `@GetMapping` serves *two completely separate roles* at two different points in the request lifecycle.

**Role 1: Handler selection.** When `DispatcherServlet` is figuring out which controller method to invoke, it evaluates `ProducesRequestCondition` against the request's Accept header. If the client's Accept header is not compatible with any of the declared `produces` values, this handler returns `null` — it's as if the URL didn't match — and the dispatcher either tries another handler or raises a 406. This is what allows you to have two methods mapped to the same URL but differentiated by their `produces` values:

```java
@GetMapping(value = "/export", produces = "application/json")
public List<Product> exportAsJson() { ... }

@GetMapping(value = "/export", produces = "text/csv")
public String exportAsCsv() { ... }
```

The request's Accept header determines which method is invoked. This is genuinely different behaviour from most request mapping conditions, which are about *what the client sends*, not *what it accepts*.

**Role 2: Restricting producible types.** Even after a handler is selected and runs, content negotiation still happens inside `writeWithMessageConverters()` to choose a converter. When `produces` is declared, it narrows the set of producible types to only what's declared, rather than allowing all converter-supported types. Without a `produces` annotation, a `List<Product>` return type could potentially be served as JSON, XML, or any other format that a registered converter supports.

One important nuance: `Accept: */*` is compatible with *any* concrete `produces` value. So a handler annotated with `produces="application/json"` will still be selected when the client sends `Accept: */*` — the wildcard matches everything. This means browser requests, which typically include `*/*` as a fallback, will match your JSON endpoints.

---

## The ignoreAcceptHeader Trap

There's a critical gotcha that trips up even experienced developers. When you configure `ignoreAcceptHeader(true)` in the `ContentNegotiationManager`, it removes the `HeaderContentNegotiationStrategy` from the strategy chain. This affects the body-writing phase — the converter selection during `writeWithMessageConverters()` won't consult the Accept header.

However, it does *not* affect `ProducesRequestCondition`. Handler mapping still reads the Accept header to filter handlers by their `produces` declarations. So you can have the strange situation where handler selection respects the Accept header but body writing ignores it. This is technically correct because they're two separate systems, but it's deeply confusing and a common exam trap.

---

## configureMessageConverters vs extendMessageConverters

This is the most common practical mistake when customising content negotiation. `WebMvcConfigurer` offers two hooks that sound similar but behave completely differently.

`configureMessageConverters()` *replaces* the entire default converter list. If you override this method and add only your custom CSV converter, you've removed Jackson from the application entirely. Every request for `application/json` will now get a 406. This is almost never what you want.

`extendMessageConverters()` *augments* the default list. Jackson, XML, and all the built-ins remain registered, and your custom converters are added alongside them. This is almost always the right choice when adding a custom format.

## How a Browser Request Gets Routed

Let's trace through a concrete example to cement everything. A browser requests `GET /api/products` and sends:

```
Accept: text/html, application/xhtml+xml, application/xml;q=0.9, */*;q=0.8
```

Your controller method is annotated with `@GetMapping(produces = "application/json")` and returns `List<Product>`.

At handler selection time, `ProducesRequestCondition` checks: is `application/json` compatible with any of the client's acceptable types? Yes — `*/*;q=0.8` is compatible with everything. Handler is selected.

At body writing time, the compatible pairs include (`*/*`, `application/json`) at quality 0.8, and nothing else (the other explicit types from the browser aren't compatible with `application/json`). After sorting, `application/json` is selected as a concrete type, Jackson serialises the list, and the response goes out as JSON.

Now imagine you remove the `produces` annotation. The producible types expand to everything Jackson supports — `application/json` and `application/xml`. The compatible pairs now include (`application/xml;q=0.9`, `application/xml`) and (`*/*;q=0.8`, `application/json`) and (`*/*;q=0.8`, `application/xml`). After specificity/quality sorting, `application/xml` at q=0.9 beats the wildcard-derived options. The browser gets XML, which was probably not your intention. This is a real-world surprise that many developers encounter when first building Spring APIs without `produces` annotations.

## ContentNegotiatingViewResolver — The Same Idea for Views

For traditional view-based controllers (not `@ResponseBody`), there's a parallel mechanism called `ContentNegotiatingViewResolver`. Instead of choosing a message converter, it chooses a *view*. It collects candidate views from all other registered `ViewResolver` instances, then uses the `ContentNegotiationManager` to find the view whose content type best matches the client's Accept header. A common configuration puts a `MappingJackson2JsonView` as a default view — this means that when the client sends `Accept: application/json` to a view-based controller, they get the model serialised as JSON rather than an HTML page, without any changes to the controller code.

---

## A Mental Model for the Whole Thing

Think of content negotiation as a matchmaking process between client preferences and server capabilities. The client makes a prioritised wish list (Accept header or format parameter). The server assembles its capabilities (registered converters or available views). The matchmaker (Spring) finds the highest-priority client wish that the server can actually fulfil. If no match exists, the response is "I can't give you anything you want" — HTTP 406.

The `produces` annotation is like a bouncer at the door: it can reject clients before they even get to the matchmaker, based purely on what they've declared they want. The `ContentNegotiationManager` is the matchmaker itself. The `HttpMessageConverter` chain is the server's capability catalogue. When you understand which component plays which role, the entire mechanism — including all the trap cases — becomes predictable rather than mysterious.
