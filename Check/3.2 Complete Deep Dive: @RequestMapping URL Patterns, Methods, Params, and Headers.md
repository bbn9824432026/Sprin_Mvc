# Complete Deep Dive: @RequestMapping URL Patterns, Methods, Params, and Headers

Let me guide you through this topic by building your understanding from the ground up. While the previous topic focused on what happens after a request reaches a handler, this topic is all about how Spring decides which handler should receive the request in the first place. This is the matching and routing logic that runs on every single HTTP request.

## Understanding RequestMapping as Multi-Dimensional Matching

When you write `@RequestMapping` on a controller method, you're doing something far more sophisticated than just saying "this URL goes to this method." You're actually defining a multi-dimensional filter specification. Think of it like this: every incoming HTTP request has multiple characteristics - its URL, its HTTP method (GET, POST, etc.), its parameters, its headers, the content type it's sending, and what content types it can accept. Your `@RequestMapping` defines criteria across all these dimensions, and Spring will only route a request to your handler if the request matches on every single dimension where you've specified criteria.

This is fundamentally different from simpler web frameworks where routing is just URL-based. In Spring MVC, the URL might match perfectly, but if the HTTP method is wrong, or if a required parameter is missing, or if the client can't accept the content type you produce, Spring rejects that mapping and keeps searching for a better match. Understanding this multi-dimensional nature is crucial because it explains why certain requests succeed while others return 404, 405, 406, or 415 errors.

Let me show you what I mean. When you write this mapping, you're simultaneously defining six independent conditions:

```java
@PostMapping(
    value = "/products/{id}",
    params = "action=update",
    headers = "X-API-Version=2",
    consumes = "application/json",
    produces = "application/json"
)
public Product updateProduct(@PathVariable Long id, @RequestBody Product product) {
    return productService.update(id, product);
}
```

For this mapping to be selected, Spring must verify that the URL matches the pattern `/products/{id}`, the HTTP method is POST (from `@PostMapping`), there's a query parameter named `action` with the value `update`, there's a header named `X-API-Version` with the value `2`, the request's Content-Type is `application/json`, and the client's Accept header indicates it can accept `application/json`. If even one of these conditions fails, Spring discards this mapping and continues searching through other registered mappings.

## How RequestMapping Annotations Are Discovered and Registered

Before we dive into each dimension, you need to understand the lifecycle of how these mappings come into existence. When your Spring application starts up, a bean called `RequestMappingHandlerMapping` performs a crucial initialization task. It scans through all beans in your application context looking for controllers. How does it know what's a controller? It uses that `isHandler()` method we discussed in the previous topic - it checks whether a class has the `@Controller` annotation or whether it has `@RequestMapping` directly on the class.

Once it identifies a controller class, it reflects over every public method looking for `@RequestMapping` annotations, or any of the specialized variants like `@GetMapping`, `@PostMapping`, etc. For each method with a mapping annotation, it constructs a `RequestMappingInfo` object. This object is the internal representation of all those matching conditions we discussed. It's not just storing the annotation attributes as strings - it's converting them into smart matching logic.

The URL patterns get parsed into pattern objects. If you're using the modern `PathPattern` system (which is the default since Spring 5.3), each pattern string is compiled into a `PathPattern` object that can efficiently match incoming URLs. The HTTP methods get stored as an enum set. The params and headers conditions get parsed into condition objects that know how to evaluate whether a request satisfies them. The consumes and produces media types get parsed into `MediaType` objects that understand wildcards and quality factors.

All of these `RequestMappingInfo` objects get stored in a registry, indexed in multiple ways for fast lookup. When a request arrives, Spring can quickly find candidate mappings by URL pattern, then apply the other dimensions to narrow down to the best match.

## Dimension One: URL Patterns and Their Matching Rules

Let's start with the most fundamental dimension - the URL pattern. This is what most developers think about first when they write `@RequestMapping`, but there's much more sophistication here than appears at first glance.

The simplest pattern is an exact literal match. When you write `@GetMapping("/products")`, you're saying this method handles requests to exactly that path. No wildcards, no variables, just a direct match. This is the most specific type of pattern, and when Spring has multiple patterns that could match a URL, the exact literal always wins.

Now let's add complexity. URI templates let you capture parts of the URL as variables. When you write `@GetMapping("/products/{id}")`, the curly braces define a path segment that can match any value, and that value gets extracted and made available to your method via `@PathVariable`. The crucial thing to understand here is that `{id}` matches exactly one path segment. It will match `/products/42` and extract `id=42`, but it will not match `/products/42/details` because that has an additional path segment after the variable.

Here's something that trips up many developers: the variable name inside the curly braces doesn't matter for pattern matching. The patterns `/products/{id}` and `/products/{productIdentifier}` are identical as far as Spring's matching logic is concerned. They both match the same URLs. If you tried to register both patterns on different methods in the same controller, Spring would throw an `IllegalStateException` at startup because they're ambiguous - there's no way to decide which method should handle a request to `/products/42`.

You can make URI templates more specific by adding regular expression constraints. When you write `/products/{id:\\d+}`, you're saying the `id` segment must consist entirely of digits. Now this pattern and `/products/{slug:[a-z-]+}` are genuinely different patterns - the first matches `/products/123` while the second matches `/products/premium-widgets`. These can coexist in the same controller without ambiguity because they match different URLs.

Wildcards give you even more flexibility. The single-star wildcard `*` matches any single path segment. So `/products/*` matches `/products/featured` and `/products/42`, but not `/products/featured/electronics` because that has multiple segments after `/products/`. The double-star wildcard `**` matches any number of path segments including zero. The pattern `/products/**` matches `/products/`, `/products/featured`, `/products/featured/electronics`, and `/products/a/b/c/d/e`. This is the least specific pattern type.

There's also a special syntax available in the modern `PathPattern` system: `{*variable}`. This captures the entire remaining path into a single variable. If you write `/files/{*path}`, a request to `/files/documents/reports/2024/q1.pdf` will match, and the variable `path` will contain `/documents/reports/2024/q1.pdf` - notice it includes the leading slash and all the intermediate slashes. This is different from `/**` which matches but doesn't capture.

## How Pattern Specificity Determines Winner

When multiple patterns could match the same URL, Spring needs a deterministic way to choose one. It uses a specificity algorithm that compares patterns to determine which is more specific. Let me walk you through the hierarchy.

Exact literal paths are always most specific. If you have mappings for `/products/featured`, `/products/{id}`, and `/products/*`, and a request comes in for `/products/featured`, the exact literal wins every time. It's the most precise match possible.

Among patterns with variables, those with regex constraints are more specific than those without. The pattern `/products/{id:\\d+}` is more specific than `/products/{id}` because it's more restrictive - it matches fewer possible URLs.

Patterns with more literal segments before variables are more specific. The pattern `/api/v1/products/{id}` is more specific than `/api/{version}/products/{id}` because it has more fixed segments.

Single-star wildcards are less specific than URI templates, and double-star wildcards are the least specific of all. This hierarchy ensures that more precise matches are preferred, which is usually what you want.

But here's the critical insight: URL pattern specificity is only one factor. Spring also considers specificity across the other dimensions. A mapping with an exact HTTP method specified is more specific than one that accepts all methods. A mapping with params conditions is more specific than one without. When comparing two mappings that could both handle a request, Spring considers all dimensions together to find the most specific overall match.

## Dimension Two: HTTP Methods and Their Special Handling

The HTTP method dimension controls which HTTP verbs your handler accepts. When you use `@GetMapping`, you're restricting the handler to only GET requests. Behind the scenes, this is equivalent to `@RequestMapping(method = RequestMethod.GET)`.

Here's something that surprises many developers: if you use plain `@RequestMapping` without specifying a method attribute at all, your handler will accept any HTTP method - GET, POST, PUT, DELETE, PATCH, HEAD, OPTIONS, even TRACE. This is rarely what you want and can be a security issue. If you intended a handler only for GET but forgot to specify the method, someone could call it with POST, potentially bypassing CSRF protection if you have that enabled.

The specialized mapping annotations like `@GetMapping`, `@PostMapping`, etc., are there to make your intent clear and prevent this mistake. They're not just syntactic sugar - they're safer because they force you to explicitly choose a method.

HEAD requests have special handling. When you define a `@GetMapping`, Spring automatically handles HEAD requests to the same URL. A HEAD request is supposed to return the exact same headers as a GET request would, but without any response body. Spring achieves this by calling your GET handler method normally, letting it execute and set all the headers it wants, but then stripping out the response body before sending the response. You don't have to write separate handlers for HEAD - they're free with every GET.

OPTIONS requests also have automatic handling. When a client sends an OPTIONS request to ask "what HTTP methods does this endpoint support?", Spring automatically responds with an Allow header listing all the methods that have handlers registered for that URL. You can override this by writing an explicit OPTIONS handler if you want custom behavior.

When a request's URL pattern matches but the HTTP method doesn't, Spring doesn't silently fail. It throws `HttpRequestMethodNotSupportedException`, which the default exception resolver converts to HTTP 405 Method Not Allowed. The response includes an Allow header showing which methods are supported. This helps clients understand what went wrong - the URL exists, but they used the wrong verb.

## Dimension Three: Request Parameters as Matching Conditions

The params dimension lets you differentiate handlers based on query parameters or form parameters. This is incredibly powerful for creating different behaviors for the same URL based on what parameters are present.

The syntax has four forms, and understanding each is important. The simplest is just a parameter name: `params = "featured"`. This means the parameter named `featured` must be present in the request. The value doesn't matter - it could be `?featured=true` or `?featured=false` or even `?featured=` with no value. As long as the parameter exists, the condition is satisfied.

You can negate this: `params = "!featured"` means the parameter must not be present. If the request has a `featured` parameter with any value, this condition fails.

You can require a specific value: `params = "type=premium"` means there must be a `type` parameter and its value must be exactly `premium`. The string `?type=standard` would not match because the value is wrong. Neither would a request with no `type` parameter at all.

You can also negate values: `params = "status!=deleted"` means either the `status` parameter must be absent, or if it's present, its value must not be `deleted`. Both `?status=active` and a request with no status parameter would satisfy this condition.

When you specify multiple param conditions, all of them must be satisfied. If you write `params = {"active=true", "type", "!deleted"}`, the request must have `active=true`, must have a `type` parameter with any value, and must not have a `deleted` parameter.

This is commonly used for API versioning or feature flags. You might have multiple handlers for `/api/products`, differentiated by a version parameter: one has `params = "version=1"`, another has `params = "version=2"`, and a third has no params condition to serve as the default. When a request comes in with `?version=1`, Spring selects the first handler. With `?version=2`, it selects the second. With no version parameter, it falls back to the default.

If the URL and method match but the params condition fails, Spring throws `UnsatisfiedServletRequestParameterException`, which becomes HTTP 400 Bad Request. The client's request was malformed according to your API's requirements.

## Dimension Four: Headers as Matching Conditions

The headers dimension works exactly like params but applies to HTTP headers instead of query parameters. You use the same four syntax forms: header presence, header absence, header equals value, and header not-equals value.

A common use case is API versioning through custom headers. Instead of `?version=1` in the URL, you might require clients to send `X-API-Version: 1` as a header. Your mapping would be `headers = "X-API-Version=1"`. This keeps version information out of the URL, which some API designers prefer.

You can require authentication headers: `headers = "X-API-Key"` means the request must have an `X-API-Key` header with any value. Your handler can then validate that value, but first Spring ensures it exists at all.

You can differentiate mobile vs web clients: `headers = "X-Client-Type=mobile"` for one handler and `headers = "X-Client-Type=web"` for another, both serving the same URL. The header tells Spring which handler to route to.

Here's an important subtlety: while you can use headers to match on `Content-Type`, you shouldn't. There's a specific `consumes` attribute for that, and it has smarter matching logic. If you write `headers = "Content-Type=application/json"`, Spring does exact string matching. But a request with `Content-Type: application/json; charset=UTF-8` would fail to match because the string is different - it has that charset parameter. The `consumes` attribute, on the other hand, understands media type semantics and would correctly match this. Always use `consumes` for Content-Type matching and `produces` for Accept matching.

The same applies to the Accept header - use `produces` instead of `headers` for better matching logic.

When headers conditions fail, Spring throws an exception that gets mapped to HTTP 400 Bad Request, just like params failures. The request didn't meet the required header criteria.

## Dimension Five: Consumes and Request Content Types

The consumes attribute controls what Content-Type your handler accepts. This only matters for requests that have a body - POST, PUT, and PATCH typically. GET and DELETE requests usually don't have bodies, so consumes has no effect on them.

When you write `consumes = "application/json"`, you're telling Spring: this handler only processes requests with JSON bodies. When a POST request arrives, Spring looks at the Content-Type header. If it says `application/json`, the mapping matches. If it says `application/xml`, the mapping is rejected.

The matching logic understands media type semantics. Media types have a type and subtype separated by a slash, like `application/json`. They can also have parameters, like `application/json; charset=UTF-8`. When checking if a request matches `consumes = "application/json"`, Spring treats `application/json`, `application/json; charset=UTF-8`, and `application/json; charset=ISO-8859-1` as all matching. The charset parameter doesn't matter - they're all JSON.

You can use wildcards. The pattern `application/*` matches any application media type - `application/json`, `application/xml`, `application/pdf`, etc. The pattern `*/*` matches anything, which is the default if you don't specify consumes at all.

You can specify multiple accepted types: `consumes = {"application/json", "application/xml"}`. This handler accepts either JSON or XML input. Spring doesn't care which one the request sends - as long as it's one of the listed types, the mapping matches.

You can use negation: `consumes = "!text/plain"` means accept anything except plain text. This is less common but occasionally useful.

When a request's Content-Type doesn't match the consumes condition, Spring throws `HttpMediaTypeNotSupportedException`, which becomes HTTP 415 Unsupported Media Type. This tells the client: "I understood your request and found a handler for this URL and method, but I can't process the media type you're sending."

Here's something important: consumes affects which mapping gets selected, but it doesn't directly affect how the request body gets deserialized. That's handled by `HttpMessageConverter` selection later in the process. The consumes check is purely about routing - ensuring the request goes to a handler that's prepared to deal with that content type.

## Dimension Six: Produces and Response Content Negotiation

The produces attribute is about what content types your handler can generate, and it works with the client's Accept header to determine compatibility. This is content negotiation - the client says what it wants, and Spring tries to find a handler that can produce something compatible.

When you write `produces = "application/json"`, you're making two declarations. First, you're saying this handler can only be selected if the client can accept JSON. Second, you're telling Spring what Content-Type to set on the response.

The matching uses the Accept header from the request. If a client sends `Accept: application/json`, and your mapping has `produces = "application/json"`, they're compatible - the mapping matches. If the client sends `Accept: application/xml`, the mapping is rejected.

The wildcard `*/*` in the Accept header means the client accepts anything. Most browsers send this as a fallback. So `Accept: */*` matches any produces condition - the client is saying "I'll take whatever you give me."

You can specify multiple producible types: `produces = {"application/json", "application/xml"}`. Now your handler can satisfy clients that want JSON and clients that want XML. When a request comes in with `Accept: application/json`, Spring notes that this handler can produce JSON and selects it. The selected media type (JSON in this case) determines which `HttpMessageConverter` will serialize the response.

This is how you can have two handlers for the same URL and HTTP method, differentiated only by what they produce. One has `produces = "application/json"` and returns objects that get serialized to JSON. The other has `produces = "text/csv"` and returns strings formatted as CSV. When a request comes in, the Accept header determines which handler gets invoked.

Quality factors add another layer. An Accept header can say `Accept: application/json;q=0.9, application/xml;q=1.0`, meaning "I prefer XML (quality 1.0) but will accept JSON (quality 0.9)." Spring considers these quality factors when multiple handlers could satisfy the request, preferring to match higher-quality acceptable types.

When no handler can produce something the client accepts, Spring throws `HttpMediaTypeNotAcceptableException`, which becomes HTTP 406 Not Acceptable. This means "I found handlers for this URL and method, but none of them can produce a content type you're willing to accept."

## How Class-Level and Method-Level Mappings Combine

This is where things get intricate, and understanding the exact combination rules is essential. When you have `@RequestMapping` on both a class and a method, Spring doesn't just ignore one or the other - it combines them according to specific rules that vary by attribute.

For URL paths, the combination is straightforward concatenation. If your class has `@RequestMapping("/api/v1")` and your method has `@GetMapping("/products")`, the effective URL is `/api/v1/products`. Spring just strings them together with a separator. If the method path starts with a slash, that's handled correctly - no double slashes.

For HTTP methods, the rule is: if the method level specifies any methods, use those and ignore class-level methods. If the method level specifies no methods, inherit from the class level. There's no intersection or merging. So if your class has `method = {GET, POST}` and your method uses `@GetMapping` (which specifies GET), the effective method set is just GET - the method level wins. But if your method uses plain `@RequestMapping` with no method attribute, it inherits `{GET, POST}` from the class.

For params and headers, both levels are applied with AND logic. If the class has `params = "version=2"` and the method has `params = "category=electronics"`, the effective condition is that both must be satisfied - the request must have `version=2` AND `category=electronics`. Both the class-level filter and the method-level filter must pass.

For consumes and produces, method-level overrides class-level entirely if the method specifies anything. This is not merging or AND logic. If the class says `produces = "application/json"` and the method says `produces = "text/csv"`, the effective produces is just `text/csv`. The class-level value is completely discarded. But if the method doesn't specify produces at all, it inherits from the class.

Why these different rules? It comes down to use cases. URL path concatenation lets you factor out common prefixes. Params and headers AND logic lets you apply common filters across all methods while adding method-specific filters. Consumes and produces override logic lets you have a default content type for most methods while overriding for specific endpoints that need different types.

## Property Placeholders for Dynamic Configuration

You can use Spring's property placeholder syntax in your URL patterns. When you write `@GetMapping("${api.base.path}/products")`, Spring resolves that placeholder at application startup by looking up the `api.base.path` property from your `application.properties`, environment variables, or other property sources.

This is powerful for configuration. You might have `api.base.path=/api/v1` in development but `/api/v2` in production. Your code doesn't change - just the property value. The mapping gets registered with the resolved path.

The resolution happens once during initialization when the `RequestMappingInfo` objects are being built. It's not dynamic per-request - it's a startup-time substitution. So you can't have different paths for different requests based on runtime conditions. The path is baked in when the application starts.

You can use this in class-level mappings too: `@RequestMapping("${api.base.path}")` on the class means all methods in that controller get the configured prefix.

## Pattern Matching: PathPattern vs AntPathMatcher

Under the hood, Spring has two different systems for parsing and matching URL patterns, and understanding the difference helps explain some behaviors.

The older system is `AntPathMatcher`, which uses string-based pattern matching. It's lenient and somewhat loose in its semantics. The newer system is `PathPattern`, introduced in Spring 5 and made the default in Spring 5.3. It parses patterns into compiled objects that can match much faster, and it's stricter about syntax.

`PathPattern` gives you features like `{*path}` for capturing remaining paths. It handles regex constraints in templates. It's strict about separators - you can't have weird edge cases where a pattern accidentally matches URLs you didn't intend.

`AntPathMatcher` is more forgiving and supports some legacy features like `*.do` style extension patterns, which are now discouraged. If you're upgrading an old application, you might find patterns that worked with `AntPathMatcher` but fail with `PathPattern` because of the stricter parsing.

You can control which system is used through `RequestMappingHandlerMapping` configuration, but for new applications, you should stick with `PathPattern` as it's faster and clearer.

## Trailing Slash Handling and Spring Version Differences

Here's a gotcha that affects upgrades: how Spring treats trailing slashes changed between versions. In Spring 5 and earlier, the default behavior was that `/products` and `/products/` were treated as the same URL. If you registered a mapping for `/products`, it would also match `/products/`.

In Spring 6, this changed. Trailing slash matching is now disabled by default. If you register `/products`, a request to `/products/` returns 404 - they're different URLs now.

Why the change? Stricter, more predictable behavior. With trailing slash matching enabled, there was ambiguity - should a request to `/products/` invoke the `/products` handler or not? Different developers expected different things. Disabling it by default makes the behavior explicit.

If you're upgrading and you need the old behavior, you can re-enable trailing slash matching in your configuration. But the recommendation is to be explicit in your URLs - if you want both to work, register both: `@GetMapping({"/products", "/products/"})`.

## Specificity Resolution and Ambiguity Detection

Let me tie together how Spring chooses among multiple matching candidates. When a request comes in, Spring finds all `RequestMappingInfo` objects whose URL pattern matches the request path. Then it applies the other dimension checks - method, params, headers, consumes, produces. Some candidates get eliminated because they fail these checks.

You might be left with multiple candidates that all match fully. Now Spring compares their specificity. It looks at how specific each pattern is, how many methods they restrict to, whether they have params conditions, etc. It calculates a specificity score for each candidate.

If one candidate is clearly more specific than all others, it wins - that's your handler. If multiple candidates have equal specificity, you have an ambiguity problem. For some cases, this is detected at startup and you get an `IllegalStateException` immediately. Two methods with identical URL patterns and method restrictions are obviously ambiguous - Spring catches this during registration.

Other ambiguities only show up at runtime. Maybe you have `/products/{id}` and `/products/{name}` - these are identically specific patterns. At startup, Spring doesn't know they'll conflict because they're on different methods. But when a request comes in for `/products/42`, both match, both have the same specificity, and Spring throws `IllegalStateException` because it can't decide between them.

The solution to ambiguities is always to add differentiating conditions. If you really need two handlers for similar patterns, differentiate them by method, params, headers, consumes, or produces. Make one handle JSON and the other XML. Or make one require a specific parameter that the other doesn't. Give Spring a way to distinguish them.

## Special Handling for HEAD and OPTIONS

Let me elaborate on the automatic handling for certain HTTP methods. When you define a GET handler, Spring registers it for GET requests, but it also implicitly handles HEAD requests to the same URL. The implementation is clever: when a HEAD request arrives, Spring invokes your GET handler normally. Your handler executes, does whatever work it does, and sets response headers (maybe cache headers, content-type, etag, etc.). Then before sending the response, Spring strips out the body. The client gets all the headers your GET handler set, but no body content.

This is correct HEAD semantics - the response should be identical to GET except for the body. You don't need to write separate HEAD handlers unless you want optimized behavior (like avoiding expensive operations since the body won't be used anyway).

OPTIONS is also auto-handled. When Spring receives an OPTIONS request, it looks at all handlers registered for that URL and collects their HTTP methods. It responds with an Allow header listing all those methods. So if you have GET and POST handlers for `/products`, an OPTIONS request to `/products` returns `Allow: GET, HEAD, POST, OPTIONS`. The client learns what operations are supported.

You can override this by writing explicit OPTIONS handlers if you want custom behavior, but the automatic handling is usually sufficient.

## The Actual Request Matching Flow at Runtime

Let me walk you through what happens when an HTTP request arrives. The `DispatcherServlet` receives it and asks `RequestMappingHandlerMapping` to find the handler. The handler mapping looks at the request's URL path and finds all `RequestMappingInfo` objects whose path pattern matches. This is the first filter - many mappings get eliminated immediately because their URL pattern doesn't match.

For the remaining candidates, Spring checks the HTTP method. Any candidates whose method restrictions don't include the request's method get eliminated, and if all candidates fail this check, Spring notes there was a method mismatch.

Next it checks params conditions against the request's query parameters and form parameters. Candidates that don't satisfy their params conditions get eliminated.

Then headers conditions against the request headers. More candidates potentially eliminated.

Then consumes conditions against the request's Content-Type header.

Then produces conditions against the request's Accept header.

After all these filters, you might have zero candidates (no match), one candidate (success), or multiple candidates (need specificity comparison). If you have zero and there were method mismatches, you get 405. If there were consumes mismatches, you get 415. If there were produces mismatches, you get 406. If there were just URL mismatches, you get 404.

If you have one candidate, that's your handler - Spring invokes it.

If you have multiple candidates, Spring sorts them by specificity. The most specific wins. If there's a tie for most specific, that's an ambiguity error.

This entire process happens on every request, but it's heavily optimized with caching and efficient data structures. The performance is excellent even with hundreds of registered mappings.

## Using Params and Headers for Versioning Strategies

A practical application of params and headers is API versioning. There are multiple approaches to versioning REST APIs, and Spring's matching dimensions support several of them.

URL versioning puts the version in the path: `/api/v1/products` vs `/api/v2/products`. This is different URL patterns, so it's straightforward.

Parameter versioning uses query parameters: `/api/products?version=1` vs `/api/products?version=2`. You'd have multiple handlers for `/api/products`, differentiated by `params = "version=1"` vs `params = "version=2"`. The same URL with different parameters routes to different handlers.

Header versioning uses custom headers: `X-API-Version: 1` vs `X-API-Version: 2`. You'd use `headers = "X-API-Version=1"` vs `headers = "X-API-Version=2"` to differentiate handlers.

Media type versioning uses custom media types in the Accept header: `Accept: application/vnd.myapi.v1+json` vs `Accept: application/vnd.myapi.v2+json`. You'd use `produces = "application/vnd.myapi.v1+json"` vs `produces = "application/vnd.myapi.v2+json"`.

Each approach has tradeoffs. URL versioning is most visible and cache-friendly but pollutes URLs. Header and media type versioning keep URLs clean but are less visible. Spring supports all these strategies equally well through its matching dimensions.

## Edge Cases With Multiple Variables and Constraints

Let me address some advanced pattern features that occasionally come up. You can have multiple variables in a single path segment using comma separation: `/locations/{latitude},{longitude}`. This matches `/locations/51.5,-0.1` and extracts both values from that one segment.

You can use regex constraints to enforce formats: `/products/{id:\\d+}` only matches numeric IDs, while `/products/{slug:[a-z-]+}` only matches lowercase alphabetic slugs with hyphens. These constraints make your patterns more specific and help with disambiguation.

The `{*path}` syntax captures everything remaining in the path. If you have `/files/{*resourcePath}` and a request to `/files/documents/2024/report.pdf`, the `resourcePath` variable gets the value `/documents/2024/report.pdf`. This is useful for file serving or proxy scenarios where you want to capture arbitrary nested paths.

Matrix variables are a rarely-used feature where you embed key-value pairs in a path segment using semicolons: `/users/42;role=admin`. To use these, you need to enable them in configuration because semicolons are normally stripped from URLs. Then you can access the matrix variable with `@MatrixVariable`.

# Final Summary for Mastery and Recall

Let me synthesize everything into principles that will stay with you.

**RequestMapping is six-dimensional matching, not just URL routing**. Every mapping defines conditions across URL pattern, HTTP method, params, headers, consumes, and produces. All six dimensions must match simultaneously for a handler to be selected. This multi-dimensional nature explains the different HTTP error codes you see: 404 for URL mismatch, 405 for method mismatch, 400 for params or headers mismatch, 415 for consumes mismatch, 406 for produces mismatch.

**URL pattern specificity follows a clear hierarchy**. Exact literals beat URI templates, which beat single wildcards, which beat double wildcards. Among templates, regex constraints make patterns more specific. When multiple patterns could match, Spring always prefers the most specific. This ensures that your more precise handlers get priority over your more general fallback handlers.

**The composed annotations like GetMapping are safer than plain RequestMapping**. When you use `@RequestMapping` without specifying a method, you're accidentally accepting all HTTP methods, which is rarely what you want and can create security issues. The specialized annotations force you to be explicit about which method you want, preventing this mistake.

**Class-level and method-level combinations follow different rules for different attributes**. URLs concatenate - the class prefix gets prepended to the method path. Methods are overridden if the method level specifies any, otherwise inherited. Params and headers apply AND logic - both levels must be satisfied. Consumes and produces are completely overridden by method-level if specified. Understanding these rules prevents confusion about what your effective mapping actually is.

**Params and headers use the same four-part syntax**. Parameter or header name alone means it must be present. Negation with exclamation point means it must be absent. Equals sign means it must have a specific value. Not-equals means it must either be absent or have a different value. Multiple conditions all apply with AND logic.

**Consumes checks the request Content-Type for routing purposes**. It doesn't directly control deserialization - that happens later via message converters. Consumes is purely about which handler gets selected. Use it instead of `headers = "Content-Type=..."` because it understands media type semantics like wildcards and charset parameters.

**Produces checks the Accept header and sets the response Content-Type**. It serves double duty: routing (ensuring the client can accept what you produce) and response configuration (setting the Content-Type header). The wildcard `*/*` in Accept headers matches any produces constraint because it means "I accept anything."

**Pattern variable names don't matter for matching**. The patterns `/products/{id}` and `/products/{productId}` are identical and ambiguous. Only the structure matters, not the variable names. Use regex constraints to actually differentiate similar patterns.

**Trailing slash behavior changed in Spring 6**. The old default where `/products` and `/products/` were equivalent is gone. Now they're different URLs unless you explicitly map both. This is stricter and more predictable but can break applications during upgrades if you relied on the old behavior.

**Property placeholders resolve at startup, not per request**. When you use `${api.prefix}` in a path, Spring substitutes the value during application initialization. You can't have dynamic paths that change based on runtime conditions. The mapping is locked in when the context starts.

**Head and Options get automatic handling**. Every GET handler automatically handles HEAD by stripping the body. Every URL automatically handles OPTIONS by returning which methods are supported. You can override these by writing explicit handlers, but usually the automatic behavior is what you want.

**Ambiguity errors can be immediate or deferred**. Identical patterns on different methods cause startup errors because Spring detects the conflict during registration. Patterns that are different but equally specific cause runtime errors when a request matches both. Either way, the solution is to add differentiating conditions through params, headers, consumes, or produces.

**Specificity includes all dimensions, not just URL patterns**. When comparing candidates, Spring looks at how specific the URL pattern is, how restrictive the method set is, whether params conditions exist, whether headers conditions exist, and whether consumes and produces are specified. A mapping with more conditions is generally more specific than one with fewer conditions.

Remember that the entire `@RequestMapping` mechanism exists to answer one question: given an incoming HTTP request with all its characteristics, which handler method should process it? The six dimensions give you fine-grained control over that decision, letting you build sophisticated routing logic that goes far beyond simple URL matching. Master these dimensions and their combination rules, and you'll understand Spring MVC's request routing at a deep level.
