# ApplicationContext vs WebApplicationContext: The Two-Container Hierarchy

Let's build your understanding from the ground up, because this topic trips up even experienced Spring developers. The key insight you need to hold in your head is this: **in a classic Spring MVC application, there are two separate Spring IoC containers running simultaneously, in a parent-child relationship.** Everything else flows from that one fact.

---

## Why Does Spring Need Two Containers at All?

Before diving into the mechanics, let's ask the "why" — because the hierarchy exists for real architectural reasons, not arbitrarily.

Imagine a large enterprise app that has two different entry points: a public-facing REST API and an internal admin panel. Both need access to the same `ProductService`, `OrderService`, and database connection. But they have completely different controllers, view resolvers, and web configurations. If you had only one container, you'd either have to cram everything together (messy, inflexible) or duplicate your service layer (wasteful and dangerous).

The two-container hierarchy solves this elegantly: **shared business logic lives in one container (the parent), and each web entry point gets its own isolated container (a child).** Multiple `DispatcherServlet` instances can each have their own child context, yet all share the same root context's services and repositories.

```
Root Context: ProductService, OrderService, UserRepository (shared by everyone)
     │
     ├── Child Context A (API DispatcherServlet): ApiController, ApiExceptionHandler
     │
     └── Child Context B (Admin DispatcherServlet): AdminController, AdminViewResolver
```

Even if your app has only one `DispatcherServlet`, the separation still enforces a healthy architectural rule: **your service layer should never depend on your web layer.** A `@Service` should never need to know about a `@Controller`. The hierarchy enforces this at the container level — the root context literally cannot see child context beans.

## The Two Containers: Who Creates Them?

### The Root ApplicationContext — Created by ContextLoaderListener

When your application starts up, the very first thing that runs is `ContextLoaderListener`, which is a standard `ServletContextListener`. It fires during servlet container (Tomcat) startup, before any request is processed. Its job is to create the **root ApplicationContext**, load your service/repository/infrastructure beans into it, and store the finished context in a `ServletContext` attribute under the key `WebApplicationContext.ROOT_WEB_APPLICATION_CONTEXT_ATTRIBUTE`.

That last step — storing in `ServletContext` — is the *handshake* mechanism. It's how the root context makes itself discoverable to whatever comes next.

### The Child WebApplicationContext — Created by DispatcherServlet

When `DispatcherServlet` initializes (either on startup if `load-on-startup=1` is set, or on the first request otherwise), it calls `initWebApplicationContext()`. The first thing it does is look in the `ServletContext` for that root context attribute. It finds the root context, creates a **new, separate child WebApplicationContext**, sets the root context as its *parent*, loads your web-layer beans (controllers, view resolvers, handler mappings) into it, and stores this child context in `ServletContext` too (under a key based on the servlet name).

So after startup you have two live containers in memory, linked in a parent-child relationship.

---

## The Visibility Rules — The Most Important Thing to Internalize

This is the conceptual core. The parent-child relationship is **strictly one-directional**:

**The child context can see all beans in the parent (root) context.** When you `@Autowire` something in a controller, Spring first searches the child context's own registry. If it doesn't find the bean there, it automatically delegates the lookup to the parent context. This is why a `@Controller` can successfully inject a `@Service` that lives in the root context — the delegation happens transparently.

**The parent (root) context cannot see any beans in the child context.** The root context has no reference to its children. If you tried to `@Autowire` a `@Controller` into a `@Service`, Spring would look in the root context, fail to find it, and throw `NoSuchBeanDefinitionException`. There is no upward delegation.

The lookup algorithm in the child context is essentially: *"Do I have it? No? Ask my parent. Parent has it? Great. Parent doesn't? Throw an exception."*

A helpful mental model: think of it like a company org chart. A junior employee (child context) can ask their manager (root context) for resources. But the manager doesn't go asking the junior employee for things — the information only flows upward.

## The ApplicationContext and WebApplicationContext Interfaces

It's worth understanding what `WebApplicationContext` *adds* over plain `ApplicationContext`. The core `ApplicationContext` interface gives you dependency injection, event publishing, i18n, and resource loading. `WebApplicationContext` extends it with two critical additions:

First, it provides access to `ServletContext` — the gateway to the servlet container's environment. Second, and more importantly, it **registers web-specific scopes: `request`, `session`, and `application`**. A plain `ApplicationContext` only knows about `singleton` and `prototype`. This means you cannot define a `@RequestScope` bean in a plain context — it simply doesn't know what "request" means as a scope.

One thing that trips people up: both the root context *and* the child context are `WebApplicationContext` implementations in a standard Spring MVC setup. `ContextLoaderListener` creates an `XmlWebApplicationContext` or `AnnotationConfigWebApplicationContext` — both implement `WebApplicationContext`. The distinction between them is purely their **role** (parent vs child) in the hierarchy, not their type.

## The Biggest Real-World Bug: Duplicate Component Scanning

This is the #1 most dangerous consequence of misunderstanding the hierarchy, and it fails silently — making it incredibly hard to debug.

Here's the scenario: a developer configures both `RootConfig` and `WebConfig` to `@ComponentScan("com.example")` without any filters. Both contexts scan the entire package. Both contexts find `ProductService` annotated with `@Service @Transactional`. Both contexts create a `ProductService` bean.

The root context creates its `ProductService` with a proper `@Transactional` proxy wrapped around it, because `TransactionManager` and the AOP infrastructure are set up in the root context.

The child context creates its own `ProductService` bean too — but as a plain, unproxied object, because the transaction infrastructure isn't being set up there.

When the `ProductController` in the child context does `@Autowire ProductService`, Spring looks in the child context's own registry *first*, finds the locally created (non-transactional) `ProductService`, and uses that. It never delegates to the parent. The controller silently gets a non-transactional service. Your `@Transactional` methods do nothing. No error is thrown. Data integrity bugs ensue in production.

The correct fix is precise component scan filtering:

```java
// Root context — scans everything EXCEPT @Controller beans
@ComponentScan(
    basePackages = "com.example",
    excludeFilters = @Filter(type = ANNOTATION, classes = Controller.class)
)
public class RootConfig { }

// Child context — scans ONLY @Controller beans
@ComponentScan(
    basePackages = "com.example",
    useDefaultFilters = false, // turn off default scanning
    includeFilters = @Filter(type = ANNOTATION, classes = Controller.class)
)
public class WebConfig { }
```

With this setup, `ProductService` exists only in the root context. The controller's autowiring delegates to the parent and gets the correctly proxied bean. Transactions work as expected.

## The ContextRefreshedEvent Trap

Both contexts, when they finish initializing, publish a `ContextRefreshedEvent`. So in a two-context setup, this event fires *twice*. To make it trickier, Spring propagates child context events *up* to parent context listeners — so a `@EventListener` sitting in the root context will receive *both* the root's own event and the child's event.

This means if you do initialization work like warming a cache or seeding a database in a `ContextRefreshedEvent` listener, it will run twice. The fix is a simple parent-check:

```java
@EventListener
public void onStartup(ContextRefreshedEvent event) {
    // Check if this is the root context — it has no parent
    if (event.getApplicationContext().getParent() == null) {
        performExpensiveInitialization(); // Now runs exactly once
    }
}
```

## Spring Boot: The Hierarchy Disappears

Spring Boot takes a much simpler approach: it uses a **single** `AnnotationConfigServletWebServerApplicationContext` that contains everything — your controllers, services, repositories, view resolvers, and even the `DispatcherServlet` itself (registered as a bean).

`ContextLoaderListener` is not used. There is no root context, no child context, no parent-child delegation. One container holds the world.

```java
ConfigurableApplicationContext ctx = SpringApplication.run(App.class, args);
System.out.println(ctx.getParent()); // null — no parent
DispatcherServlet ds = ctx.getBean(DispatcherServlet.class); // found — it's a bean!
```

This eliminates the entire class of duplicate-scan bugs, because there's only one registry to scan. It also means `ContextRefreshedEvent` fires exactly once. The trade-off is that you lose the ability to have multiple `DispatcherServlet` instances with isolated web contexts, but in practice almost no modern application needs that.

## A Quick Mental Self-Test

To check your understanding, try answering these before looking at the answers:

Can a `@Service` in the root context `@Autowire` a `@Controller` from the child context? (*No — parent cannot see child*.) If both contexts define a bean named `userService`, which one does a controller get? (*The child's own — local registry is checked first, so parent delegation never happens*.) In Spring Boot, how many times does `ContextRefreshedEvent` fire at startup? (*Once — single context*.)

If you can reason through those confidently, you have a solid grasp of the hierarchy. The underlying principle is always: **child sees parent, parent is blind to child, and anything that breaks that mental model is a bug waiting to happen.**
